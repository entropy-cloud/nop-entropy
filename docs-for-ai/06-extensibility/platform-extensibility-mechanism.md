# Nop 平台的通用可扩展机制

## 这篇文档的定位

这篇文档讨论的是 **Nop 平台整体的可扩展设计**，不是某个业务模块的功能点。

如果你是为了做业务实现时的技术选型，而不是研究平台原理，优先先读 `./how-to-use-extensibility-in-business-implementation.md`。

核心判断：

Nop 的独特性不在于“某个引擎提供了多少扩展点”，而在于它把 **扩展能力本身** 做成了一套通用基础设施，可被 ORM、工作流、页面、规则、报表、IoC、任务编排等多个子系统复用。

换句话说，Nop 不是为每个引擎分别设计一套扩展机制，而是先设计一套 **元模型 + XDSL + Delta + 编译期元编程 + 保留层** 的通用机制，再让各个引擎建立在这套机制之上。

## 和传统框架的根本区别

很多传统框架的扩展路径主要是：

- Java 接口/SPI
- 注解
- 继承
- AOP
- 钩子回调

这些手段当然也有价值，但它们主要发生在 **Java 运行时**。

Nop 的扩展机制前移到了更高层：

1. **先定义 DSL 的元模型**
2. **再用 DSL 描述应用和引擎配置**
3. **再通过 Delta 和继承做差量定制**
4. **必要时通过编译期元编程把 DSLx 翻译回基础 DSL**
5. **最后才落到 Java 运行时对象和执行逻辑**

因此它的扩展不是“先写代码，再找钩子挂进去”，而是“先改模型和模型组合关系，再决定是否需要 Java”。

## 一个总公式

在 Nop 里，最值得记住的不是某个 API，而是这条公式：

```text
App = Delta x-extends Generator<DSL>
```

含义是：

- 应用不是一坨手写代码直接堆出来的
- 生成器先给出基础结构
- 用户通过 Delta 和保留层做差量覆盖
- 最终结果是多层模型合成出来的

这也是为什么 Nop 能把“二次开发”和“基础产品升级”放在同一个统一框架里考虑。

## 一条主线

如果只用一句话概括 Nop 的扩展机制，可以写成：

```text
XDef 定义 DSL -> XDSL 描述系统 -> Delta/保留层做差量覆盖 -> x:gen-extends/x:post-extends 做编译期展开 -> DslModelParser/init/analyze 形成运行时模型
```

这条主线比“某个模块支持扩展”更重要，因为它描述的是整个平台共享的工作方式。

## 三个关键词

### 1. 模型先于代码

在 Nop 里，很多你在其他框架中会直接写成 Java 的东西，会先被提升为模型：

- 数据结构是模型
- 页面结构是模型
- 工作流结构是模型
- IoC 装配也是模型

因此扩展默认优先发生在模型层，而不是代码层。

### 4. 扩展属性默认先于新增节点

Nop 的一个非常关键、也非常容易被忽略的设计是：

- **XDSL 中的任意节点，默认都可以增加扩展属性**
- **除非显式指定要检查的名字空间，否则带名字空间的属性通常不参与 xdef 校验**
- **这些扩展属性可以在编译期被 XLib / XPL / `x:post-extends` 读取，并据此展开为正式 DSL 结构**

这意味着很多新增语义并不需要：

- 先去改 xdef 增一个正式属性
- 先去改运行时对象加一个字段
- 先去改引擎分支识别一种新节点

它们可以先表现为：

- `biz:*`
- `wf:*`
- `xpt:*`
- 其他自定义名字空间属性

然后在编译期被翻译成运行时真正理解的结构。

### 2. 差量先于复制

Nop 的默认思路不是复制整份基础文件再改，而是：

- 继承基础模型
- 只写变化部分
- 让最终结果通过合并得到

这也是 Delta、保留层、`x:extends="super"` 这些机制的共同目标。

### 3. 编译先于运行时硬编码

当基础 DSL 不够友好时，Nop 优先考虑在编译期把 DSLx 翻译成 DSL，而不是直接把新语义塞进运行时引擎。

这样做的结果是：

- 运行时内核更稳定
- 扩展语法更自由
- 不同领域都能复用同一套编译与合并框架

## 六层扩展机制

## 1. 元模型层：先定义 DSL，再谈扩展

Nop 的 DSL 不是随手写的 XML 约定，而是先由 XDef 定义。

XDef 提供：

- 元素/属性结构定义
- 类型系统
- 列表 key 规则
- 引用与复用规则
- 校验能力

这一步的意义是：

- 扩展首先是扩 DSL，而不是先改 Java 类
- DSL 一旦被 XDef 描述，就可以获得统一的解析、校验和工具支持
- 多个子系统可以共享同一种扩展方法论

平台锚点：

- `nop-kernel/nop-xdefs/src/main/resources/_vfs/nop/schema/`
- `nop-kernel/nop-xdefs/.../xdsl.xdef`

## 2. 模型层：所有子系统都先落到 XDSL

ORM、workflow、page、beans、xbiz、task、rule 这些看似不同的东西，在平台内部首先都是 XDSL 模型。

这意味着：

- 它们都走统一的模型解析链
- 它们都可以使用 `x:schema`
- 它们都可以使用 `x:extends`
- 它们都可以使用 `x:gen-extends` / `x:post-extends`
- 它们都可以被 Delta 覆盖
- 它们的节点默认都可以附加名字空间扩展属性，供编译期消费

关键点不是“工作流支持继承”，而是“**所有 XDSL 都支持继承**”。

## 3. 组合层：`x:extends` + `x:override` + Delta

这是 Nop 最常用、最基础的扩展能力。

模型不是一次性写死，而是按层合成：

- 基础模型
- 生成模型
- 当前手写模型
- Delta 层覆盖模型
- tenant 层或环境层补充模型

这里的关键不是简单“支持 include”，而是：

- 有统一的合并语义
- 有 `replace/remove/merge` 等算子
- 有 `super:` 语义
- 有 VFS 层式资源解析支持

这套机制是通用的，不依赖业务领域。

它被同样用在：

- ORM 扩展
- beans 覆盖
- 页面覆盖
- 权限文件聚合
- YAML/JSON 模型差量加载

平台锚点：

- `DeltaResourceStore`
- `DeltaResourceStoreBuilder`
- `XDslExtender`
- `GlobalFunctions.loadDeltaJson`

## 4. 编译期元编程层：`x:gen-extends` / `x:post-extends`

这是 Nop 可扩展性和一般配置继承最大的差别之一。

很多平台只能做“在已有模型上覆盖几个字段”。

Nop 还可以做：

- 根据上层模型自动生成下层模型片段
- 在编译期把一种 DSLx 展开成基础 DSL
- 让运行时完全不知道扩展语法的存在
- 读取节点上的名字空间扩展属性，据此决定生成/展开结果

这就是为什么很多在别的框架里需要“往引擎里加一种新节点/新配置对象”的需求，在 Nop 里可以变成：

- 新增一种更友好的 DSLx
- 再用 `x:post-extends` 把它翻译成已有 DSL

因此，扩展往往不需要直接修改运行时引擎，而是先修改“编译前模型到编译后模型”的转换过程。

这是一种真正的 **元编程式扩展**，而不只是配置叠加。

这里最重要的认识是：

- 运行时引擎不一定要认识 DSLx
- 只要编译期能把 DSLx 稳定翻译成 DSL，扩展语法就成立

因此很多“想加一种新语法”的需求，不该第一反应去改运行时，而该先判断是不是一个编译期问题。

典型做法就是：

1. 在现有节点上加一个或一组扩展属性
2. 用 `meta-gen.xlib`、`control.xlib`、`meta-prop.xlib` 这一类编译期库去读取这些属性
3. 生成或改写运行时真正消费的标准结构

## 5. 保留层与生成层：把“生成”和“手写”纳入统一扩展模型

Nop 的生成物通常不是最终可编辑层，平台会保留稳定的手写层。

例如：

- 非下划线文件扩展下划线生成文件
- 页面保留层覆盖 `_gen` 产物
- 手写 xbiz / xmeta / auth 文件覆盖生成基础

这意味着 codegen 不是一次性 scaffolding，而是长期参与系统演化。

因此平台的“可扩展”不只是运行期可扩展，也包括：

- 生成链可扩展
- 生成结果可覆盖
- 升级时保留层仍然有效

这也是 Nop 能同时支撑“模型驱动生成”和“持续二次开发”的关键。

## 6. 模型到运行时桥接层：解析后还能继续派生语义

XDSL 解析不是把 XML 直接映射成 POJO 就结束。

Nop 在模型加载后还会有一层统一的初始化/分析过程，例如：

- `DslModelParser` 负责从 XDSL 解析到对象
- `INeedInit` 负责模型初始化
- 各模型分析器会建立索引、派生字段、合法性校验和图结构

这一步的意义是：

- 扩展不仅是“把新字段读进来”
- 还可以在 init/analyze 阶段把声明式模型转成高效运行态结构
- 不同引擎仍可共享同一个模型装配框架

因此，Nop 的运行时对象通常是“编译后、合并后、初始化后”的结果，而不是某个原始文件的直接镜像。

## 这套机制为什么是通用的

这套机制与业务领域无关，原因在于它不依赖“审批”“订单”“表单”这些语义，而依赖以下通用抽象：

- 元模型
- 模型继承
- 模型差量覆盖
- 编译期转换
- 保留层
- 模型初始化
- 统一资源定位与分层加载

因此：

- `nop-wf` 可以用它做 `oa.xwf`
- `nop-orm` 可以用它做模型扩展和 codegen
- 页面 DSL 可以用它做 `_gen` + 手写页混合
- auth/menu 可以用它做聚合继承
- YAML/JSON 模型也可以加载 Delta

真正可复用的不是某个 workflow 模板，而是这整套扩展基础设施。

## 五种常见扩展落位

面对一个新需求，通常可以把它落在下面五种位置之一：

| 扩展落位 | 适用问题 | 典型形式 |
|------|------|------|
| **源模型** | 基础结构要变 | `model/*.orm.xml`、`.xwf`、`.view.xml`、`.api.xml` |
| **Delta 层** | 要覆盖已有产品或已有基础层 | `_vfs/_delta/...` + `x:extends="super"` |
| **保留层** | 要覆盖生成结果但保留升级路径 | 非下划线文件扩展 `_gen` / `_*.xbiz` / `_*.xmeta` |
| **编译期元编程** | 想要更高层 DSLx 或自动展开 | `x:gen-extends`、`x:post-extends`、`.xpl`、`.xgen` |
| **运行时 Java** | 前四层都不足，必须进执行逻辑 | BizModel、Processor、引擎实现、Provider/SPI |

正确姿势不是记住“永远不要写 Java”，而是 **先判断前四层能否解决**。

## 一个决策树

```text
新需求来了
├─ 是结构变化吗？
│  └─ 先改源模型 / XDSL
├─ 是对现有产品或平台资源做差量覆盖吗？
│  └─ 先用 Delta
├─ 是对生成结果做稳定手写覆盖吗？
│  └─ 先用保留层
├─ 是基础 DSL 不够友好吗？
│  └─ 先想 x:gen-extends / x:post-extends / XLib / XPL
└─ 前面都不够
   └─ 再考虑 Java 运行时扩展
```

## 与传统 SPI/AOP 的关系

这并不是说 Nop 不需要 Java 扩展点。

更准确地说：

- Java SPI/AOP/接口扩展仍然存在
- 但它们不再是第一扩展层
- 它们位于一套更高层模型机制的后面

因此，Nop 和传统框架的差别不是“有没有 SPI”，而是“SPI 在整个扩展体系里排第几层”。

## 扩展属性为什么重要

如果没有“节点默认可带名字空间扩展属性”这条规则，很多扩展都会被迫走两条更重的路径之一：

- 为了一个局部语义去修改 xdef 正式结构
- 为了一个局部语义去修改运行时 Java 模型或引擎

有了这条规则之后，可以先采用更轻的路径：

- 先把语义挂在扩展属性上
- 再由编译期库决定如何解释它

这就是 Nop 很多“领域能力退出引擎”的技术前提。

## 正式属性 vs 扩展属性

### 正式属性

特点：

- 由 xdef 明确声明
- 是基础 DSL 契约的一部分
- 基础运行时或编译链通常直接理解它

适合：

- 已成为稳定公共语义
- 多个模块和运行时核心都直接依赖它

### 扩展属性

特点：

- 挂在现有节点上
- 主要由编译期模板、XLib、XPL、生成器消费
- 经常只是 DSLx 的输入，而不是最终运行时字段

适合：

- 局部领域语义
- 某条生成链或某类方言需要的附加信息
- 暂时不想提升为基础 DSL 契约的语义

一个常见误解是把扩展属性当成“临时 hack”。更准确的理解是：**它是 Nop 通用可扩展机制的正式组成部分，只是层级比基础 schema 更高。**

## 扩展属性通常如何被消费

最常见的不是运行时直接读取，而是编译期消费：

1. 在现有节点上增加名字空间扩展属性
2. `x:gen-extends` / `x:post-extends` / XLib / XPL 读取这些属性
3. 生成正式 DSL 结构、标准属性或运行时模型片段
4. 运行时只看到展开后的结果

因此，扩展属性经常是中间语义载体，而不是最终契约。

## 三类常见扩展属性用法

### 1. 资源归属或元信息提示

例子：`biz:moduleId`

它不改变 ORM `<entity>` 的基础结构，但为 codegen 和运行时页面路由提供模块归属信息。

### 2. 领域规则附着在现有节点上

例子：`wf:upLevel`、`wf:permissions`

它们不必催生新的 actor 节点类型，而是附着在现有 actor 配置上。

### 3. 生成器或模板指令

例子：`xpt:*`

它们主要给报表/导入模板转换器提供元编程输入。

## 什么时候该把扩展属性升级为正式属性

可以考虑升级的信号：

1. 多个独立模块稳定依赖这条语义
2. 基础运行时内核本身也要直接理解它
3. 仅靠编译期消费已经不够清晰
4. 它已经从“局部方言”变成“基础 DSL 契约”

如果它只是某类模板、某类页面、某类 workflow 方言需要的附加信息，通常先保留为扩展属性更轻。

## 什么时候要显式限制名字空间

默认放宽不等于任何场景都应无限开放。

需要显式限制名字空间的典型场景：

- 安全敏感的编译执行入口
- 受限 DSL，只允许少数自定义标签/属性
- 不希望任意名字空间污染结构的局部解析器

这时应通过显式检查规则限制允许的名字空间，而不是依赖默认策略。

## 一个重要结论：很多“引擎功能”其实可以退出引擎

在 Nop 的设计里，很多其他框架会塞进引擎核心的能力，可以被拆出去：

- 领域模板
- 扩展属性
- 标签库
- DSLx 到 DSL 的编译期转换
- 保留层覆盖

所以面对新需求时，不要默认问：

- 需不需要给引擎新增枚举
- 需不需要新增一个 Java 节点类型
- 需不需要在核心状态机里加分支

应该先问：

1. 能不能只是扩 XDef/XDSL
2. 能不能只是加一个 Delta
3. 能不能只是做一个 `x:post-extends` 转换
4. 能不能只是加一个 XLib 标签
5. 能不能只是加一个保留层文件

只有这些都不够，才考虑 Java 运行时扩展。

## 默认扩展顺序

当你要做平台或平台模块定制时，默认按下面顺序判断：

1. 改源模型
2. 改 Delta
3. 改非下划线保留层
4. 改 XDef / XDSL 结构
5. 改 `x:gen-extends` / `x:post-extends` / XPL / XLib
6. 最后才改 Java 运行时代码

这个顺序不是经验偏好，而是平台架构本身的默认设计。

## 反模式

以下写法通常都说明你绕过了平台的通用可扩展机制：

- 直接改生成文件
- 有 DSL 却绕过 DSL 直接写 Java 配置
- 需要一处小扩展却复制整份基础文件
- 能用 Delta 却 fork 基础产品源码
- 需要一套新语法却直接往运行时引擎塞硬编码分支
- 把某个模块的扩展性误解为该模块自己的特例

## `nop-wf` 为什么只是案例，不是起点

`nop-wf` 很适合拿来展示这套机制，因为 `oa.xwf`、`oa.xlib`、工作流 schema、设计器 DSL 都很直观。

但 `nop-wf` 不是这套机制的来源，它只是其中一个受益模块。

真正的来源在平台底层：

- XDef / XDSL
- XDslExtender
- DslModelParser
- DeltaResourceStore
- XPL / XLib / xgen

所以正确的叙述顺序应该是：

1. 先理解平台级可扩展机制
2. 再看 `nop-wf` 如何把这套机制用于审批领域

而不是反过来。
