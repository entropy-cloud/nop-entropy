# 业务实现时如何利用 Nop 的可扩展设计

## 这篇文档是写给谁的

这篇文档是写给 **在业务实现阶段做技术决策的 AI** 的。

目标不是解释 Nop 为什么这样设计，而是让 AI 在面对真实需求时，知道：

- 应该优先利用哪些平台扩展机制
- 什么时候该改模型、Delta、保留层、XLib、编译期生成
- 什么时候才应该新增 Java 代码
- 怎样做才符合 Nop 的最佳实践，而不是把问题做重

如果你只看一篇，请先看这篇。

## 先给结论

在 Nop 里做业务实现时，默认不是：

- 先写 Java
- 先改服务层
- 先给引擎补分支

默认应该是：

1. 先判断业务语义能否落到模型
2. 再判断能否通过 Delta 或保留层覆盖
3. 再判断能否通过扩展属性、XLib、编译期元编程表达
4. 只有这些都不够，才写 Java

## AI 的默认决策顺序

面对一个业务需求，按这个顺序问自己：

### 1. 这是业务事实变化，还是执行逻辑变化？

如果是这些内容：

- 新增实体/字段/关系/字典
- 新增页面字段
- 新增标准 CRUD 结构
- 新增流程节点/步骤配置
- 新增报表模板字段

优先考虑 **模型层**，不要先写 Java。

### 2. 这是改“源头结构”，还是改“现有基础结果”？

- 改源头结构：优先改源模型/XDSL
- 覆盖已有平台资源：优先 Delta
- 覆盖生成结果：优先保留层

这一步的目标是先确定落位，不要一上来就实现逻辑。

### 3. 这是基础 DSL 不够，还是基础 DSL 足够但表达不方便？

- 基础 DSL 足够表达，但写起来太繁琐：优先考虑扩展属性 + 编译期展开
- 基础 DSL 根本表达不了：再考虑是否需要升级 xdef 或运行时逻辑

### 4. 这个需求能否只作为“附加语义”挂在现有节点上？

如果可以，优先考虑：

- 名字空间扩展属性
- XLib / XPL
- `x:post-extends`

不要急着新增节点、新枚举、新运行时字段。

### 5. 真正需要 Java 吗？

只有在下面这些情况，才默认进入 Java：

- 需要事务、存储、远程调用、复杂算法协同
- 编译期无法完成的运行时行为
- BizModel / Processor / 引擎逻辑必须新增执行路径

## 常用扩展模式速查

AI 做业务实现时，最常用的其实就下面几种模式：

| 模式 | 什么时候优先用 | 典型形式 |
|------|------|------|
| **源模型** | 基础结构、字段、关系、步骤、页面骨架要变 | `model/*.orm.xml`、`.xwf`、`.view.xml` |
| **Delta** | 覆盖已有平台资源或已有基础层 | `_vfs/_delta/...` + `x:extends="super"` |
| **保留层** | 覆盖生成结果但不改生成物 | 非下划线文件扩展 `_gen` / `_*.xbiz` / `_*.xmeta` |
| **扩展属性 + 编译期展开** | 基础 DSL 足够，但想附加更高层语义 | `biz:*`、`wf:*`、`xpt:*` + `x:post-extends` / XLib |
| **标签库/XLib** | 想把规则外置为声明式组件 | `.xlib`、XPL 标签 |
| **`@BizLoader` / Delta 类** | 只做 API/GraphQL 局部增强 | `@BizLoader(autoCreateField = true)` |
| **Java 运行时代码** | 前面几层都不够 | BizModel、Processor、引擎实现 |

默认不要倒过来用。也就是：不要先选 Java，再回头想模型/Delta 能不能做。

## 业务实现中的最佳实践

## 场景 1：新增业务字段

优先做法：

1. 改 ORM 源模型
2. 重新生成
3. 通过保留层/页面层决定是否展示
4. 只有自动生成链不够时，再补 Java

不推荐：

- 先改 DTO
- 先改返回对象
- 先改 `_gen`

## 场景 2：平台模块差量定制

优先做法：

1. 先找是否可以 Delta 覆盖
2. 路径对齐原资源
3. 用 `x:extends="super"`
4. 只写变化部分

不推荐：

- 复制整份平台文件到业务模块
- 直接改平台源码

## 场景 3：页面/控件行为调整

优先做法：

1. 先看模型/XMeta/domain/control 是否已能表达
2. 再看是否能通过扩展属性影响控件匹配
3. 再看是否需要保留层 view/page 覆盖

不推荐：

- 直接手写整页替代生成基线
- 直接把 UI 规则下沉到 Java

## 场景 4：想给现有 DSL 增加一层业务友好语法

优先做法：

1. 先用现有节点 + 扩展属性表达新语义
2. 用 `x:post-extends` / XLib / XPL 在编译期展开
3. 最终仍落回基础 DSL

不推荐：

- 一上来修改运行时引擎识别新节点

## 场景 5：只想给已有 API/GraphQL 多返回一点信息

优先做法：

1. 先判断是不是基础模型字段
2. 如果不是，优先 `@BizLoader`
3. 字段不存在时，再看 Delta + `@BizLoader(autoCreateField = true)`

不推荐：

- 为了一个展示字段回源改很多底层结构

## 场景 6：工作流/规则/报表这类 DSL 需要局部增强

优先做法：

1. 先看模板继承
2. 再看扩展属性
3. 再看 XLib / XPL
4. 再看编译期展开

不推荐：

- 把领域规则直接写进引擎主循环

## 扩展属性在业务实现中的用法

业务实现时，AI 还要特别注意一条规则：

- **现有节点默认通常就可以挂名字空间扩展属性**
- **这些属性很多时候是给编译期工具看的，不是直接给运行时对象看的**

所以如果需求只是“给现有节点附加一点局部语义”，先想：

1. 能不能挂成 `biz:*`、`wf:*`、`xpt:*` 之类扩展属性
2. 能不能由 `control.xlib`、`meta-gen.xlib`、`meta-prop.xlib`、`x:post-extends` 去消费它
3. 能不能避免回头修改基础 xdef 或运行时引擎

只有当这个语义已经稳定到必须成为基础 DSL 契约时，才考虑把它升级为正式属性。

## AI 应该怎样理解“最佳实践”

在 Nop 里，“最佳实践”不是某个具体 API，而是 **优先走最上层、最稳定、最可复用的扩展层**。

通常意味着：

- 优先模型，不优先代码
- 优先差量，不优先复制
- 优先编译期，不优先运行时硬编码
- 优先保留层，不优先手改生成物
- 优先已有平台机制，不优先自造旁路实现

## 一张决策表

| 需求类型 | AI 默认优先方案 | 最后手段 |
|------|------|------|
| 结构变化 | 源模型 / XDSL | Java |
| 平台资源覆盖 | Delta | 直接改平台源码 |
| 生成结果补定制 | 保留层 | 改 `_gen` |
| 更高层业务语法 | 扩展属性 + `x:post-extends` / XLib | 改运行时引擎 |
| 规则外置 | XLib / 模板 / 扩展属性 | Java if/else |
| API 局部增强 | `@BizLoader` / Delta | 大改基础对象 |

## 一张反模式表

| 反模式 | 更好的做法 |
|------|------|
| 先写 Java，再想模型是否可表达 | 先判断模型/Delta/保留层/扩展属性 |
| 复制整份平台文件 | 优先 Delta |
| 手改 `_gen` | 改源模型或保留层 |
| 为局部语义改运行时引擎 | 先试扩展属性 + 编译期展开 |
| 为局部展示字段大改基础对象 | 优先 `@BizLoader` / Delta |

## AI 的自检问题

在真正下手前，AI 应至少问自己：

1. 这个需求是不是可以只改模型？
2. 这个需求是不是可以只做 Delta？
3. 这个需求是不是可以只改保留层？
4. 这个需求是不是可以先挂成扩展属性？
5. 这个需求是不是可以用 `x:post-extends` / XLib 在编译期展开？
6. 为什么我现在真的需要写 Java？

如果前 5 个问题都没有认真判断，就直接写 Java，通常不是 Nop 的最佳实践。

## 你读完本页后应该做到什么

读完本页后，AI 应该能做到两件事：

1. **理解 Nop 的可扩展设计不是抽象理念，而是业务实现时的默认决策顺序**。
2. **在实现业务时，优先选用模型、Delta、保留层、扩展属性、编译期元编程这些更轻、更稳定的路径。**
