## 自包含性要求（P0 - 必须满足）

### 文档引用规范

#### 仅引用 docs-for-ai 目录内文档

所有文档链接必须指向 `docs-for-ai` 目录内的文件，不得引用其他目录。

| 正确示例 | 错误示例 |
|---------|---------|
| `[相关文档](./data-access.md)` | `[外部文档](../dev-guide/xxx.md)` |
| `[XScript](../05-xlang/xscript.md)` | 引用主 `docs/` 目录的文件 |

#### Java 类引用只写全类名

提及 Java 类时只提供全限定类名，不引用源码路径。

| 正确示例 | 错误示例 |
|---------|---------|
| `io.nop.api.core.annotations.biz.BizLoader` | `nop-api-core/src/main/java/.../BizLoader.java` |

#### 禁止外部链接

- ❌ 所有 `http://` 或 `https://` 开头的外部链接（XML Schema 命名空间除外）
- ❌ 外部文档、视频教程、示例项目的链接
- ❌ 问题反馈渠道链接（GitHub Issues、Gitee Issues 等）

**例外**：XML Schema 命名空间可以保留（如 `http://nop-xlang.github.io/schema/*.xdef`）

### 文档元信息规范

**禁止文档元信息**：文档中不得包含版本、修改人、修改时间等信息。

- ❌ "文档版本"、"最后更新"、"作者"、"维护者"、"状态"
- ❌ "文档版本: vX.X"、"最后更新: YYYY-MM-DD"

> 文档必须始终是最新的，不需要版本管理信息。

## 定期维护

### 每月检查

- 外部链接是否新增加
- 代码示例是否仍与源码一致
- 是否有新增加的通用知识

### 每季度审查

- 文档结构是否合理
- 是否有新的平台特性需要补充
- 示例代码是否需要更新

### 每次大版本更新

- 全面审查所有文档
- 更新 API 示例
- 检查废弃功能是否已删除
