# 广义可逆计算：面向软件构造与演化的语义坐标系和差量代数

## 摘要

长期演化的软件系统通常同时面临生成、定制、升级、客户变体和局部修复等多类构造活动。现有方法分别从模型驱动工程、软件产品线、特征导向编程、面向差量编程、语言工作台和版本控制等方向处理其中一部分问题，但这些活动在工程中经常落入不同表示空间和工具链，导致生成结果难以继续定制，客户变体难以随基线升级，文本级补丁噪声过高，扩展点机制又要求平台预先猜测变化位置。

本文提出广义可逆计算（Generalized Reversible Computation, GRC）的系统化表述，把“软件如何被构造并长期演化”视为一个统一的理论问题。GRC 的核心构造关系为 `Y = F(X) ⊕ Δ`：`F(X)` 是由生成器在某一 DSL 语义坐标系中产生的基线结构，`Δ` 是作用于同一或可转换坐标系中的结构化差量，`⊕` 是差量合并与叠加算子。GRC 的关键并不是简单使用补丁，而是把变化所在的表示空间本身视为可设计对象：通过主动设计稳定语义坐标系，使变化能够稀疏表达、独立组合、条件剥离，并在终端统一验证。本文给出 GRC 的核心概念、第一性原理、差量空间质量标准、条件化代数性质、Nop/XLang 参考实现，以及企业级产品线定制场景的案例分析。附录 `proof-v2.md` 对一个受限但非平凡的 GRC/XLang tree-delta calculus 给出严格的结合律证明，为这一范式中的预合并性质提供一个可独立审查的形式支点。

GRC 是一条独立发展的构造路线，与 DOP/FOP/MDE 等工作有交汇，也有不同的出发点。DOP 模块化变化，MDE 组织生成过程，FOP/FeatureHouse 发展结构叠加；GRC 则进一步把变化所在的坐标空间本身作为设计对象，并将生成式基线、DSL 语义坐标、潜在结构差量和阶段化验证纳入同一构造链。

关键词：广义可逆计算，差量代数，语义坐标系，领域特定语言，软件产品线，模型驱动工程，面向差量编程，XLang，XDef，Nop 平台

## 1. 引言

软件工程中的许多长期困难都来自同一个根源：变化无法被稳定地表示和组合。一个软件产品初始版本可以通过代码生成、脚手架或模型驱动工具快速产生；一旦进入客户定制、长期升级和多版本维护阶段，变化往往散落到源码分支、配置开关、插件接口、手写补丁、数据库脚本和构建流程中。结果是标准化资产与客户差异逐渐纠缠，平台升级成本持续上升，所谓复用退化为拷贝、分叉和人工合并。

版本控制系统证明了差量化思想的工程价值，但文本行级差量缺乏领域语义。模型驱动工程提高了抽象层次，但生成结果和后续定制经常脱节。软件产品线工程和 DOP 将变体差异提升为显式模块，但 core 的来源、差量坐标系的稳定性、多 DSL 资产的协同，以及生成式主干与差量机制的统一关系仍需要更一般的构造框架。

GRC 的出发点是：软件构造不应被理解为一次性地从源码构建制品，而应被理解为在一组语义坐标系中不断叠加结构化变化。对于可模型化的软件资产，一个理想构造应满足：

```text
Y = F(X) ⊕ Δ
```

其中 `X` 是源模型或上游模型，`F` 是确定性生成器或加载器，`Y` 是目标模型或最终制品，`Δ` 是封装定制、演化或修复的结构化差量，`⊕` 是在选定结构空间中定义的合并算子。这个公式提出三个相互关联的设计问题：基础结构如何生成，变化在哪个坐标系中表达，变化如何组合并在何处验证。

由此得到本文的中心命题：软件构造理论的核心问题不应只限于“程序如何执行”，还必须回答“变化在哪个语义空间中被表达、组合、验证与剥离”。当这个空间被设计正确时，生成、定制、升级、修复和变体管理可以落入同一构造关系 `Y = F(X) ⊕ Δ`。DSL 的价值不只是抽象语法，而是为未来变化提供稳定坐标；生成器的价值也不只是吐出文本，而是产生仍然可以被差量叠加的可演化基线。

本文的贡献如下：

1. 提出 GRC 的可审查构造命题：软件构造和演化可在主动设计的语义坐标系中统一为 `Y = F(X) ⊕ Δ`，并由此把变化空间设计提升为软件工程的第一类理论问题。
2. 给出差量空间的质量判据，包括坐标稳定性、稀疏表达、结构闭包、结合性、条件可逆性和终端验证边界。
3. 以“最小信息表达”为第一性原理，解释 DSL、生成器、差量、潜在结构空间和阶段化验证为何在同一构造关系中同时出现，并由此说明 GRC 不是某个局部技巧，而是一种复杂性管理方法论。
4. 总结 GRC 的最小代数核，并通过一个明确定义的核心 calculus 给出组合性质证明。
5. 说明 Nop/XLang 如何作为参考实现，将 XDef、XDSL、XNode、`x:extends`、Xpl、虚拟文件系统和 S-N-V 加载流程映射到 GRC 构造关系。
6. 通过企业级产品线定制案例以及与 MDE、FOP、DOP、SPLE、BX、Language Workbench、VCS 等工作的比较，展示 GRC 的解释力、适用边界和方法论位置。

本文主要讨论构造模型、第一性原理、条件化代数性质和参考实现。它的可检验内核是：在稳定语义坐标系和潜在结构空间中，关键差量 calculus 可以获得条件化结合律和可预合并性。Nop/XLang 提供了实践这一构造关系的开源机制和工程样本；工业案例展示适用场景和工程路径。

## 2. 问题：变化空间比变化本身更重要

### 2.1 文本差量的成功和局限

Git 等版本控制系统把软件演化表示为差量，这是现代协作开发的基础。但 Git 的默认坐标系是文件路径和文本行。文本行对于人类协作足够通用，却不是稳定的领域坐标。一次格式化、排序、移动函数或生成器重排都可能产生大量无业务意义的 diff。两个业务上独立的变更也可能因为靠近同一段文本而冲突。

文本差量的问题不在于“差量”概念错误，而在于差量空间质量不足。坐标空间噪声越高，差量越难复用、预合并、审计和自动迁移。

### 2.2 生成与定制的割裂

模型驱动工程和脚手架工具擅长从模型生成代码或配置。但传统生成流程常把生成结果视为最终文件，后续定制要么手工修改生成物，要么通过 generation gap、partial class、hook、插件和模板条件分支等机制绕开生成器。这些机制可以解决局部问题，但它们没有统一回答：生成结果如何继续作为可演化基线被结构化差量作用。

如果生成器只输出文本，而不输出稳定语义坐标系中的结构，那么后续差量只能退回到文本 patch 或预留扩展点。生成和演化仍是两套心智模型。

### 2.3 扩展点机制的预测负担

插件、策略模式、AOP、hook 和特性开关都可以处理变化，但它们通常要求基础平台预先知道哪里会变、以何种方式变。面对长期 ToB 产品线，客户需求往往不是有限可枚举的开关集合，而是在数据模型、流程、界面、规则、权限、集成接口等多个层面持续出现的新差异。

GRC 的判断是：应尽量把整个可模型化资产放入可寻址结构空间，而不是仅在少数位置预埋扩展点。变化应当作用于语义坐标，而不是绕过坐标进入任意代码片段。

### 2.4 核心观察

变化治理的关键不是是否使用 patch，而是 patch 所在的空间。不同空间的差量质量差异很大：

| 差量空间 | 坐标示例 | 优点 | 局限 |
|---|---|---|---|
| 二进制空间 | byte offset | 完备、机械可求 | 人不可读，语义噪声极高 |
| 文本空间 | file + line | 通用、生态成熟 | 坐标脆弱，业务语义弱 |
| AST/FST 空间 | class/method/node | 结构化，优于文本 | 常绑定语言语法或类型层级 |
| 文件系统层 | path + inode-like path | 适合环境和镜像叠加 | 语义粒度仍偏文件系统 |
| DSL 语义空间 | entity/column/step/action | 坐标稳定，业务意义强 | 需要建模和工具支持 |
| CRDT/OT 空间 | operation/id/clock | 并发收敛能力强 | 领域语义需额外建模 |

GRC 选择把设计重点前移：先设计适合未来变化的语义坐标系，再在该空间中定义差量和合并。

扩展点可以理解为预先挑选出来的少数坐标。它们在变化有限且可预见时有效，但当未来变化的自由度超出枚举能力时，预设扩展点会退化为新的瓶颈。GRC 的场式观点是：尽可能让模型整体成为语义坐标场，使每个业务相关元素都拥有稳定地址；未来变化可以离开基础模型独立表达，却仍能通过坐标确定性地重新附着。

## 3. GRC 的基本模型

### 3.1 构造公式

GRC 的基本构造关系为：

```text
Y = F(X) ⊕ Δ
```

其中：

- `X`：源模型、领域模型、配置模型或上游表示。
- `F`：确定性生成器、转换器、解释器或加载器。
- `F(X)`：由生成器产生的基线结构。
- `Δ`：在某个稳定语义坐标系中表达的结构化差量。
- `⊕`：将差量叠加到基线结构上的合并算子。
- `Y`：经过合并、规范化和验证后的目标模型或可运行制品。

该公式允许递归应用：生成器自身、DSL 定义、页面模型、ORM 模型、工作流、报表模板、权限配置和部署配置都可以进入同一构造模式。

这个公式不只是一个工程记号，而是一个方法论判断：对长期演化的软件系统，构造与演化不应再被理解为两套分离的活动，而应被理解为同一结构空间中的两种尺度不同的差量叠加。一个从空基线出发生成完整系统的过程，可以看作一次“创世差量”；一次局部补丁、客户定制或平台升级，则是作用于非空基线的增量差量。于是，构造与演化共享同一语言、同一坐标系和同一组合框架。

在这个意义上，GRC 的核心主张并不是“所有软件都应写成 DSL”，而是：凡是值得长期演化治理的部分，都应尽可能进入一个可寻址、可组合、可验证的语义结构空间。只有在这样的空间里，`Δ` 才不仅是修补历史遗留代码的临时补丁，而是可以被命名、版本化、组合和审计的一等资产。

### 3.2 语义坐标系

语义坐标系是由 DSL 元模型或领域结构提供的一组稳定寻址规则。例如 ORM 模型中的：

```text
/orm/entities/entity[@name='Order']/columns/column[@name='status']/@length
```

这个坐标依赖 `entity.name` 和 `column.name` 等稳定 key，而不是物理行号或数组下标。语义坐标至少应满足：

- 唯一性：被差量作用的领域元素有明确坐标。
- 稳定性：格式化、重排和无关插入不应改变既有坐标。
- 层级性：父子关系可表达删除、替换和递归合并。
- 可规范化：不同文本表象可映射到同一结构坐标。
- 可验证：最终结构可由 DSL schema 或业务规则验证。

一个足够强的坐标系应在概念上支持 `get(path)` 与 `set(path,value)`：凡是需要长期演化的业务相关信息，都应能被稳定寻址和替换。类型系统和类-成员结构是有用的分类机制，但不是充分的坐标系统：多个对象可以具有相同类型，同类列表元素若用数组下标定位会在插入和排序后漂移。因此，GRC 要求通过 DSL 元模型显式声明 stable key，把身份坐标从偶然文本位置和物理顺序中解放出来。

GRC 中 DSL 的首要价值不是“更好看的语法”，而是为演化提供坐标系。

### 3.3 差量

差量 `Δ` 是一个可独立命名、存储、组合和审计的变化对象。它可以表示新增、覆盖、删除、替换、追加、环绕、插入顺序约束和生成式扩展等操作。与普通 patch 不同，GRC 差量的目标是落在设计过的结构空间中。

在代数层，全量和差量没有本质类型区别。一个完整模型可以看作从空模型出发的“创世差量”：

```text
M = ∅ ⊕ M
```

工程上仍保留“差量”一词，是因为它通常是稀疏的，只表达相对于某个基线的少量变化。

更强的工程要求是：差量与全量应尽量在同一坐标空间中同构表达，并由同一 schema 约束。这样，“差量的差量”仍然是普通差量，客户定制、产品升级和局部修复都可以被版本化、组合、审计和剥离。GRC 因此把变化从分支、hook、条件判断和临时 patch 中释放出来，变成可命名、可治理的一等资产。

### 3.4 生成器

生成器 `F` 不限于传统代码生成器。凡是将输入模型转换为目标结构的确定性机制，都可以视为 GRC 中的生成器，包括模板、编译器、解释器、模型转换器和资源加载器。

GRC 对生成器的要求是：在给定输入、配置和外部资源版本时，输出应可被视为同一语义坐标系中的确定结构。如果生成器读取当前时间、随机数或未版本化外部状态，则它不属于本文代数性质的前提，除非这些状态被显式纳入输入。

### 3.5 “可逆”的含义

GRC 中的可逆不是运行时指令级双射，也不是承诺所有业务操作都能无损回滚。它是一组构造期能力：

- 结构可组合：差量可以在潜在结构空间中预合并。
- 局部可剥离：在记录前像和删除证据时，可构造补偿差量恢复局部结构。
- 表象可往返：DSL、可视化表单、Excel、JSON、XML 等表象可在语义等价意义下转换。
- 副作用可补偿：外部不可逆边界通过证据对象和补偿操作治理。

因此，“可逆”在本文中是设计目标和分级工程承诺，而不是全局无条件逆元声明。

更准确地说，GRC 所说的“可逆”不是对物理时间或运行时状态演化的声明，而是对构造关系的一组分级承诺：

- 在代数层，差量应尽量可组合、可预合并、可分析。
- 在表达层，不同表象应尽量围绕同一语义内核实现条件化往返。
- 在过程层，后来者应能以结构化差量修正先前构造结果，而不必把整个历史重写为新的手工分叉。
- 在边界层，真正不可逆的外部副作用要被显式隔离、记录和补偿，而不是被伪装成“普通模型合并”。

因此，GRC 中的“可逆”更接近一种复杂性治理原则：它要求把可组合的部分尽量组织成可逆核，把不可逆的部分明确压缩到边界，而不是让二者在整个系统中无差别扩散。

## 4. 差量空间的质量标准

一个差量机制是否具有工程价值，取决于其所在空间的质量。本文提出以下判据。

### 4.1 坐标稳定性

差量的坐标应随业务身份稳定，而不是随文本位置、数组下标或生成顺序漂移。同级可重复节点应由 schema/XDef 明确指定 stable key，或由通用 XNode 规则中的 `x:key-attr`、`x:unique-attr`、`x:id` 等 identity 机制定位。`name`、`id`、`value` 只有在 schema 或通用 fallback 规则把它们声明为身份字段时才是 stable key，不能把 `value` 理解为无条件默认 key。

### 4.2 最小信息表达

稀疏差量是最小信息表达原则在演化问题上的体现。一个表达应当对领域本质复杂性保持完整，同时尽量删除由通用语言、框架细节、执行顺序和环境偶然性引入的技术噪声。DSL 通过只使用领域概念表达领域结构来降低偶然复杂性；差量则通过只表达从已知基线到目标状态所需的信息来降低演化噪声。

在这个意义上，`Δ` 不是“小补丁”的工程习惯，而是已知 `Base` 后描述 `App` 的最经济形式：

```text
App = Base ⊕ Δ
```

如果重复描述整个 `App`，就把已有基线中的信息再次复制进变化说明，增加了同步和冲突成本。

这一定义在本文中作为第一性原理使用。它同时解释了三件事：

1. GRC 倾向于 DSL，是因为只有当表达空间足够贴近问题空间时，系统才可能“表达且仅表达需要表达的信息”。
2. GRC 把差量提升为核心实体，是因为在已知基线存在时，最经济的演化表达通常不是重写整体，而是表达相对于基线的结构差异。
3. GRC 强调潜在结构空间与终端验证边界，是因为中间阶段需要暂存那些对后续组合仍有意义、但尚未进入最终可观测形态的证据；过早压缩这些信息会破坏组合性质。

从这个角度看，代数性质不是外加装饰，而是构造空间质量的表现：当一个空间能够保留必要信息、降低噪声并支持稳定组合时，它才适合承担长期演化治理。

由此可得到四个可检查的设计判据：

1. 坐标压缩：同一业务变化是否可以在不依赖物理位置和偶然顺序的前提下被稳定寻址。
2. 差量压缩：给定基线后，变化描述是否只需表达新增信息，而不必复制大段未变结构。
3. 组合保真：多个差量在预合并过程中是否能够保留对后续语义仍然必要的证据，而不是过早丢失信息。
4. 表象压缩：不同表象之间是否围绕同一语义内核组织，而不是各自携带大量无法对齐的冗余细节。

若一个差量空间在这四点上持续表现更好，就可以说它比另一个空间更接近 GRC 所要求的最小信息表达。这个判据既是方法论判断，也为后续经验研究提供了可操作的比较方向。

### 4.3 稀疏表达

差量应只表达必要变化。若一个客户定制只调整一个流程步骤，就不应复制整条流程。稀疏性可以用受影响坐标数与基线坐标数的比例粗略度量。

### 4.4 结构闭包

合并应在结构空间内产生一个潜在模型，而不是产生无法解析的冲突文本。业务一致性可以在终端验证阶段判定，但结构合并本身应尽量封闭。

### 4.5 可预合并性

多个差量应能在不立即依赖具体基线细节的情况下预合并。结合律是该能力的核心代数性质：

```text
(A ⊕ B) ⊕ C = A ⊕ (B ⊕ C)
```

在本文采用的潜在结构空间中，后文附录给出一个受限核心的证明。

### 4.6 条件可逆性

删除和覆盖如果不记录前像，一般不可无损恢复。因此 GRC 不把可逆性说成无条件群结构，而是要求在需要剥离或回滚的场景中显式保留前像、tombstone、顺序约束和规范化证据。

### 4.7 终端验证边界

结构合并阶段允许短暂存在业务上不完整的潜在模型。最终模型必须经过规范化、投影和验证。这个阶段分离避免了中间状态过早验证破坏结合律。

## 5. 代数核与证明边界

### 5.1 形式化对象

GRC 首先是一种构造方法论，但它需要一个可审查的形式核心来支撑预合并、结构闭包和终端验证等性质。本文因此区分两个层次。

第一层是方法论命题：GRC 主张软件构造应优先在稳定语义坐标系中组织，并以生成式基线、结构化差量、潜在结构空间和终端验证边界来治理复杂性。

第二层是形式命题：在上述方法论所选出的若干核心 calculus 中，可以证明条件化结合律与可预合并性。附录 `proof-v2.md` 给出的结果属于这一层：它证明了一个受限、线性化的 GRC/XLang tree-delta calculus 在语义商空间上的结合律。

### 5.2 直觉核：潜在模型与覆盖合并

令 `C` 为带前缀关系的语义坐标集合。每个坐标 `c` 有值域 `V_c`，并引入删除标记 `⊥`。潜在模型是有限偏函数：

```text
p: C ⇀ ⋃(V_c ∪ {⊥})
```

若 `p(c)` 未定义，表示该潜在模型没有在坐标 `c` 上表达信息。若 `p(c)=⊥`，表示删除意图。潜在模型空间允许暂时保存 tombstone、virtual node 和顺序约束等中间证据。

### 5.3 核心覆盖合并

核心合并算子 `⊕` 定义为右侧覆盖：

```text
(p ⊕ q)(c) = q(c), if c ∈ Dom(q)
(p ⊕ q)(c) = p(c), if c ∉ Dom(q) and c ∈ Dom(p)
```

空潜在模型是单位元。逐坐标分情况可证明：

```text
(p ⊕ q) ⊕ r = p ⊕ (q ⊕ r)
```

该性质使核心覆盖差量链可以先规约为单一差量，再作用到基线。

### 5.4 删除与最终投影

删除不应在中间合并步骤中被物理丢弃。最终投影 `Pr` 根据删除闭包去除被删除坐标及其后代。结合律保证的是：

```text
Pr((p ⊕ q) ⊕ r) = Pr(p ⊕ (q ⊕ r))
```

一般不保证每一步投影后继续合并仍等价。若 `Pr` 在中间执行并丢弃 tombstone，后续差量可能失去删除证据。

### 5.5 从直觉核到树形 calculus

XDSL 树可编码为语义坐标上的有限偏函数：节点存在、属性、文本、children stable key 和顺序约束都映射为坐标或潜在证据。这一坐标级直觉核解释了“保留潜在证据、延后终端投影”为何对组合性质至关重要：tombstone、virtual node 和顺序约束在中间阶段仍可能影响后续差量的解释。

完整树语义还需要处理单坐标覆盖无法表达的情形。`merge` 是对子树递归合并；`remove` 表达删除某个 stable key 匹配到的节点；`replace` 是子树级常量操作，表示用新子树整体替换旧子树，并清除旧子树中未被新子树声明的内容；`bounded-merge` 还会在祖先路径上支配整棵遗漏子树的保留/裁剪语义。因此，前述坐标级 calculus 是一个直觉核；树级操作则需要进入附录所定义的潜在树状态空间和递归 tree-delta carrier。

对于 `replace/remove/merge` 混合、ancestor/descendant 重叠、around/super、append 和顺序插入等扩展操作，GRC 的一般方法是两步走：先把它们放回潜在结构空间，再决定哪些可被局部幺半群化，哪些必须作为递归树操作或确定性解释函数处理。附录对其中一个受限但非平凡的 tree-delta 子片段完成了严格证明，为该方法提供了一个代表性核心。

这一安排形成了自然的研究层次：统一视角给出空间设计原则，代表性 calculus 给出形式支点，后续工作可以继续扩展到更多工程变体。

具体到附录，目前被严格证明的对象是：固定线性顺序、只改变括号而不改变顺序、children 依 stable key 匹配、override 限于 `remove`、`replace`、`merge`、`bounded-merge` 的 tree-delta calculus。

顺序插入、around/super 和更多 DSL 特有 override 可沿同一框架逐步纳入。

由此得到三层相互衔接的研究对象：

| 层次 | 当前状态 | 代表问题 |
|---|---|---|
| 抽象方法论 | 本文主张并论证 | 为什么要先设计变化空间，而不是事后修补变化机制 |
| 严格核心 calculus | 附录已证明 | 受限 tree-delta 片段在语义商空间上的条件化结合律 |
| 实现符合性 | 后续展开 | 真实 XLang/Nop child 匹配、`empty_p` 语义、顺序规约和各 override 分支是否逐项对齐附录 carrier |

这三个层次共同构成本文的理论位置：正文给出方法论与直觉核，附录证明一个代表性严格核心，完整实现语义则沿着同一 carrier 继续细化。

在操作层面，不同扩展进入形式体系的方式也不同：

- 单坐标扩展操作嵌入局部幺半群。
- 跨坐标和混合操作解释为潜在状态空间上的确定性端函数，组合来自函数复合。

### 5.6 定理陈述

在以下条件下，本文所讨论的核心 GRC calculus 满足条件化结合律并支持差量预合并：

1. 参与合并的模型和差量位于同一稳定语义坐标系，或可确定性转换到同一潜在状态空间。
2. 合并链在潜在结构空间中完成，中间不丢弃 tombstone、virtual node、顺序约束等证据。
3. 扩展操作已被定义为局部幺半群操作或确定性端函数。
4. 规范化、投影、排序和业务验证在合并链结束后统一执行，且这些后处理过程确定。
5. 生成器输出确定，且可编码到参与合并的结构空间。

在这些条件下，任意括号化得到同一潜在结果；经过确定性终端处理后，得到同一可观测模型或同一验证失败。对树级 `remove/replace/merge/bounded-merge` 的严格版本，请以附录中的受限 tree-delta calculus 为准。

## 6. Nop/XLang 参考实现

Nop 平台是 GRC 的一个参考实现。它不是把 GRC 作为额外插件附加到已有框架，而是在语言和加载机制层面将 DSL、生成器和差量合并统一起来。

### 6.1 XNode：统一结构层

Nop 将 XML、JSON、YAML 等 XDSL 解析为统一的 `XNode` 树结构。差量合并不直接发生在强类型 Java 对象层，而先发生在统一结构层。这样不同 DSL 可以共享同一套合并机制。

退回到统一结构层有三个效果：

- 合并算法与具体业务语义解耦。
- 所有 DSL 都可以复用 `x:extends` 等差量能力。
- 最终业务合法性由后续 XDef 规范化和验证判定。

### 6.2 XDef：DSL 的元模型

XDef 是定义 XDSL 的元 DSL。它描述节点结构、属性、约束、stable key、对象映射和工具链提示。通过 XDef，新的 DSL 不需要从零实现解析、校验、IDE 支持和差量机制。

GRC 中“语言即坐标系”的工程落点就在 XDef：schema 不只是类型检查工具，也是坐标稳定性的来源。

### 6.3 `x:extends`：内置差量合并

`x:extends` 表示当前模型声明其基础模型。执行语义是先加载基础模型，再把当前模型作为差量叠加：

```xml
<orm xmlns:x="/nop/schema/xdsl.xdef"
     x:extends="base.orm.xml"
     x:schema="/nop/schema/orm/orm.xdef">
  <entities>
    <entity name="Order">
      <columns>
        <column name="status" length="40" />
      </columns>
    </entity>
  </entities>
</orm>
```

其中 `entity[@name='Order']` 和 `column[@name='status']` 是由 XDef 指定的稳定坐标。差量只表达 `status.length` 的变化，而不复制整个 ORM 模型。

Nop 中的 stable key 不是简单约定所有 `name` 或 `id` 属性都有身份语义，而主要由 XDef 明确声明。例如列表容器可通过 `xdef:body-type="list"` 与 `xdef:key-attr="name"` 指定子元素匹配键，具体节点也可通过 `xdef:unique-attr` 声明唯一属性。对于缺少 schema 的通用 XNode，父节点可用 `x:key-attr` 指定 children key，节点也可通过 `x:unique-attr`、`x:id` 等提供 identity。因此，stable key 是 DSL 元模型的一部分，而不是 XML 属性名的偶然约定。

有序列表不是通过物理下标定位，而是通过 stable key 加顺序约束处理。在支持该语义的 Nop/XLang 规范化路径中，`x:before` / `x:after` 可被规约到通用 merge 语义：它们表达当前节点与目标 key 的相对顺序，若多个差量最终指向同一个 stable key，合并后仍只保留一个节点，并由确定性规约给出最终位置。因此，工程实现的关键在于保持顺序规约过程的确定性。

删除通过结构化标记表达：

```xml
<column name="status" x:override="remove" />
```

“保留删除意图”是语义要求，不是要求最终 XNode 中可见地保存 tombstone。为了支持差量预合并，系统必须能区分“未声明”与“声明删除”，并且在预合并、顺序规约、替换规约完成前不能把影响后续语义的证据有损丢弃。具体实现可以使用内部操作对象、端函数规约、一次性线性合并或不可观察的临时标记；合并完成并通过 XDef 规范化后，`x` 命名空间的合并指令应从最终运行期模型中消失。

### 6.4 Xpl：生成器

Xpl 是可在加载期执行的模板和元编程语言。`x:gen-extends` 和 `x:post-extends` 可以把生成结果作为差量参与合并。由此，生成器不再只是一次性吐出文本，而是产生可继续被差量叠加的结构。

在 Nop/XLang 中，`x:extends` 不是普通面向对象继承，而是加载期结构差量合并声明。Nop 的 Map/Set 在相关解析路径上保持解析顺序，因此多值 `x:extends` 具有确定的线性化顺序。若当前模型声明 `x:extends="A,B"`，并且 `x:gen-extends` 生成 `C,D`，`x:post-extends` 生成 `E,F`，则等价合并顺序是：

```text
A ⊕ B ⊕ C ⊕ D ⊕ currentNode ⊕ E ⊕ F
```

也就是说，`x:gen-extends` 位于当前节点之前，常用于生成基线；`x:post-extends` 位于当前节点之后，常用于对设计器产物或手写模型做统一增强。这一机制把“生成器输出”和“人工差量”放入同一结构合并链中，而不是让生成结果退化为不可继续演化的文本。

示例：

```xml
<orm x:schema="/nop/schema/orm/orm.xdef">
  <x:gen-extends>
    <pdman:GenOrm src="/demo/model/app.pdma.json"
                  xpl:lib="/nop/orm/xlib/pdman.xlib" />
  </x:gen-extends>

  <entities>
    <entity name="Order">
      <columns>
        <column name="remark" length="200" />
      </columns>
    </entity>
  </entities>
</orm>
```

这里 `pdman:GenOrm` 生成基线 ORM 结构，当前文件继续以差量方式修改生成结果。

### 6.5 加载器作为生成器

许多框架本身通过加载配置文件来构造运行时对象。GRC 将资源加载器视为生成器：加载器先完成基础模型与差量模型的合并，再把最终模型交给原框架。这提供了低侵入采纳路径。

例如，对一个配置驱动框架，无需重写框架核心，只需提供差量感知加载器：

1. 根据当前 `deltaId` 和虚拟文件系统定位基础文件和差量文件。
2. 将 XML/JSON/YAML 解析为 XNode。
3. 执行 `x:extends`、`x:gen-extends` 和结构合并。
4. 规范化并验证最终模型。
5. 把最终模型交给既有框架处理。

Nop 对 GRC 最关键的工程化表达之一是 Delta VFS。客户或环境差量可以放在 `_vfs/_delta/{deltaId}/...` 下，并保持与基础资源相同的虚拟路径；Delta 文件通常通过 `x:extends="super"` 引用被覆盖层。这样，客户定制不是复制基础文件，也不是修改生成物，而是在同一虚拟路径和同一 XDef 坐标系中声明结构差量。平台升级时，新的基础层重新加载，客户 Delta 仍按相同坐标叠加，最终由规范化和验证阶段暴露不兼容点。

### 6.6 S-N-V 阶段分离

Nop 的加载流程可概括为 S-N-V：

| 阶段 | 名称 | 作用 |
|---|---|---|
| S | Structure Merge | 解析结构、执行差量合并和生成式扩展，保留潜在证据 |
| N | Normalization | 展开简写、应用默认值、收集和规约顺序约束，产生规范结构 |
| V | Validation/Compilation | 执行业务验证并编译为运行期使用的强类型静态模型 |

S-N-V 是 GRC 代数性质的工程边界。结构合并阶段允许“虚时间”中的暂时不一致；业务合法性只在终端统一判定。Delta 计算被压缩到加载期或编译期完成，运行时面对的是已规范化和验证后的普通静态模型，不需要知道差量历史。

## 7. 工业产品线案例

### 7.1 场景

某大型金融核心系统需要以同一标准产品为基础交付给多个客户。各客户在下单流程、风控规则、数据字段、界面展示和外部系统集成上存在差异。传统做法通常在标准分支之外维护客户分支，或在主干代码中累积条件判断、策略接口和扩展点。

这种做法的问题是：

- 客户差异散落在代码和配置中，难以独立审计。
- 标准产品升级时，每个客户分支都要人工合并。
- 新变化经常落在未预设扩展点的位置。
- 运行时条件分支增加了测试矩阵和行为理解成本。

### 7.2 GRC 重构方式

GRC 的处理方式是先把可模型化部分提升为 DSL 坐标系。例如，下单流程被外部化为任务流模型：

```yaml
name: placeOrder
steps:
  - name: creditValidation
    bean: validateCreditStep
    when: "order.customer.isVip()"

  - name: promotionApplication
    bean: applyPromotionStep


  - name: stockChecking
    bean: checkStockStep

  - name: statusFinalization
    bean: finalizeStatusStep
```

对于客户 A 的定制，不修改标准流程文件，而提供差量模型：

```yaml
"x:extends": /flows/placeOrder.task.yaml
steps:
  - name: creditValidation
    x:override: replace
    bean: customerAValidateCreditStep

  - name: customerAFraudCheck
    bean: customerAFraudCheckStep
    "x:after": stockChecking
```

其中 `x:after` 不是运行期插入动作，而是加载期顺序约束；在支持该语义的规范化路径中，它会并入列表 merge，最终仍以 `name` 等 stable key 合并为唯一节点。其严格结合律依赖顺序约束收集和排序规约的确定性。

在客户 A 环境中，加载器根据 `deltaId` 合并标准模型与客户差量，得到最终任务流。标准流程仍保持只读，客户定制以独立差量资产存在。

### 7.3 工程效果

该模式带来以下效果：

- 标准基线和客户定制分离，客户差异以 `Δ` 文件版本化。
- 客户差量以流程步骤名、字段名、页面组件名等语义坐标定位。
- 基线升级时，客户差量可重新叠加到新基线，并由终端验证暴露不兼容点。
- 运行时不需要执行大量客户条件分支，加载期即可剪裁出目标模型。
- 变化审计从搜索代码分支变成检查差量文件集合。

该案例说明的是一种构造模式：流程级客户差异可以从主干代码和客户分支中移出，表达为可独立版本化的结构差量；这些差量在加载期叠加到标准模型，并在最终验证阶段暴露不兼容点。更系统的经验研究可进一步量化差量稀疏度、升级复用率和合并冲突变化。

从全文主轴看，案例同时体现了 GRC 的两条母题。第一，构造即演化：客户交付不是对标准主干的旁路修改，而是在同一构造链中的增量叠加。第二，分级可逆性：结构层允许保留潜在差量并延后验证，而组织边界、部署边界和业务副作用则在更外层治理。

### 7.4 可复核材料

除商业案例外，本文的核心机制可通过开源材料复核：

- Nop 平台源码提供 XDef、XDSL、XNode、`x:extends`、Xpl 和加载期差量合并实现。
- 本仓库中的 `docs/theory/proof-v2.md` 给出自完备的结合律附录证明。
- 多个 Nop 模块中的 ORM、页面、任务流、报表和配置模型可作为 DSL 坐标系与差量机制的工程样本。

这些材料展示了 GRC 的三个层次如何在同一开源系统中落地：

1. 方法论层：XDef 把 DSL 元模型同时变成类型约束与 stable-key 坐标定义。
2. 构造层：`x:extends`、`x:gen-extends`、Delta VFS 与加载器作为生成器把 `Y = F(X) ⊕ Δ` 变成统一加载链。
3. 证明层：附录对其中一个核心 tree-delta 片段给出严格结合律证明，为预合并安全提供形式支撑。

这些材料分别对应概念解释、结构边界和形式证明三个层面的支撑。

### 7.5 失败边界

GRC 的适用性依赖变化能否进入稳定语义空间。以下情况通常需要额外机制：

- 无法稳定建模的一次性需求。
- 依赖外部不可逆副作用的运行时操作。
- 需要强并发编辑且没有 CRDT/OT 语义的差量空间。
- 生成器输出不稳定或依赖未版本化环境。
- DSL 坐标设计错误，例如列表元素没有 stable key。

这些情形说明，GRC 的代数性质来自结构空间设计，而不是来自任意变化本身。

## 8. 相关工作

本节比较围绕六个问题展开：变化是否是一等资产，坐标系是否被主动设计，生成结果是否仍可被差量寻址，全量与差量是否共享表示，潜在状态与终端投影边界是否显式，以及是否存在跨 DSL 的统一实现层。

| 维度 | MDE | FOP/FeatureHouse | DOP | BX/Lenses | Language Workbench | GRC |
|---|---|---|---|---|---|---|
| 生成式基线 | 中心能力 | 非核心 | 通常给定 core | 转换相关 | 可支持 | 中心能力 |
| 变化一等资产 | 弱/外置 | feature module | delta module | edit/update | 非中心 | delta asset |
| 主动坐标设计 | 模型层部分支持 | FST/语法树 | 目标语言或模型结构 | 对齐/trace | AST/metamodel | DSL 语义坐标 |
| 全量/差量同构 | 通常不同 | 通常不同 | 通常不同 | 视 edit calculus 而定 | 通常不同 | 同一潜在模型空间 |
| 潜在状态/终端投影 | 非中心 | 部分隐含 | 部分隐含 | 问题不同 | 非中心 | 显式 S-N-V 边界 |
| 跨 DSL 合并层 | 非中心 | FST 路线 | 非核心 | 非目标 | 语言组合 | XNode/XDef |

### 8.1 软件产品线工程

软件产品线工程（SPLE）通过领域工程和应用工程管理产品家族共性与差异。GRC 与 SPLE 目标一致，但关注更底层的构造机制：如何把可变性表达为同构的结构化差量，并在 DSL 坐标系中叠加到生成式基线上。

### 8.2 Feature-Oriented Programming 和 FeatureHouse

FOP 将功能特征模块化。FeatureHouse 进一步提出语言无关的 Feature Structure Tree 和 superimposition，把多类 artifact 纳入统一树合并。FST 已经揭示了坐标洞察：路径提供地址，一旦地址固定，合并就可以泛化。GRC 与其共享“树结构可作为组合空间”的认识，但 GRC 更强调差量与全量同构、删除和替换的潜在证据、生成器与差量的统一关系，以及由领域 DSL 元模型主动设计坐标，而不仅仅依赖 artifact 语法树。

### 8.3 Delta-Oriented Programming

DOP 将变化表示为 delta module，并引入 modify/remove 等操作来构造产品线变体。DOP 是 GRC 最接近的学术相关工作之一。简言之，DOP 使变化模块化；GRC 使变化的坐标系本身显式化并可设计。

GRC 与 DOP 的差异主要在以下方面：

| 维度 | DOP | GRC |
|---|---|---|
| 基本公式 | Product = Core + selected Deltas | Y = F(X) ⊕ Δ |
| Core 来源 | 通常作为给定核心产品或纯 delta 组合结果 | 由生成器和 DSL 构造的基线 |
| 坐标系 | 常由目标语言、类结构或模型结构承载 | 明确要求主动设计 DSL 语义坐标系 |
| 差量/全量关系 | delta module 与产品通常语法不同 | 代数层全量和差量同处潜在模型空间 |
| 删除语义 | 支持 remove 等操作 | 删除保留 tombstone，并区分潜在合并与终端投影 |
| 多 DSL 协同 | 不是核心问题 | DSL 图谱和生成链是核心问题 |
| 实现路径 | DeltaJ、DeltaEcore 等 | XDef/XDSL/XLang/Nop 参考实现 |

DOP 主要形式化 delta module 路线，而 GRC 从生成器、语义坐标系、潜在结构空间和阶段化验证出发，试图给出一个更一般的构造框架。二者可被理解为围绕差量化构造问题形成的不同技术路线。

从全文主轴回看，相关工作的分水岭不在于是否“也用了差量”，而在于它们是否同时回答两个问题：构造是否与演化共享同一框架，可逆性是否被组织为一个分级承诺。沿着这两条轴线，GRC 与 MDE、FOP、DOP、BX、Language Workbench 等工作既有重叠也有清晰差别。

### 8.4 模型驱动工程

MDE/MDA 将模型作为主要信息源，通过模型转换和代码生成提高一致性。GRC 接受生成式构造的价值，但指出生成结果必须继续落在可差量叠加的结构空间中。否则，生成与定制仍然割裂。

### 8.5 Language Workbench

语言工作台提供 DSL 定义、编辑器和语言组合能力。GRC 与其都重视 DSL，但 GRC 的重点不是投影编辑或语法定制，而是 DSL 作为变化坐标系以及统一结构层上的差量代数。XDef 可以看作一种面向差量构造的轻量元 DSL 路线。

### 8.6 Bidirectional Transformation 和 Lenses

BX/Lenses 研究相关模型之间的更新传播，关注 `get/put` 的一致性律。GRC 的问题范围不同：它不试图解决任意两个既有模型之间的同步，而是主张预先设计稳定坐标系，并在此基础上组织生成、差量和终端验证。Lenses 可用于 GRC 的表象往返子问题，但不是 GRC 的整体构造模型。

### 8.7 VCS、Kustomize、Docker 和 OpenUSD

Git、Kustomize、Docker layer 和 OpenUSD layer 都展示了差量叠加的工程价值。GRC 将它们视为不同差量空间中的成功实例，而不是 GRC 的完整实现。它们说明“基础 + 层/补丁/意见”的结构在多个领域独立出现；GRC 进一步把该结构提升为软件构造的显式设计原则。

### 8.8 XVCL 和 Frame Technology

XVCL/Frame Technology 通过 frame、slot 和适配机制表达相似制品之间的变化。它们与 GRC 都重视“相似但不同”的复用模式。GRC 的差异在于不依赖手工插入 slot 作为唯一扩展点，而是通过 DSL 元模型和 stable key 将整个模型结构转化为可寻址坐标空间。

## 9. 讨论与限制

### 9.1 模型化成本

GRC 的收益依赖模型化。对于生命周期短、一次性、需求尚未稳定的脚本，构造 DSL 坐标系可能不经济。GRC 更适合长期演化、变体较多、需要标准化交付的软件产品线和平台型系统。

### 9.2 坐标设计失败

如果 DSL 使用物理下标定位列表元素，或缺少业务稳定 key，差量坐标会漂移。此时 GRC 的代数性质不成立。坐标设计是 GRC 的核心工程责任。

### 9.3 生成器确定性

生成器必须在给定输入下确定。若生成器隐式读取当前时间、随机数、外部服务状态或未版本化文件，则构造关系会退化为过程脚本，无法享有本文证明的性质。

### 9.4 不可逆边界

删除无前像、外部副作用、数据库迁移和第三方系统调用都可能不可逆。GRC 对这些边界的策略是记录证据、构造补偿，并把可组合的结构核与不可逆的外部边界分离。

### 9.5 并发差量

本文主要讨论线性化差量链。并发编辑需要 CRDT、OT、三方合并或领域特定冲突规约；这些机制可以作为特定子差量空间接入 GRC，而不是由核心覆盖语义自动推出。

### 9.6 经验评估

本文的案例说明了工程可行性。未来研究可进一步报告差量稀疏度、合并成功率、升级复用率、缺陷率和维护成本等指标，并比较长期产品线系统与一次性应用中的适用差异。

## 10. 结论

GRC 提出一种面向软件构造与演化的统一视角：通过主动设计 DSL 语义坐标系，将生成式基线和结构化差量放入同一构造关系 `Y = F(X) ⊕ Δ`。它的核心贡献不是发明“补丁”或“代码生成”，而是把变化空间的设计提升为软件工程的第一类问题。

附录中的形式结果为这一方法论提供了一个可审查的核心支点：一个受限但非平凡的 GRC/XLang tree-delta calculus 在语义商空间上满足条件化结合律。该结果说明，只要坐标设计、潜在结构空间与终端验证边界处理得当，差量预合并、客户定制复用、基线升级和加载期模型剪裁就可以获得明确的形式支撑。Nop/XLang 通过 XDef、XDSL、XNode、`x:extends`、Xpl 和 S-N-V 加载流程提供了一个系统实现，展示了如何把 GRC 从构造公式落到工具链。

GRC 与 DOP/FOP/MDE 等工作并非简单替代关系。DOP 将差量模块引入软件产品线构造，MDE 强调模型到制品生成，FOP/FeatureHouse 发展了语言无关结构合并。GRC 试图把这些方向统一到“生成器 + 语义坐标系 + 差量代数 + 阶段分离”的构造框架下。对于需要长期演化和高度定制的软件产品线，GRC 提供了一条可形式化、可实现、可渐进采纳的系统化路径。

在这一视角下，软件可以被重新理解为存在于多个语义坐标系中的可演化结构：生成器生产基线，差量携带变化，合并代数把二者组织为可治理的软件生命史。若这一视角成立，那么“如何设计变化空间”将不再是实现细节，而会成为下一代软件构造理论的核心问题之一。
