# Nop AI Agent 安全与权限设计

## 1. 目标

本篇定义 Agent 的权限、安全边界和访问控制模型，回答下面问题：

1. Agent 如何限制可调用工具
2. Agent 如何限制可访问文件
3. 子 agent 如何继承且不能提升权限
4. 哪些访问控制必须在程序层保证，而不能依赖 prompt

## 2. 设计原则

安全设计遵循四条原则：

1. 权限默认收敛
2. 程序校验优先于 prompt 约束
3. 子 agent 只能继承或收缩权限，不能提升权限
4. 文件路径和系统目录必须有明确保护边界

## 3. 权限模型

建议权限模型至少包含两层：

- 工具权限
- 文件访问权限

必要时可扩展：

- 子 agent 调用权限
- shell 环境限制
- 网络访问权限

## 4. 权限配置

第一阶段建议保留配置化权限方案。

推荐结构：

```yaml
agent: coder
extends: agent-default

tools:
  allowed: [read-file, write-file, call-agent]
  denied: [delete-file]

file-access:
  - pattern: ".nop/**"
    access: read-only
```

语义：

- `extends`
  - 允许继承默认权限模板
- `tools.allowed`
  - 显式允许的工具
- `tools.denied`
  - 显式禁止的工具
- `file-access`
  - 对文件路径的访问级别定义

推荐配置来源：

- 默认权限：`.nop/.permissions/agent-default.yml`
- 特定 agent 权限：`.nop/.permissions/agent-{name}.yml`
- 主配置：`.nop/config.yml` 或等价主配置入口

推荐优先级：

1. 运行时显式参数
2. 环境变量
3. Session 级临时配置
4. Agent 专属配置
5. 默认配置

热加载不应作为第一阶段强承诺。更稳妥的策略是：

- 第一阶段只要求配置加载和校验
- 是否支持热加载留待后续实现再正式定义

## 5. 工具权限

### 5.1 检查时机

工具权限必须在每次工具执行前检查。

### 5.2 推荐规则

建议顺序：

1. 先匹配 deny
2. 再匹配 allow
3. 未命中 allow 时默认拒绝

### 5.3 为什么不能只靠 prompt

因为：

- 模型可能幻觉工具名
- prompt 不能替代安全边界
- 安全规则必须可测试、可审计、可复现

## 6. 文件访问权限

### 6.1 访问级别

建议保留三种基本级别：

- `read`
- `read-only`
- `deny`

### 6.2 路径匹配

推荐使用基于 glob 的顺序匹配：

- 从上到下匹配
- 第一条命中的规则生效

### 6.3 路径规范化

所有文件访问都必须先规范化路径，再做权限判断。

这一步必须防御：

- `..` 目录穿越
- 符号链接绕过
- 绝对路径逃逸

## 7. 系统目录保护

建议保留下面的默认保护思路：

- `.nop/**` 默认只读
- `.nop/.permissions/**` 默认拒绝访问
- `.nop/.system/**` 默认只读

其中 `.nop/.permissions/` 既是配置来源目录，也是对普通 agent 默认不可见的受保护目录。

更一般地说，系统控制目录和权限配置目录必须与普通工作目录隔离。

## 8. Subagent 权限继承

### 8.1 基本规则

子 agent 只能继承父 agent 的权限，不能提升权限。

### 8.2 继承语义

推荐语义：

- 工具权限 = 父权限与子配置的交集或收缩
- 文件权限 = 父权限与子配置的交集或收缩
- 未明确授权的提升行为一律拒绝

### 8.3 为什么必须这样

因为 `call-agent` 本质上是能力扩张入口。如果子 agent 可以提升权限，那么任意 prompt 都可能绕过父 agent 的约束。

## 9. 与工具验证的关系

权限检查和工具验证是不同层次：

- 权限检查
  - 你能不能调用这个工具或路径
- 工具验证
  - 你传入的参数是否合法、安全、符合 schema

这两层都必须存在。

## 10. Shell 与高风险工具

对 shell、http 请求、文件写入、删除类工具，建议额外做程序级限制：

- 允许工作目录白名单
- 环境变量白名单
- 输出大小限制
- 命令超时
- 进程组清理

这些约束不应交给 prompt 决定。

## 11. 日志与审计

安全相关执行建议记录：

- sessionId
- agentId
- toolName
- 访问路径
- 权限命中规则
- 拒绝原因

这样可以支持排查、回放和安全审计。

## 12. 第一阶段建议保留的安全能力

建议第一阶段正式保留：

1. tool allow/deny
2. file-access rule
3. path normalization
4. system dir protection
5. subagent 不可提权
6. 高风险工具的程序级限制

## 13. 本篇结论

本篇固定了 Agent 安全与权限的第一阶段边界：

- 工具权限和文件访问控制必须程序化执行
- 子 agent 只能继承或收缩权限，不能提升权限
- 系统目录和权限目录必须有默认保护
- 路径规范化和高风险工具限制属于硬边界

这些不是 prompt 约定，而是运行时必须强制执行的安全策略。
