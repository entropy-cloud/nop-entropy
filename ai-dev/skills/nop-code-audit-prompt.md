# Nop-Code 模块审计提示词

> 此文件是通用的可复用审计提示词模板，用于对 nop-code 模块进行系统性代码质量审计。
> 每次审计的具体发现应记录到 `ai-dev/audits/` 目录下的独立报告中。

## 审计目标

对 nop-code 模块从多个维度进行系统性审计，识别架构、设计、实现和质量层面的问题。审计应：
- 跳出当前实现的限制思考
- 对照 Nop 平台的约定和最佳实践
- 给出具体的改进方向和优先级

## 模块概述

nop-code 是 Nop 平台的代码分析与索引模块，提供符号搜索、类型层级、调用链、图分析等能力。

模块组成（审计时需确认实际结构）：
- `nop-code-core`：核心分析层
- `nop-code-dao`：ORM 实体与数据访问
- `nop-code-meta`：xmeta 元数据定义
- `nop-code-service`：业务逻辑层
- `nop-code-web`：前端页面
- `nop-code-app`：应用入口
- `nop-code-api`：API 接口定义
- `nop-code-codegen`：代码生成
- `nop-code-lang-*`：多语言解析器

## 审计维度与检查清单

### 维度一：架构分层与模块边界

#### 1.1 模块依赖合理性

检查项：
- [ ] api 层是否只定义接口和 DTO？接口是否放在独立模块中？
- [ ] service 层是否只依赖 api + core + dao？是否引入了不应有的上层依赖？
- [ ] web 层是否只依赖 service？是否包含业务逻辑？
- [ ] 框架特定依赖（Quarkus/Spring）是否隔离到 app 模块？
- [ ] 各模块之间是否存在循环依赖？
- [ ] 其他模块需要引用 nop-code 接口时，是否必须依赖整个 service 模块？
- 注：`dao/biz/` 下的 `IXxxBiz extends ICrudBiz<Entity>` 接口是代码生成的 DAO 层 CRUD 契约，绑定 DAO 实体，放在 dao 模块是正确的。不应作为"业务接口放错位置"的审计发现。

#### 1.2 层级职责划分

检查项：
- [ ] core 层是否保持纯计算，无框架特定依赖（Quarkus/Spring/MyBatis 等）？注：`jakarta.inject`/`jakarta.annotation` 是 Java 标准规范（JSR-330/JSR-250），Nop IoC 基于此标准，不算框架依赖。
- [ ] dao 层的 ORM 实体是否被实际使用？是否存在定义了但从未读写的实体？
- [ ] service 层是否通过 dao 层访问数据？是否存在绕过 DAO 直接使用内存存储的情况？
- [ ] web 层页面是否功能完整？是否存在空壳页面？

### 维度二：数据架构与持久化

#### 2.1 存储架构一致性

检查项：
- [ ] 是否存在多套独立的数据存储（内存 vs 数据库）互不连通的情况？
- [ ] 设计文档描述的数据流与实际实现是否一致？
- [ ] 应用重启后数据是否可恢复？
- [ ] 是否存在定义了完整表结构但从未被写入或读取的 ORM 实体？

#### 2.2 ORM 实体设计

检查项：
- [ ] 主键设计是否规范？
- [ ] 审计字段（createTime/updateTime 等）是否与配置一致？
- [ ] 关系定义是否完整（to-one/to-many）？
- [ ] xmeta 是否覆盖所有实体？displayName 和 dict 定义是否完整？
- [ ] 实体设计与设计文档是否对齐？如不对齐，是否已更新文档？

### 维度三：API 设计与 BizModel 规范

#### 3.1 API 层模型隔离

检查项：
- [ ] BizModel 的公开方法（`@BizQuery`/`@BizMutation`/`@BizLoader`）返回值是否合理？
- [ ] 是否存在返回值中包含敏感字段但 xmeta 未限制其可见性的情况？
- 注（BizModel 返回实体是标准做法）：**BizModel 方法返回实体对象（包括 ORM 实体和 core 层模型）是 Nop 平台的标准模式，不需要返回 DTO。** 原因：
  - `CrudBizModel.get()` 直接返回 `T extends IOrmEntity`（平台基类设计）
  - `GraphQLExecutor.fetchSelections()` 根据客户端 GraphQL selection 逐字段获取值（`BeanPropertyFetcher` → `BeanTool.getProperty()`）
  - 客户端只能看到 xmeta 中定义的、且在 selection 中请求的字段，不会暴露整个实体
  - API 契约由 xmeta（字段可见性/类型）和 GraphQL selection（客户端实际获取的字段）共同决定
  - **不应将"BizModel 返回实体"作为审计发现**，这不是架构问题
- 注（内部服务接口）：`ICodeIndexService` 等内部服务接口不对外暴露，返回 core 层模型（如 `CodeSymbol`、`CodeFileAnalysisResult`）是合理的——内部以高性能、高便捷性为目标。

#### 3.2 接口抽象完整性

检查项：
- [ ] BizModel 是否通过接口调用，还是存在 `instanceof` + 强转到实现类的情况？
- [ ] 接口是否覆盖了所有被外部使用的方法？
- [ ] 是否可以替换实现类而不影响调用方？

#### 3.3 错误处理规范

检查项：
- [ ] 是否统一使用 `NopException` + `ErrorCode`？
- [ ] 是否存在 `throw new RuntimeException(...)` 的违规？注：底层工具类（如 `DigestHelper`）使用 `IllegalStateException` 表示前置条件违反是可接受的，不属于违规。
- [ ] 异常是否包含足够的上下文参数（`.param(...)`）？

#### 3.4 API 参数一致性

检查项：
- [ ] 必选/可选参数标注是否一致（`@Optional` 使用）？
- [ ] 分页参数命名是否统一？
- [ ] 相同语义的参数是否在不同方法间命名一致？

### 维度四：核心分析引擎

#### 4.1 语言解析覆盖

检查项：
- [ ] 已支持的语言特性是否完整（如 Java 的 Record、sealed class、pattern matching）？
- [ ] 不支持的特性是否被静默忽略？是否有日志或标记？

#### 4.2 类型系统

检查项：
- [ ] 类型信息是字符串还是有结构化表示？
- [ ] 是否可以程序化解析泛型参数？
- [ ] 是否支持类型间的继承/实现关系追踪？

#### 4.3 算法质量

检查项：
- [ ] 图算法是否有正确的复杂度分析？
- [ ] 是否有处理大图的优化策略（降级、分片）？
- [ ] 是否存在潜在的数值溢出或效率问题？

#### 4.4 内存管理

检查项：
- [ ] 是否存在数据冗余（同一数据的多份存储）？
- [ ] 是否有驱逐机制（LRU/TTL/大小限制）？
- [ ] 分析完成后是否释放不再需要的大对象（如源码内容）？
- [ ] 是否有 OOM 风险？

### 维度五：前端页面与用户体验

#### 5.1 页面功能完整度

检查项：
- [ ] 每个页面的 CRUD 操作是否完整？
- [ ] 查询表单（queryForm）配置是否正确？是否有结果展示区域？
- [ ] 是否存在空壳页面（仅继承生成页面，无自定义）？

#### 5.2 关键功能覆盖

检查项：
- [ ] 后端已有 API 是否都有对应的前端入口？
- [ ] 可视化需求（树形、图表）是否有对应的前端组件？

#### 5.3 E2E 测试覆盖

检查项：
- [ ] E2E 测试是否覆盖了所有关键页面？
- [ ] 是否只覆盖了正向流程，缺少错误场景？

### 维度六：行业对标与能力差距

检查项：
- [ ] 与 LSP 标准对比，缺少哪些能力？这些能力是否在模块定位范围内？
- [ ] 与同类工具（SonarQube/Sourcegraph/ArchUnit）对比，核心差距在哪里？
- [ ] 模块定位是否清晰（代码索引 vs 静态分析 vs 架构分析）？
- [ ] 增量分析、持久化存储、查询优化的工程化成熟度如何？

### 维度七：可扩展性与多语言支持

检查项：
- [ ] 新增语言是否只需实现接口并注册？
- [ ] 是否支持跨语言分析？
- [ ] 增量分析是否高效（避免全量重建）？
- [ ] 外部符号引用（JDK、第三方库）是否可索引？

### 维度八：测试质量

检查项：
- [ ] 单元测试覆盖率如何？是否有明显的测试缺口？
- [ ] 是否有集成测试覆盖关键流程？
- [ ] 是否有并发/线程安全测试？
- [ ] 是否有性能基准测试？
- 注（不需要测试的模块）：
  - `nop-code-api`：空壳模块，无代码可测
  - `nop-code-app`：打包工程（仅含 Application 启动类），不需要测试
  - `nop-code-codegen`：代码生成辅助功能，`NopCodeCodeGen` 是调试用的 main 方法，不需要测试
  - `nop-code-meta`：无 Java 源码，仅含 xmeta 资源文件和代码生成模板，不需要测试
- 注（内存管理——当前存在全量累积问题）：`ProjectAnalyzer` 虽然分批提交任务，但所有批次结果通过 `fileResults.addAll()` 累积在内存中，分析完成后才一次性持久化。**不符合"内存中最多保留一个批次数据"的要求**。应检查是否需要改为流式持久化（每批次即时写入 DB 并释放）。

## 审计输出格式

每次审计应产出以下结构：

### 发现分级

| 级别 | 定义 | 示例 |
|------|------|------|
| P0 架构断裂 | 影响正确性和可用性的严重问题 | 存储层断裂、核心功能不可用 |
| P1 代码质量 | 影响可维护性和规范合规的问题 | 抽象泄漏、错误处理不规范 |
| P2 能力缺口 | 影响功能完整性的缺失 | 语言特性不支持、页面缺失 |
| P3 架构演进 | 影响长期可扩展性的改进 | 类型系统重构、外部引用 |

### 每个发现应包含

1. **问题描述**：简洁说明发现了什么
2. **具体证据**：引用相关代码位置（文件路径、类名、方法名）
3. **影响分析**：说明该问题的后果
4. **改进建议**：给出具体的修复方向（而非模糊建议）
5. **优先级**：P0/P1/P2/P3 分级

### 审计报告存放

审计报告存放于 `ai-dev/audits/nop-code-audit-YYYY-MM-DD.md`。

## 与通用审计技能的关系

本文件是针对 nop-code 模块的特定审计提示词。除了本文件，以下通用审计技能也适用于 nop-code 模块：

- **`ai-dev/skills/deep-audit-prompts.md`**：20 维度系统性深度审核手册，支持多轮迭代深挖 + 独立复核。本文件的维度与 deep-audit 的维度存在交叉（如维度一 ≈ deep-audit 维度 01+02，维度三 ≈ deep-audit 维度 07+09），但 deep-audit 提供了更严格的子 agent 调度、深挖追加和独立复核机制。
- **`ai-dev/skills/open-ended-adversarial-review-prompt.md`**：开放式对抗性审查，不绑定固定维度，以代码中的异常信号为切入点发现高价值问题。

推荐组合使用方式：
1. **快速审计**：直接使用本文件，按维度逐项检查。
2. **系统性深度审核**：使用 `deep-audit-prompts.md` 的多轮迭代 + 独立复核机制，目标模块设为 nop-code。
3. **发现意外问题**：使用 `open-ended-adversarial-review-prompt.md`，目标模块设为 nop-code，找出维度化审计容易遗漏的跨领域问题。
4. **互补验证**：先用 deep-audit 系统扫描，再用 adversarial review 补充发现。

## 使用方式

1. **每次重大迭代前**，使用此提示词对相关维度重新审计
2. **PR 审查时**，重点检查维度三（API 规范）和维度四（core 层质量）
3. **季度回顾时**，全面覆盖所有维度，评估改进路线图进展
4. 审计时使用 explore agent 并行探索相关代码，而非凭记忆填写
5. **需要更严格审计时**，使用 `deep-audit-prompts.md` 的多轮迭代机制
