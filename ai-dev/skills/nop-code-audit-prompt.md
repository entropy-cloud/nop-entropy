## 模块概述

nop-code 是 Nop 平台的代码分析与索引模块，提供符号搜索、类型层级、调用链、图分析等能力。

模块组成：
- `nop-code-core`：核心分析层（符号模型、类型系统、图算法、多语言解析器接口）
- `nop-code-dao`：ORM 实体与数据访问
- `nop-code-meta`：xmeta 元数据定义
- `nop-code-service`：业务逻辑层（索引管理、分析调度）
- `nop-code-web`：前端页面
- `nop-code-app`：应用入口
- `nop-code-api`：API 接口定义
- `nop-code-codegen`：代码生成
- `nop-code-lang-*`：多语言解析器（Java、TypeScript 等）

## 专属维度 A：核心分析引擎质量

### A.1 语言解析覆盖

检查项：
- [ ] 已支持的语言特性是否完整（如 Java 的 Record、sealed class、pattern matching）？
- [ ] 不支持的特性是否被静默忽略？是否有日志或标记？

### A.2 类型系统

检查项：
- [ ] 类型信息是字符串还是有结构化表示？
- [ ] 是否可以程序化解析泛型参数？
- [ ] 是否支持类型间的继承/实现关系追踪？

### A.3 算法质量

检查项：
- [ ] 图算法是否有正确的复杂度分析？
- [ ] 是否有处理大图的优化策略（降级、分片）？
- [ ] 是否存在潜在的数值溢出或效率问题？

### A.4 内存管理

检查项：
- [ ] 是否存在数据冗余（同一数据的多份存储）？
- [ ] 是否有驱逐机制（LRU/TTL/大小限制）？
- [ ] 分析完成后是否释放不再需要的大对象（如源码内容）？
- [ ] 是否有 OOM 风险？
- 注（当前已知问题）：`ProjectAnalyzer` 虽然分批提交任务，但所有批次结果通过 `fileResults.addAll()` 累积在内存中，分析完成后才一次性持久化。**不符合"内存中最多保留一个批次数据"的要求**。应检查是否需要改为流式持久化（每批次即时写入 DB 并释放）。

## nop-code 特有的误报校准

以下不是问题，不应作为审计发现：
- `nop-code-api` 没有测试（空壳模块，无代码可测）
- `nop-code-app` 没有测试（打包工程，仅含启动类）
- `nop-code-codegen` 没有测试（调试用 main 方法）
- `nop-code-meta` 没有测试（无 Java 源码，仅含 xmeta 和生成模板）
- `dao/biz/` 下的 `IXxxBiz extends ICrudBiz<Entity>` 接口放在 dao 模块（代码生成的 CRUD 契约）
- BizModel 方法返回实体对象（平台标准模式，xmeta + GraphQL selection 控制字段可见性）
- `ICodeIndexService` 等内部服务接口返回 core 层模型（内部以高性能为目标）
- 底层工具类使用 `IllegalStateException` 表示前置条件违反（可接受，不属于 NopException 违规）
