# AGE 文档审计提示词手册

> **定位**: 基于吸引子引导工程（AGE）方法论的文档审计提示词，用于独立审计项目文档体系是否有效承载吸引子、维护轨迹收敛能力。采用多轮迭代深挖 + 独立复核的调度架构。
> **适用场景**: 新项目初始化后首次审计、阶段性文档健康检查、owner-doc 更新后的一致性验证。
> **执行前提**: 审计者必须先阅读 AGENTS.md 和 docs-for-ai/INDEX.md（设计阶段模式除外），理解当前项目的路由基线和文档职责划分。
> **输出格式**: 每个发现按统一格式输出（见附录 A）。

---

## 审计模式

本手册支持两种审计模式，主 agent 在启动时必须先选定模式：

### 模式一：全面审计（full）

适用于已有代码实现的项目。审计完整文档体系，包含外部路由可达性、文档-代码对齐、轨迹记录、控制机制等全部维度。

**适用时机**：项目初始化后首次审计、阶段性健康检查、owner-doc 大幅更新后。

### 模式二：设计阶段审计（design-only）

**仅审计设计文档自身**，不涉及：
- 外部路由可达性（不检查 AGENTS.md 路由、docs-for-ai/INDEX.md 导航）
- 代码与实现的一致性（不验证文档描述是否与代码匹配）
- 轨迹记录完整性（设计阶段尚未进入实现，无轨迹可查）
- 平台特定的代码相关检查（生成链路、Delta 定制等需要代码的检查项）

**只聚焦两件事**：
1. **设计文档自身的完整性** — 设计是否自洽、覆盖了声明范围内的关键方面、无遗漏
2. **设计概念的可实现性** — 设计描述的概念和技术方案在原理上是否可实现、逻辑链条是否闭合

**适用时机**：新子系统设计评审、设计文档定稿前的质量检查、设计阶段尚无代码可验证时。

**设计阶段审计的核心问题**：

> 这个设计文档能不能作为实现的可靠起点？读完它之后，实现者是否会因为设计自身的模糊、遗漏或逻辑断裂而被迫回头做大量"填补空白"的工作？

### 模式与维度映射

| 维度 | 全面审计 | 设计阶段审计 | 说明 |
|------|---------|-------------|------|
| A 吸引子可发现性 | ✓ | ✗ | 设计阶段不涉及外部路由 |
| B Owner-Doc 内容质量 | ✓（全部 5 子组） | ✓（仅 B.1 + B.2 + B.3 + B.4） | 设计阶段跳过 B.5 文档-代码对齐 |
| B'.设计文档自检 | ✗ | ✓ | 设计阶段专用维度，深入检查设计完整性 |
| C 轨迹记录完整性 | ✓ | ✗ | 设计阶段无轨迹记录 |
| D 控制机制有效性 | ✓ | ✗ | 依赖外部路由 |
| E 抗漂移能力 | ✓ | ✗ | 依赖实现阶段机制 |
| F Nop 平台特定 | ✓ | ✗ | 依赖代码实现 |

## 审计总览

### 全面审计维度（6 个）

按 AGE 三层模型组织：

| 层次 | 维度编号 | 维度名称 | 审核目标 |
|------|---------|---------|---------|
| **吸引子层** | A | 吸引子可发现性 | 新 session 能否在 3 步以内定位当前吸引子 |
| | B | Owner-Doc 内容质量 | owner docs 是否准确、完整、精确、可实现（含 5 子组） |
| **轨迹层** | C | 轨迹记录完整性 | plans/logs/bugs 是否足以恢复历史上下文 |
| **控制层** | D | 控制机制有效性 | 路由/验证/审计等机制是否引导轨迹收敛 |
| | E | 抗漂移能力 | 文档体系能否抵抗结构性漂移 |
| **平台特定** | F | Nop 平台文档特定检查 | Nop 平台特有的文档健康检查 |

### 设计阶段审计维度（2 个）

聚焦设计文档本身：

| 维度编号 | 维度名称 | 审核目标 |
|---------|---------|---------|
| **B** | Owner-Doc 内容质量（B.1+B.2+B.3+B.4） | 设计文档内部一致性、完整性、精确性、概念可实现性 |
| **B'** | 设计文档自检 | 设计逻辑闭环、概念定义精确性、约束收敛能力、边界条件覆盖 |

## AGE 文档审计的核心问题

AGE 把工程文档分为三层：

- **吸引子层（Attractor）**: 系统应长期收敛向的稳定结构。承载者是一组 owner docs（design、architecture、context）。
- **轨迹层（Trajectory）**: 实际演化路径的记录。承载者是 plans、logs、bugs、audits。
- **控制层（Control）**: 持续校正轨迹的局部机制。承载者是 routing（路由）、verification（验证）、audit prompts（审计提示词）、skills。

**AGE 文档审计不是"检查文档写得漂不漂亮"。它回答一个前提问题：**

> 如果下一次 AI session 打开这个仓库，它能否仅凭文档结构就收敛到正确的方向，而不是逐步漂移？

## 调度架构

### 总体流程

```
主 agent（读取本文档，选择要执行的维度）
  │
  ├── 准备阶段：收集文档基线，生成命令基线
  │
  ├── 阶段一 — 迭代深挖（每个维度独立）
  │   ├── Task → 维度 A 第 1 轮（初审）
  │   ├── Task → 维度 A 第 2 轮（追加深挖）
  │   ├── ...（最多 10 轮，直到无新发现）
  │   │
  │   ├── Task → 维度 B 第 1 轮（初审，含 5 子组）
  │   ├── Task → 维度 B 第 2 轮（追加深挖）
  │   ├── ...
  │   │
  │   ├── [各维度独立执行]
  │   │
  │   └── 阶段一完成：每个维度均深挖至收敛
  │
  ├── 阶段二 — 独立复核（每个维度独立）
  │   ├── Task → 维度 A 维度复核（读取完整文件，逐条保留/降级/驳回）
  │   ├── Task → 维度 A 子项复核（高风险逐条）
  │   │
  │   ├── Task → 维度 B 维度复核
  │   ├── [各维度独立复核]
  │   │
  │   └── 汇总已复核通过的结果
  │
  └── 阶段三 — 跨维度汇总（所有维度完成后）
      └── 主 agent 生成 AGE 文档审计汇总报告
```

### 调度方法

1. 主 agent 读取本文档，选择要执行的**维度**。
2. **准备阶段**：
   a. 主 agent 收集文档基线（见"准备阶段基线收集"节）。
   b. **禁止阅读 `ai-dev/audits/`、`ai-dev/plans/`、`ai-dev/bugs/`、`ai-dev/lessons/` 等历史记录**。只审计当前 live 文档，不要浪费时间回查历史。
3. **阶段一 — 迭代深挖**：
   a. 主 agent 用"共享提示词前缀 + 该维度正文"拼接出完整 prompt，派发**第 1 轮（初审）子 agent**。
   b. 初审完成后，将发现保存到 `ai-dev/analysis/{日期}-doc-age-audit-{标识}/{维度编号}-{名}.md`。
   c. 如果第 1 轮有发现，主 agent 派发**第 2 轮（追加深挖）子 agent**，prompt 中包含：
      - 共享提示词前缀
      - 该维度正文（相同）
      - 已保存发现全文
      - 深挖追加指令（见"深挖追加提示词模板"）
   d. 第 2 轮完成后，新发现追加到同一个维度文件（标注 `## 深挖第 2 轮追加`）。
   e. 重复 c-d，直到某轮输出无新发现或达到 10 轮上限。
   f. 如果第 1 轮就无发现，直接进入阶段二维度复核（复核 agent 确认零发现）。
4. **阶段二 — 复核**：
   a. 深挖结束后，主 agent 派发一个**独立的维度复核子 agent**，输入为该维度完整文件（含所有深挖追加），要求它重新读文档与代码，输出"保留 / 降级 / 驳回"的逐条判断。
   b. 维度复核完成后，主 agent 必须对高风险或不确定发现项，再派发**独立的子项复核子 agent**逐项验证；低风险项可按文件或按模式批量复核。
5. 只有在"深挖完成 + 维度复核 + 必要的子项复核"都完成后，该维度结果才允许进入最终汇总。
6. 所有维度完成后，主 agent 汇总**已复核通过**的结果，生成「AGE 文档审计汇总报告」（格式见附录 B）。

### 深挖追加提示词模板

从第 2 轮起，主 agent 在维度正文后追加以下内容：

```text
---

## 深挖追加指令（第 N 轮）

以下是本维度前 N-1 轮已保存的全部发现：

[粘贴前 N-1 轮的完整发现文本]

你的任务：

1. 读取上述已有发现，理解本维度已经覆盖了哪些文档、模式和路径。
2. 基于已有发现暴露的文档、模式和关联路径，**继续深挖**本维度中尚未覆盖的盲区：
   - 已有发现涉及的文档是否有同类型问题未被检出？
   - 已有发现的模式是否在其他文档中也存在？
   - 是否有与已有发现相关的、但尚未检查的文档路径？
   - 该维度的执行步骤中是否有尚未充分覆盖的步骤？
3. **只输出新发现的条目**。不要重复已有发现中已经报告的内容。
4. 如果经过仔细检查后确实没有发现新的问题，输出："未发现新的问题。深挖结束。"
5. 如果仍能找到零散细节，但这些细节只是低价值、机械重复、缺少明确风险或缺少可行动建议，不要为了凑数量继续报告；输出："未发现新的高价值问题。深挖结束。"
6. 每个新发现仍需遵守统一的发现格式（文件路径 + 行号 + 证据片段 + 严重程度 + 现状 + 风险 + 建议）。

【重要】发现完整性要求：
- 每个新发现必须按完整格式输出（### 标题 + 文件 + 行号 + 3-10行证据片段 + 严重程度 + 现状 + 风险 + 建议 + 误报排除）。
- 禁止将发现压缩为关键词、一句话摘要或表格行。
- 禁止只输出标题列表然后说"详情见上方"。
- 如果发现数量较多导致输出较长，宁可分多个子 agent 输出，也不要压缩每个发现的内容。
```

### 保存与追加规则

主 agent 在保存深挖结果时必须遵守以下规则：

1. **第 1 轮结果**：将子 agent 输出的完整发现正文写入文件。如果子 agent 输出被压缩（发现条目少于 8 行），主 agent 必须要求重新输出。
2. **追加深挖结果**：在文件末尾追加 `## 深挖第 N 轮追加` 标题，然后写入新发现的完整正文。**不修改、不压缩、不覆盖**前 N-1 轮已保存的内容。
3. **复核结论追加**：在深挖全部结束后，追加 `## 维度复核结论` 和 `## 子项复核结论`。复核结论引用已有发现编号（如 `[A-01]`），不重写发现内容。

### 并行策略

- 同批次的**第 1 轮（初审）**可以并行派发。
- 同一维度的**深挖追加轮次**必须串行（每轮依赖前一轮结果）。
- 同一维度的**维度复核**必须在该维度深挖阶段全部结束后进行。
- 同一维度的**子项复核**必须在该维度复核完成后进行。
- **不同维度之间**的深挖、复核可以并行（互不依赖）。

批次策略：

**全面审计**：

| 批次 | 维度 | 可否并行 |
|------|------|---------|
| 第一批 | A, F | 互相独立，可并行 |
| 第二批 | B | 维度 B 较重，建议独立批次 |
| 第三批 | C, D, E | 互相独立，可并行 |

**设计阶段审计**：

| 批次 | 维度 | 可否并行 |
|------|------|---------|
| 第一批 | B（B.1+B.2+B.3+B.4）, B' | 互相独立，可并行 |

> **注意**：维度 B（Owner-Doc 内容质量）包含 5 个子检查组，工作量最大。建议在初审时由单个子 agent 完成全部 5 个子检查组的评估，然后在深挖追加轮次中按子检查组分别深入。
> **设计阶段注意**：维度 B 和 B' 可以并行初审，但 B' 的深挖可能产生与 B 交叉的发现。主 agent 在汇总时需要去重。

### 准备阶段基线收集

主 agent 在派发维度前，应收集以下文档基线并作为 prompt 的一部分提供给子 agent：

1. `AGENTS.md` — 路由表、保护区域、验证命令
2. `docs-for-ai/INDEX.md` — 文档路由和目录职责
3. 项目上下文文件 — `docs-for-ai/00-start-here/project-context`
4. 所有 owner doc 的目录列表
5. 最近的 dev log 目录列表
6. `ai-dev/` 下各子目录的文件列表（不读内容，只看结构）
7. `docs-for-ai/` 下的完整文件树（不读内容，只看结构）

子 agent 需要时可直接读取具体文件内容。如果主 agent 未提供基线，子 agent 应明确标注"缺少基线"，不能把手工估算当确定事实。

### 子 Agent 提示词装配结构

下文每个维度给出的内容是**维度正文**，不是可单独复制执行的完整 prompt。主 agent 派发前，必须先拼接共享前缀。

完整 prompt 由两部分组成：

```
1. 共享提示词前缀（固定，所有维度相同）
2. 维度正文（本维度目标 + 必读文档 + 执行步骤 + 额外输出要求）
```

### 共享提示词前缀

#### 全面审计模式

主 agent 在派发任一维度前，应先附加以下共享前缀：

```text
你正在执行 nop-entropy 项目的 AGE 文档体系审计。这是一个基于 Nop 平台的 Java 多模块 Maven 工程。

AGE（吸引子引导工程）核心概念：
- 吸引子层（Attractor）：系统应长期收敛向的稳定结构，承载者是 owner docs
- 轨迹层（Trajectory）：实际演化路径的记录，承载者是 plans/logs/bugs
- 控制层（Control）：持续校正轨迹的局部机制，承载者是 routing/verification/audit prompts

执行前先阅读：
1. docs-for-ai/INDEX.md（文档导航基线）
2. AGENTS.md（代码规范、验证命令、agent 工作流）
3. docs-for-ai/00-start-here/project-context（项目上下文）
4. docs-for-ai/01-repo-map/module-groups.md（模块分组）
5. ai-dev/analysis/00-analysis-writing-guide.md（分析报告写作规范）

注意：当前审计基线按 **live 文档和代码为准** 执行。不以历史日志、已关闭计划或口头约定为准。

通用审计口径：
1. 以当前文档和代码为准，不以历史日志、已关闭计划或口头约定为准。
2. 不重复报告已收敛的问题。
3. 不把"看起来不优雅"当问题，必须有结构性原因或可量化风险。
4. 每个发现必须可定位：文件路径 + 行号范围 + 证据片段（引用文档原文或代码片段）。
5. 初审输出只是线索，不是最终事实；最终结论必须经过独立复核。
6. 如果某条发现看起来像历史上的高频误报模式，必须明确写出"为什么这次不是同类误报"。

严重程度判级：
- P0: 吸引子断裂 — 新 session 会被误导到错误方向
- P1: 收敛能力退化 — 轨迹会逐步漂移但不会立即出错
- P2: 维护成本增加 — 不影响正确性但增加理解成本
- P3: 理想化改进 — 当前可工作但有优化空间

AGE 文档审计常见误报校准（以下**不是**问题，不应作为审计发现）：
- owner doc 中使用"建议"而非"必须"表述推荐做法（这是合理的弹性空间）
- 轨迹记录（logs/plans）的格式细节不一致（只要内容完整即可）
- 非核心文档缺少完美的结构（聚焦 owner doc 质量）
- docs/ 目录下的旧文档与 docs-for-ai/ 不完全一致（docs/ 不是审计对象）
- AGENTS.md 中引用的命令行参数在不同 OS 下略有差异（只要核心语义正确）

【发现条目完整性 — 强制规则，不可违反】：
1. 每个发现必须按统一格式（附录 A）完整输出，包含文件路径、行号范围、3-10 行证据片段、严重程度、现状、风险、建议、信心水平。
2. **禁止将发现压缩为一行标题或表格单元格。**
3. **禁止用摘要替代完整条目。**
4. **证据片段不可省略。** 如果因上下文限制无法贴出，必须明确标注"[证据片段待补充]"并列出需要查看的具体行号范围。
5. 每个发现条目最少 9 行（文件、行号、证据、严重程度、现状、风险、建议、信心水平、误报排除），少于 9 行的条目视为不合格。
6. 复核结论表是补充索引，不能替代发现正文。
```

维护规则：

- 共享方法论、严重程度判级只维护在"共享提示词前缀"中。
- 各维度正文只保留该维度特有的目标、检查项、执行步骤和额外输出要求。
- 如果某个维度需要额外约束，只写该维度新增部分，不要重复抄写共享前缀内容。

#### 设计阶段审计模式

主 agent 在设计阶段模式下派发维度前，使用以下前缀（替代全面审计前缀）：

```text
你正在执行 AGE 文档体系审计 — 设计阶段模式。

本次审计的范围仅限于设计文档本身，不涉及代码实现、外部路由或实现历史。

AGE（吸引子引导工程）核心概念：
- 吸引子层（Attractor）：系统应长期收敛向的稳定结构，承载者是设计文档
- 轨迹层（Trajectory）：实际演化路径的记录（实现阶段才产生）
- 控制层（Control）：持续校正轨迹的局部机制（实现阶段才需要）
- 本审计只评估吸引子载体层的设计质量

【AGE 层次校准 — 设计阶段特有的抽象层次要求】：

设计文档是吸引子，不是实现蓝图。它应该精确地定义：
- 概念及其边界（"这个系统里有什么、它们之间是什么关系"）
- 约束和不变量（"这个系统必须满足什么性质"）
- 数据流和控制流的形状（"东西怎么流动、经过哪些节点"）
- 关键决策的理由（"为什么是这样而不是那样"）

设计文档**不需要**定义（这些属于实现层）：
- 方法名、参数名、返回值类型
- 精确的 API 签名或接口方法列表
- 具体的数据结构字段定义
- 实现级别的类图、序列图细节
- 可转化为 lint 规则的代码级约束

审计时的正确判级思路：
- 设计说"组件 A 负责数据校验"——够了，不需要列出校验方法签名
- 设计说"数据从源端经过清洗进入存储层"——够了，不需要描述方法调用链
- 设计说"此约束保证数据一致性"——够了，不需要转化为代码断言
- 设计说"使用事件驱动架构"——够了，不需要列出每个事件的 payload 结构

审计范围：
- 只读目标设计文档及其直接引用的参考文档
- 不读代码实现（设计阶段不存在）
- 不读外部路由文件（AGENTS.md、INDEX.md）
- 不读轨迹记录（plans/logs/bugs）

通用审计口径：
1. 以设计文档原文为准，评估设计概念和逻辑的质量。
2. 不重复报告已收敛的问题。
3. 不把"看起来不优雅"当问题，必须有结构性原因或可量化风险。
4. 每个发现必须可定位：文件路径 + 行号范围 + 证据片段（引用设计文档原文）。
5. 初审输出只是线索，不是最终事实；最终结论必须经过独立复核。
6. 如果某条发现看起来像历史上的高频误报模式，必须明确写出"为什么这次不是同类误报"。

严重程度判级（设计阶段）：
- P0: 设计概念断裂 — 核心概念缺失定义、关键关系未声明、数据流有不可弥合的缺口
- P1: 设计概念模糊 — 关键概念的边界不清或约束不够精确，会导致实现者做出不同的解读
- P2: 设计覆盖不足 — 声明范围内的明显遗漏（如只描述 happy path），但不影响核心概念
- P3: 设计可优化 — 概念和逻辑正确但可以更简洁或更精确

设计阶段审计常见误报校准（以下**不是**问题，不应作为审计发现）：
- 设计文档中未列出方法签名、参数名、返回值类型（设计层不需要）
- 设计文档中未定义精确的数据结构字段（设计层定义概念边界即可）
- 设计文档中使用"建议"而非"必须"表述推荐做法（这是合理的弹性空间）
- 设计中未指定具体的第三方库版本号（设计阶段选择技术方向即可）
- 设计文档缺少实现代码示例或伪代码（设计阶段不要求）
- 设计中声明了"后续优化"或"第二阶段"的延后项（只要当前范围明确即可）
- 设计未覆盖与当前需求无直接关系的极端边界条件（只要声明了范围边界即可）
- 设计中未指定精确的错误码或异常类型（这些是实现层细节）
- 设计中用自然语言描述交互契约而未给出精确的接口定义（设计层可以这样做）
- 设计中定义了概念但未指定其在代码中的映射形式（包/类/方法）（这是实现层的事）

【发现条目完整性 — 强制规则，不可违反】：
1. 每个发现必须按统一格式（附录 A）完整输出，包含文件路径、行号范围、3-10 行证据片段、严重程度、现状、风险、建议、信心水平。
2. **禁止将发现压缩为一行标题或表格单元格。**
3. **禁止用摘要替代完整条目。**
4. **证据片段不可省略。** 必须引用设计文档原文。
5. 每个发现条目最少 9 行（文件、行号、证据、严重程度、现状、风险、建议、信心水平、误报排除），少于 9 行的条目视为不合格。
6. 复核结论表是补充索引，不能替代发现正文。

【设计阶段特有约束】：
1. 只审计设计文档内容，不以"与代码不一致"作为发现理由。
2. 只评估设计概念的可实现性，不要求设计与当前代码结构匹配。
3. 发现的建议应该是设计层面的改进，不是"去改代码"。
4. 不以"缺少方法签名/参数定义/精确类型"作为发现——这些是实现层的事。
5. 以"概念是否清晰、约束是否明确、逻辑是否闭环"为评判标准，不以"是否足够详细到能直接写代码"为标准。
```

## 发现条目完整性守则

> **本节是所有子 agent 的硬性约束，与通用审计口径同级。** 违反本节规则的维度文件视为不合格，主 agent 必须要求子 agent 重新输出。

### 问题：发现被过度压缩

历史审核中，维度文件的发现被层层压缩为关键词或表格行，导致读者无法理解问题是什么、影响是什么、该怎么做。

**反模式：深挖轮次只记关键词**

```markdown
## 深挖轮次

- 第 1 轮: 路由表指向不存在的文件、owner doc 缺少优先级声明。
- 第 2 轮: 文档-代码不一致、过时内容。
```

**正确做法**：每轮深挖的每个发现必须是完整条目：

````markdown
## 第 1 轮（初审）

### [A-01] INDEX.md 路由指向不存在的文件

- **文件**: `docs-for-ai/INDEX.md:42`
- **证据片段**:
  ```markdown
  | 服务层 | `docs-for-ai/02-core-guides/service-layer.md` |
  ```
- **严重程度**: P0
- **现状**: INDEX.md 第 42 行路由指向 `service-layer.md`，但该文件实际不存在于 `docs-for-ai/02-core-guides/` 目录中。
- **风险**: 新 session 按路由查找时会失败，无法定位服务层指导文档，吸引子可发现性断裂。
- **建议**: 确认文件是否已重命名或合并，更新 INDEX.md 中的路由路径。
- **信心水平**: 确定
- **误报排除**: 已验证目录下确实不存在该文件，不是路径别名问题。
- **复核状态**: 未复核
````

### 维度文件质量检查清单

主 agent 在接收子 agent 输出后、保存文件前，必须检查以下各项。任何一项不通过，都必须要求子 agent 补充完整后再保存：

1. **每个发现条目 ≥ 9 行**：文件路径、行号范围、证据片段、严重程度、现状、风险、建议、信心水平、误报排除缺一不可。
2. **证据片段存在**：每个发现必须有 3-10 行文档原文或代码片段，或明确标注 `[证据片段待补充]`。
3. **深挖轮次不是关键词列表**：每一轮的每个发现都是完整的 `###` 级条目，不是一行关键词。
4. **复核结论引用了发现编号**：不是独立重写，而是引用上方已有的发现条目编号。
5. **最终保留项有文件路径和严重程度**：不是纯文字列表。
6. **零发现维度也需说明检查范围**：列出读过的关键文件和检查过什么，不能只写"零发现"。

### 维度 A：吸引子可发现性（Attractor Discoverability）

**子 Agent 提示词**：

```text
以下为"维度 A"的维度正文。派发时必须与上文"共享提示词前缀"拼接。

审计维度 A：吸引子可发现性

目标：检查新 AI session 能否在 3 步以内定位到当前吸引子。这是文档审计的前置维度——如果吸引子不可发现，其他维度的发现将无法被有效消费。

必读文档：
- AGENTS.md
- docs-for-ai/INDEX.md
- docs-for-ai/00-start-here/project-context

执行步骤：

1. 检查入口文件（AGENTS.md）是否存在且指向当前活跃的 owner docs：
   - 路由表是否覆盖主要任务类型
   - 路由目标是否是真实存在的文件
   - 从入口文件出发，能否通过不超过 2 层链接找到每个关键 owner doc
2. 检查吸引子载体可识别性：
   - 读者能否区分 stable owner docs vs dated records vs control tools
   - 文件命名是否帮助区分（stable 文件名 vs 日期前缀文件名）
3. 检查优先级声明：
   - owner docs 之间是否有明确的优先级/先验声明
   - 文档间冲突时的裁决规则是否明确
4. 检查活跃状态标注：
   - project-context 是否标注了当前活跃需求、活跃 owner doc、文档新鲜度

   - 设计 2-3 个典型场景（如"修改服务层业务逻辑"、"新增 ORM 实体"），仅凭文档路由决定该读哪些文件
   - 记录路由是否无歧义、每步链接是否可达

判定等级：
- 健康：入口 → 路由 → owner doc 三步闭环，优先级和活跃状态明确

输出格式：同标准发现格式。最后输出吸引子可发现性总评（健康/退化/断裂）和路由测试结果。
```

### 维度 B：Owner-Doc 内容质量（Owner-Doc Content Quality）

**子 Agent 提示词**：

```text
以下为"维度 B"的维度正文。派发时必须与上文"共享提示词前缀"拼接。

审计维度 B：Owner-Doc 内容质量

目标：检查 owner docs 的内容是否准确、完整、精确、可实现。这是文档审计最重的维度。

AGE 视角：owner doc 是吸引子的载体层。载体层的质量直接影响吸引子的精确性——如果载体层模糊、矛盾、过时或不可实现，吸引子就退化为口号，系统无法向其收敛。

本维度分为五个子检查组。全面审计模式执行全部五个；设计阶段模式跳过 B.5（文档-代码对齐）。

必读文档：
- docs-for-ai/ 下的所有核心指南（service-layer.md, model-first-development.md, delta-customization.md, ioc-and-config.md, api-and-graphql.md, error-handling.md, testing.md 等）
- ai-dev/design/ 下的设计文档
- 对应 owner doc 中引用的代码文件（全面审计模式，用于验证文档-代码对齐）

执行步骤：

**B.1 内部一致性检查**：
1. 检查同一 owner doc 内的术语一致性（同一概念是否始终使用同一术语）。
2. 检查是否存在自相矛盾（前半部分断言被后半部分推翻）。
3. 检查层级一致性（声明的分层架构是否在接口定义中遵守）。
4. 检查交叉引用一致性（文档 A 引用文档 B 时描述的内容是否与 B 实际一致）。
5. 检查数据流一致性（描述的数据流是否可追踪、无断裂）。

**B.2 设计完整性检查**：
1. 检查接口定义是否包含所有必要的输入、输出、前置条件、后置条件、错误情况。
2. 检查生命周期覆盖（有状态对象/流程是否覆盖了创建、使用、销毁全生命周期）。
3. 检查配置维度完整性（可配置系统是否列出了所有维度、默认值、合法范围）。
4. 检查交互契约完整性（模块间交互是否明确了同步/异步、超时、重试、幂等性）。
5. 检查非功能需求覆盖（性能、安全、并发、可观测性）。
6. 检查已知限制声明（是否明确声明了当前设计的限制和非目标）。

【设计阶段适配】：设计阶段 B.2 的"完整性"指概念层面的完整性，不是实现细节的完整性。调整如下：
- 第 1 条改为：检查概念间的交互契约是否声明了必要的语义（如"谁负责什么、数据如何流动"），不要求精确的方法签名或参数定义。
- 第 3 条改为：检查可配置的设计维度是否被识别（如"哪些方面允许变化"），不要求列出精确的配置项名称和默认值。
- 第 4 条改为：检查模块间交互的语义是否明确（如"这个交互是同步还是异步"），不要求精确的超时数值。

**B.3 精确性与先进性检查**：
1. 检查关键设计断言是否可证伪（能否明确判定违反）。
2. 检查设计规则是否可执行（是否具体到可转化为 lint 规则或审计检查项）。
3. 检查原创概念定义是否精确（如 Nop 的"可逆计算"、XDSL 的语义定义）。
4. 检查设计是否有针对性思考（非行业模板照搬）。
5. 检查抽象层次是否适当。
6. 检查重要设计决策是否有理由和否决方案记录。

【设计阶段适配】：设计阶段 B.3 的"精确性"指概念和约束的精确性，不是实现细节的精确性。调整如下：
- 第 1 条改为：检查关键概念和约束是否可证伪（能否给出一个明确违反该约束的例子）。这里的"证伪"是概念层面的："能不能想出一个场景，它看起来符合描述但实际违反了设计意图"。
- 第 2 条在设计阶段不适用（lint 规则是控制层机制），替换为：检查设计规则是否可以用自然语言无歧义地表达（不含"适当"、"合理"等模糊词）。
- 第 5 条在设计阶段强调：设计是否停留在正确的抽象层次——既不是空洞的口号（"保持良好的架构"），也不是过度的实现细节（方法签名级别的描述）。

**B.4 可实现性检查**：
1. 检查设计是否与项目声明的技术栈一致。
2. 检查从设计描述到代码的实现路径是否可推导。
3. 检查设计是否可增量实现。
4. 检查设计依赖的外部系统/库是否真实可用。
5. 检查关键行为是否可通过自动化测试验证。
6. 【设计阶段】检查设计概念的原理可行性——不要求当前技术栈已实现，但原理上必须成立。

**B.5 文档-代码对齐检查**（全面审计模式专用，设计阶段跳过）：
1. 抽取 owner doc 中 3-5 个关键断言，验证代码中是否真实成立。
2. 检查文档描述的模块、接口、配置文件是否在代码库中存在。
3. 检查文档中是否有描述已不存在的模块或已改变的 API。
4. 检查生成产物边界是否清楚标明。
5. 检查文档中的术语、模块名、接口名是否与代码一致。

输出格式：
对每个子检查组分别输出评级和发现：

### [B.{N}-{序号}] 简短标题
- **文件**: 文档路径 + 行号（以及对应的代码路径，如果是 B.5）
- **子检查组**: B.1 / B.2 / B.3 / B.4 / B.5
- **证据片段**: 引用文档原文或代码片段
- **严重程度**: P0/P1/P2/P3
- **现状**: ...
- **风险**: ...
- **建议**: ...

最后输出维度 B 综合评级：
- B.1 内部一致性：{自洽 / 局部矛盾 / 核心矛盾}
- B.2 设计完整性：{完整 / 范围内部分缺失 / 明显遗漏}
- B.3 精确性与先进性：{精确 / 模糊-可用 / 退化-口号}
- B.4 可实现性：{可实现 / 部分可实现 / 不可实现}
- B.5 文档-代码对齐：{全面审计填写：对齐 / 局部脱节 / 严重脱节；设计阶段填写：N/A}
- 综合评级：{健康 / 退化 / 断裂}
```

### 维度 B'：设计文档自检（Design-Only Audit Core）

> **本维度仅在设计阶段审计模式下使用。** 全面审计模式不执行本维度。

**子 Agent 提示词**：

```text
以下为"维度 B'"的维度正文。派发时必须与上文"设计阶段共享提示词前缀"拼接。

审计维度 B'：设计文档自检

目标：深入检查设计文档作为"吸引子"的质量。吸引子不规定实现细节，但它必须足够精确，使得实现者在面对实现层的选择时能够做出正确的局部决策——而不是各自朝不同方向发散。

核心问题：如果三个不同的实现者各自独立从这个设计出发，他们构建的系统会收敛到相同的形状吗？还是各自发散？

本维度与维度 B 配合使用。维度 B 做结构性的质量评估（一致性/完整性/精确性/可实现性），维度 B' 做深度的概念闭环验证。

AGE 层次校准（必须遵守）：
- 吸引子层关注"是什么"和"为什么"，不关注"怎么写"。
- 发现不能以"缺少方法签名/参数定义/精确类型"为理由。
- 发现必须以"概念不清/约束模糊/逻辑断裂/因果链缺失"为理由。

必读文档：
- 目标设计文档（通常在 ai-dev/design/ 下）
- 设计文档中引用的外部参考文档（如有）
- 与设计相关的需求描述（如有）

禁止阅读：
- 代码实现（设计阶段不存在）
- 外部路由文件（AGENTS.md 路由表等）
- 轨迹记录（plans/logs/bugs）

执行步骤：

1. **设计逻辑闭环验证**：
   - 设计中描述的数据流是否从头到尾可追踪、无断裂点？
   - 设计中声明的约束是否在每个涉及的场景中都被遵守？
   - 设计中的状态转换是否构成完整的状态机（有无遗漏的转换、悬挂状态）？
   - 设计中引用的外部依赖是否都有明确的语义定义（"这个外部系统提供什么保证"，而不是"某个外部系统会提供"）？

2. **概念定义精确性**：
   - 设计引入的每个核心概念是否有清晰、无歧义的定义？
   - 概念之间的关系是否明确（组合/聚合/依赖/继承）？
   - 是否存在"概念名相同但含义不同"或"不同概念名但实际是同一事物"的情况？
   - 概念的边界是否清晰（什么属于这个概念、什么不属于）？
   - 每个概念是否可以用一个具体的例子来说明"这属于 X"和"这不属于 X"？

3. **约束的收敛能力验证**（替代"实现路径可推导性"）：
   - 设计中声明的约束是否足以限制实现者不做错误的设计选择？
   - 是否存在多个合理的实现方案，而设计没有给出裁决标准？（如果存在，实现者会各自选择不同方案，导致系统发散）
   - 设计中是否有隐含的假设（如"数据是最终一致的"）没有显式声明，会导致实现者做出相反的假设？
   - 增量实现的优先级是否明确（先做什么后做什么，但不需要精确到每个实现步骤）？

4. **边界条件与异常路径覆盖**：
   - 设计是否只描述了 happy path？
   - 错误场景的语义是否被考虑（不需要精确的错误码，但需要说明"出了这种错系统应该怎么表现"）？
   - 设计中的降级策略是否有方向性描述（如"此组件不可用时系统降级为只读模式"）？
   - 设计声明的范围边界是否清晰（什么在范围内、什么不在范围内）？

5. **设计内部因果链验证**：
   - 设计中"因为 A 所以 B"的推理链是否成立？
   - 是否存在结论缺乏前提、前提推不出结论的情况？
   - 设计中的性能/可靠性/安全性断言是否有支撑依据？
   - 是否存在隐含假设未被显式声明？

6. **可实现性验证（概念层面）**：
   - 设计依赖的技术方案在原理上是否可行（不要求当前技术栈已具备，但原理上必须成立）？
   - 设计中的并发/分布式场景是否考虑了一致性问题（声明了什么一致性模型，而不是精确的实现机制）？
   - 设计是否引入了不必要的复杂性（是否可以用更简单的方案达到同样效果）？

严重程度判级（设计阶段专用）：
- P0: 设计概念断裂 — 核心概念缺失定义、关键关系未声明、数据流有不可弥合的缺口
- P1: 设计约束不足 — 关键约束模糊或缺失，多个实现者会做出不同的合理选择，导致系统发散
- P2: 设计覆盖不足 — 声明范围内的明显遗漏（如只描述 happy path），但不影响核心概念
- P3: 设计可优化 — 概念和逻辑正确但可以更简洁或更精确

输出格式：同标准发现格式。每个发现必须引用设计文档的具体位置和原文。
发现理由只能是以下类型：
- 概念定义缺失或模糊
- 约束不够精确或缺失
- 逻辑链断裂或因果不成立
- 隐含假设未显式声明
- 范围边界不清晰

发现理由**不能是**以下类型（属于实现层，不是设计层的问题）：
- 缺少方法签名或参数定义
- 缺少精确的数据结构字段
- 缺少错误码或异常类型定义
- 缺少具体的类/包/模块映射

最后输出设计文档自检总评：
- 设计逻辑闭环：{完整 / 局部断裂 / 严重断裂}
- 概念精确性：{精确 / 局部模糊 / 核心模糊}
- 约束收敛能力：{强（实现者会做出一致选择）/ 中（大部分场景收敛）/ 弱（实现者会发散）}
- 综合判定：{设计可作为实现起点 / 设计需补充后可实现 / 设计需重做}
```

## 严重程度标准

| 等级 | 含义 | 示例 |
|------|------|------|
| **P0** | 吸引子断裂 — 新 session 会被误导到错误方向 | owner doc 描述的模块边界与代码实际完全不同；路由表指向不存在的文件；核心设计断言自相矛盾 |
| **P1** | 收敛能力退化 — 轨迹会逐步漂移但不会立即出错 | 缺少优先级声明导致两个 owner doc 矛盾时无法裁决；验证命令是占位符；关键接口定义遗漏了错误路径；设计断言不可证伪 |
| **P2** | 维护成本增加 — 不影响正确性但增加理解成本 | 轨迹记录不完整；部分文档命名不规范；术语混用但核心断言链完整；审计 prompt 未定制 |
| **P3** | 理想化改进 — 当前可工作但有优化空间 | 日志格式可以更紧凑；目录结构可以更扁平；设计决策缺少否决理由记录 |

## 执行建议

### 全面审计

- **频次**: 项目初始化后必做一次；此后每 2-4 周或在 significant owner-doc 变更后做一次。
- **独立性**: 文档审计应由独立子 agent 执行，不由当前 session 的主 agent 自审。
- **修复优先级**: P0（吸引子断裂）应立即修复，P1 应在下一个 session 开始前修复。
- **与代码审计的关系**: 建议先做文档审计确认吸引子可发现，再做代码深度审计。代码审计中发现文档-代码不一致时，回头补充文档审计发现。

### 设计阶段审计

- **频次**: 每个新子系统的设计文档定稿前必做一次；设计大幅修改后重做。
- **独立性**: 文档审计应由独立子 agent 执行，不由设计者（当前 session 的主 agent）自审。
- **修复优先级**: P0（设计逻辑断裂）必须在进入实现前修复。P1（概念模糊）应在实现开始前修复。
- **与全面审计的关系**: 设计阶段审计完成后，进入实现阶段时应再做一次全面审计。设计阶段审计的发现可直接作为全面审计维度 B 的输入。
- **与代码审计的关系**: 设计阶段审计通过后，代码实现完成时应做代码深度审计（deep-audit-prompts.md），验证实现是否与设计一致。
