# AGE 文档审计提示词

> **定位**: 基于吸引子引导工程（AGE）方法论的文档审计提示词，用于独立审计项目文档体系是否有效承载吸引子、维护轨迹收敛能力。
> **适用场景**: 新项目初始化后首次审计、阶段性文档健康检查、owner-doc 更新后的一致性验证。
> **执行前提**: 审计者必须先阅读 AGENTS.md 和 docs-for-ai/INDEX.md（或 docs/index.md），理解当前项目的路由基线和文档职责划分。

---

## 1. AGE 文档审计的核心问题

AGE 把工程文档分为三层：

- **吸引子层（Attractor）**: 系统应长期收敛向的稳定结构。承载者是一小组 owner docs（design、architecture、context）。
- **轨迹层（Trajectory）**: 实际演化路径的记录。承载者是 plans、logs、bugs、audits。
- **控制层（Control）**: 持续校正轨迹的局部机制。承载者是 routing（路由）、verification（验证）、audit prompts（审计提示词）、skills。

**AGE 文档审计不是"检查文档写得漂不漂亮"。它回答一个前提问题：**

> 如果下一次 AI session 打开这个仓库，它能否仅凭文档结构就收敛到正确的方向，而不是逐步漂移？

这个前提问题分解为六个审计维度。

---

## 2. 审计维度

### 维度 A: 吸引子可发现性（Attractor Discoverability）

**问题**: 新 session 能否在 3 步以内定位到当前吸引子？

**检查项**:

1. **入口文件存在且有效**: AGENTS.md（或等价入口）是否指向当前活跃的 owner docs？
2. **路由可达**: 从入口文件出发，能否通过不超过 2 层链接找到每个关键 owner doc？
3. **吸引子载体可识别**: 读者能否区分哪些文档是吸引子载体（stable owner docs），哪些是轨迹记录（dated records），哪些是控制工具（prompts, checklists）？
4. **优先级声明存在**: owner docs 之间是否有明确的优先级/先验声明（如 "architecture/README.md owns precedence, design docs own local contracts"）？
5. **活跃状态标注**: project-context 或等价文件是否标注了当前活跃需求、活跃 owner doc、文档新鲜度？

**判定标准**:

| 等级 | 含义 |
|------|------|
| 健康 | 入口 → 路由 → owner doc 三步闭环，优先级和活跃状态明确 |
| 退化 | 路由可达但缺少优先级声明或活跃状态过期 |
| 断裂 | 新 session 无法通过文档定位当前吸引子 |

### 维度 B: Owner-Doc 内容质量（Owner-Doc Content Quality）

**问题**: owner docs 的内容本身是否准确、完整、精确、可实现？

这是文档审计最重的维度。维度 A 检查"能不能找到"，维度 B 检查"找到了之后读到的内容值不值得信任"。

AGE 视角下，owner doc 是吸引子的载体层。载体层的质量直接影响吸引子的精确性——如果载体层模糊、矛盾、过时或不可实现，吸引子就退化为一句口号，系统无法向其收敛。

本维度分为五个子检查组。

#### B.1 内部一致性（Internal Consistency）

同一份 owner doc（或同一组相互引用的 owner docs）内部是否自洽？

**检查项**:

1. **术语一致**: 同一概念是否始终使用同一术语？是否存在同一文档中"A 模块"和"Module A"混用、有时叫"引擎"有时叫"运行时"的情况？
2. **无自相矛盾**: 文档前半部分的断言是否被后半部分的描述推翻？是否有"原则上说 X，但实际允许 Y"却未解释为何豁免的情况？
3. **层级一致**: 如果文档声明了分层架构（如 Layer 1-4），每一层的接口定义是否遵守了自己声明的依赖规则？
4. **交叉引用一致**: 文档 A 引用文档 B 时描述的内容，是否与文档 B 的实际内容一致？
5. **数据流一致**: 如果文档描述了数据流，流经的每个节点和转换是否可追踪、无断裂？

**判定标准**:

| 等级 | 含义 |
|------|------|
| 自洽 | 术语、断言、层级、引用全部一致，无不解释的豁免 |
| 局部矛盾 | 存在不影响核心结论的小矛盾（如示例中用了旧名称），但核心断言链完整 |
| 核心矛盾 | 关键断言之间互相矛盾，读者无法确定应遵循哪个 |

#### B.2 设计完整性（Design Completeness）

owner doc 是否覆盖了它声称要覆盖的领域的关键方面？是否有应该存在但缺失的设计决策？

**检查项**:

1. **接口定义完整**: 声明的接口是否包含所有必要的输入、输出、前置条件、后置条件、错误情况？还是只有"成功路径"？
2. **生命周期覆盖**: 如果文档描述了一个有状态的对象或流程，是否覆盖了创建、使用、销毁的全生命周期？是否遗漏了异常路径和边界条件？
3. **配置维度完整**: 如果文档描述了一个可配置的系统，是否列出了所有配置维度、默认值、合法范围？还是只说"可配置"？
4. **交互契约完整**: 如果文档描述了模块间交互，是否明确了同步/异步、超时、重试、幂等性、错误传播？
5. **非功能需求覆盖**: 是否考虑了性能、安全、并发、可观测性？还是只有功能描述？
6. **已知限制声明**: 是否明确声明了当前设计的已知限制和非目标？（声称无所不能的文档不可信）

**完整性不是要求"覆盖一切"，而是要求"声明了的范围内没有遗漏"。**

判定方法：对每个 owner doc 声明的职责范围，检查是否在声明的范围内有明显缺失。不要求超出声明范围。

#### B.3 精确性与先进性（Precision and Advancement）

owner doc 是精确的"方程"还是模糊的"口号"？设计是否体现了超越行业平均水平的思考？

**AGE 视角**: 吸引子的精确性决定了收敛的质量。模糊的吸引子只能阻止明显的错误，精确的吸引子能引导系统向更优结构收敛。同时，如果吸引子退化为行业平均解（如"用标准三层架构"），它不提供收敛方向——AI 会自然收敛到平均解，不需要吸引子。

**检查项**:

1. **断言可证伪**: 每个关键设计断言是否可以明确地被违反？还是写成"应该保持良好的模块化"这种无法判定对错的表述？
2. **规则可执行**: 设计规则是否具体到可以直接转化为 lint 规则、测试断言或审计检查项？还是需要"理解精神"后自行判断？
3. **原创概念清晰**: 如果文档引入了本项目特有的概念（如 Nop 的"可逆计算"、XDSL 的"x:extends"语义），是否给出了足够精确的定义，使新读者可以独立判断一个实现是否符合该概念？
4. **非平庸设计**: 核心设计决策是否体现了针对性的思考（解决本项目的特定问题），还是可以原封不动套用到任何项目？
5. **抽象层次适当**: 设计是否在正确的抽象层次上描述？既不是过于具体的代码伪实现，也不是过于抽象的"关注点分离"？
6. **决策理由可追溯**: 重要的设计决策是否解释了"为什么这样设计"而不是仅仅描述"设计了什么"？是否记录了被否决的替代方案及否决理由？

**判定标准**:

| 等级 | 含义 |
|------|------|
| 精确 | 关键断言可证伪，规则可执行，原创概念有严格定义，设计决策有理由 |
| 模糊-可用 | 整体方向正确但部分关键断言不够精确（如缺少边界条件），可以补充而不需要重写 |
| 退化-口号 | 大量"应该"、"合理"、"良好实践"类表述，无法判定合规与否；或完全套用行业模板无针对性 |

#### B.4 可实现性（Implementability）

owner doc 描述的设计在当前技术栈、当前团队能力、当前时间约束下是否可实现？

**检查项**:

1. **技术栈兼容**: 设计是否与项目声明的技术栈一致？是否隐含假设了项目未引入的依赖或能力？
2. **实现路径可推导**: 从设计描述到第一个可工作的代码，是否存在可推导的实现路径？还是需要大量"自行理解"的隐含步骤？
3. **增量可实现**: 设计是否可以增量实现（先做核心、再做扩展）？还是必须"全部完成才能工作"？
4. **依赖可行**: 设计所依赖的外部系统、库、服务是否真实可用？版本约束是否明确？
5. **测试可验证**: 设计的每个关键行为是否可以通过自动化测试验证？是否设计了不可测试的架构约束？

**判定标准**:

| 等级 | 含义 |
|------|------|
| 可实现 | 技术栈兼容，有清晰的增量实现路径，依赖真实可用 |
| 部分可实现 | 核心路径可实现但存在未解决的依赖或技术不确定性 |
| 不可实现 | 设计隐含假设了不存在的能力，或需要一次性完成无法增量的改造 |

#### B.5 文档-代码对齐（Doc-Code Alignment）

owner docs 描述的结构是否与当前代码和配置一致？

**检查项**:

1. **代码存在性验证**: owner doc 中描述的模块、接口、配置文件是否在代码库中真实存在？
2. **行为一致性验证**: 文档描述的行为（如"模块 A 不依赖模块 B"）是否与代码实际依赖关系一致？
3. **过时内容识别**: owner doc 中是否有描述已不存在的模块、已移除的功能、已改变的 API？
4. **生成产物边界**: 文档中是否清楚标明哪些文件是生成的（不应手改）、哪些是手写的？
5. **命名一致性**: 文档中的术语、模块名、接口名是否与代码中的实际命名一致？

**抽样方法**:

- 对每个 owner doc，抽查 2-3 个关键断言，验证代码中是否真实存在且行为一致
- 对文档间交叉引用，验证引用目标是否存在且内容对齐

### 维度 C: 轨迹记录完整性（Trajectory Record Completeness）

**问题**: 轨迹记录（plans、logs、bugs）是否足以让未来 session 恢复历史上下文？

**检查项**:

1. **计划闭合证据**: 每个已完成的 plan 是否有 closure audit 证据？
2. **日志覆盖度**: 最近的 significant code change 是否有对应的 dev log 记录？
3. **Bug 记录质量**: 非平凡的 bug 是否有根因分析和回归测试记录？
4. **轨迹-吸引子关联**: plan 的 closure 是否明确裁决了"是否需要更新 owner doc"？
5. **历史可恢复**: 如果一个未来 session 需要理解"为什么系统是现在这个样子"，轨迹记录是否足以回答？

**判定标准**:

| 等级 | 含义 |
|------|------|
| 完整 | 每个闭合 plan 有审计证据，每个 significant change 有日志，每个 bug 有根因 |
| 部分完整 | 大部分闭合轨迹有记录，但存在无日志的 significant change 或无审计的 plan |
| 缺失 | 轨迹记录稀疏，未来 session 无法恢复关键决策上下文 |

### 维度 D: 控制机制有效性（Control Effectiveness）

**问题**: 路由、验证、审计等控制机制是否真正引导轨迹向吸引子收敛？

**检查项**:

1. **路由无歧义**: AGENTS.md 的任务路由表是否覆盖主要任务类型？是否存在"不知道该读哪个文档"的场景？
2. **验证命令真实**: 文档中引用的验证命令（build、test、lint）是否是真实可执行的？
3. **审计 prompt 适配**: 现有审计 prompt 是否针对本项目的保护区域和关键风险定制？还是停留在通用模板？
4. **Protected Areas 可执行**: 保护区域（plan-first、ask-first）的定义是否具体到可执行的程度？
5. **去重机制存在**: 审计和评审是否有去重规则，避免重复报告已知问题？

### 维度 E: 抗漂移能力（Drift Resistance）

**问题**: 文档体系在 AI 高速迭代下能否抵抗结构性漂移？

**检查项**:

1. **自验证陷阱防护**: 是否有机制防止"同一 session 既生成又判断完成"？（如独立 closure audit）
2. **增量更新路径**: owner doc 是否有明确的增量更新规则，还是只有"全部重写"一种方式？
3. **文档新鲜度监控**: 是否有机制检测文档过时（如 freshness 标注、定期审计触发）？
4. **单一职责**: 每个文档目录是否只承担一种职责？是否存在职责混淆（如 plan 文件混入了 design 内容）？
5. **命名规范执行**: stable docs 是否使用稳定文件名？dated records 是否包含日期？

## 3. 审计执行流程

### Step 1: 收集基线

读取以下文件（不读代码，只读文档结构）:

1. `AGENTS.md` — 路由表、保护区域、验证命令
2. `docs-for-ai/INDEX.md`（或 `docs/index.md`）— 文档路由和目录职责
3. 项目上下文文件 — 应用层项目读 docs/context/project-context，Nop 平台读 docs-for-ai/00-start-here/project-context
4. 所有 owner doc 的目录列表（不读内容，只看结构）
5. 最近的 dev log 目录列表

### Step 2: 逐维度评估

对每个维度，执行检查项并记录：

```markdown
### 维度 {X}: {名称}

- **评级**: 健康 / 退化 / 断裂
- **证据**:
  - {具体观察，引用文件路径和行号}
- **问题列表**:
  - {每条问题，标注严重程度}
```

维度 B（Owner-Doc 内容质量）需要额外按五个子检查组输出：

```markdown
### 维度 B: Owner-Doc 内容质量

- **B.1 内部一致性**: {自洽 / 局部矛盾 / 核心矛盾}
- **B.2 设计完整性**: {完整 / 范围内部分缺失 / 明显遗漏}
- **B.3 精确性与先进性**: {精确 / 模糊-可用 / 退化-口号}
- **B.4 可实现性**: {可实现 / 部分可实现 / 不可实现}
- **B.5 文档-代码对齐**: {对齐 / 局部脱节 / 严重脱节}
- **综合评级**: {健康 / 退化 / 断裂}
- **证据**:
  - {每个子检查组的证据，引用具体文档路径和断言位置}
- **问题列表**:
  - {每条问题，标注严重程度和所属子检查组}
```

### Step 3: 吸引子覆盖验证

从 owner doc 中抽取 3-5 个关键断言（如"模块 A 不依赖模块 B"、"X 接口是唯一公共入口"），抽样验证代码中是否真实成立。记录每条断言的验证结果。

### Step 4: 模拟新 Session

设计 2-3 个典型场景（如"修改服务层业务逻辑"、"新增 ORM 实体"、"修复跨模块 bug"），仅凭文档路由决定该读哪些文件、受什么约束。记录路由是否无歧义。

### Step 5: 汇总与评级

```markdown
## 总评

- **A 吸引子可发现性**: {评级}
- **B Owner-Doc 内容质量**: {评级}（B.1 内部一致性 / B.2 完整性 / B.3 精确性 / B.4 可实现性 / B.5 代码对齐）
- **C 轨迹记录完整性**: {评级}
- **D 控制机制有效性**: {评级}
- **E 抗漂移能力**: {评级}

## 总体判定

- [ ] 健康 — 文档体系有效承载吸引子，可支持 AI 高速迭代
- [ ] 需维护 — 存在退化或局部断裂，但可在短期内修复
- [ ] 需重建 — 文档体系与代码严重脱节，无法有效引导轨迹收敛

## 优先修复建议

1. {最高优先级的修复项}
2. {次高优先级的修复项}
3. ...

## 审计盲区自评

{本次审计可能遗漏的问题类型}
```

## 4. 严重程度标准

| 等级 | 含义 | 示例 |
|------|------|------|
| **P0** | 吸引子断裂 — 新 session 会被误导到错误方向 | owner doc 描述的模块边界与代码实际完全不同；路由表指向不存在的文件；核心设计断言自相矛盾 |
| **P1** | 收敛能力退化 — 轨迹会逐步漂移但不会立即出错 | 缺少优先级声明导致两个 owner doc 矛盾时无法裁决；验证命令是占位符；关键接口定义遗漏了错误路径；设计断言不可证伪 |
| **P2** | 维护成本增加 — 不影响正确性但增加理解成本 | 轨迹记录不完整；部分文档命名不规范；术语混用但核心断言链完整；审计 prompt 未定制 |
| **P3** | 理想化改进 — 当前可工作但有优化空间 | 日志格式可以更紧凑；目录结构可以更扁平；设计决策缺少否决理由记录 |

## 5. Nop 平台特定检查

如果被审计项目是 Nop 平台或基于 Nop 的项目，额外检查以下项：

1. **docs-for-ai vs docs 边界**: 是否有开发者在 `docs/` 或 `docs-for-ai-old/` 中寻找信息？`docs-for-ai/INDEX.md` 是否明确声明了规范性？
2. **生成文件边界文档化**: 是否有文档清楚说明 `_` 前缀文件是生成的、不应手改？
3. **Delta 定制路径**: 是否有路由指引"需要定制时应修改哪个 Delta 文件"？
4. **模型优先链路文档**: 是否有文档描述 model → codegen → dao → meta → service → web 的完整链路？
5. **XDSL schema 可发现性**: 新开发者能否找到 xdef schema 定义并理解 XDSL 的扩展语义？

| Prompt | 审计对象 | 与本文档的关系 |
|--------|---------|---------------|
| `deep-audit-prompts.md` | 代码质量和架构合规 | 代码审计前应先通过文档审计确认吸引子可发现 |
| `open-ended-adversarial-review-prompt.md` | 代码开放审查 | 对抗审查中发现"文档与代码不一致"时应触发文档审计 |
| `plan-reviewer-prompt.md` | 计划可执行性 | 计划审计假设文档路由可用；文档审计验证这个假设 |
| `plan-closure-audit-prompt.md` | 计划闭合质量 | 闭合审计中的"owner-doc 裁决"依赖文档体系健康 |
| `documentation-routing-audit-prompt.md` | 文档路由表正确性 | 路由审计是本文档维度 A（吸引子可发现性）的一个子集 |
| **本 prompt** | **文档体系整体健康** | **审计文档是否有效承载吸引子，覆盖六个维度（含内容质量五子组）** |

## 7. 输出保存

审计结果保存到 `ai-dev/audits/{日期}-doc-age-audit/` 目录下：

```
ai-dev/audits/YYYY-MM-DD-doc-age-audit/
├── summary.md          # 总评和优先修复建议
├── dimension-A.md      # 吸引子可发现性详细发现
├── dimension-B.md      # Owner-Doc 内容质量详细发现（5 个子检查组）
├── dimension-C.md      # 轨迹记录完整性详细发现
├── dimension-D.md      # 控制机制有效性详细发现
└── dimension-E.md      # 抗漂移能力详细发现
```

如果审计范围较小，可以将所有维度合并到 `summary.md` 中。

## 8. 执行建议

- **频次**: 项目初始化后必做一次；此后每 2-4 周或在 significant owner-doc 变更后做一次。
- **独立性**: 文档审计应由独立子 agent 执行，不由当前 session 的主 agent 自审。
- **修复优先级**: P0（吸引子断裂）应立即修复，P1 应在下一个 session 开始前修复。
- **与代码审计的关系**: 建议先做文档审计确认吸引子可发现，再做代码深度审计。代码审计中发现文档-代码不一致时，回头补充文档审计发现。
