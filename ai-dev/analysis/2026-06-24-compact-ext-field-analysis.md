# 紧凑扩展字段方案分析

> Status: resolved
> Date: 2026-06-24
> Scope: nop-sys 模块、ORM 模型设计
> Conclusion: 推荐采用位掩码模式的紧凑扩展字段方案，表名 `NopSysCompactExtField`，放在 nop-sys 模块

## Context

### 需求概述

用户提出一种**紧凑扩展字段方案**，核心思路是：

- 通过一个 `varchar` 字段存储多个可枚举字段
- 每个字符位置对应一个独立的枚举值
- 通过配置表定义每个字符位的含义、编码名称、显示名称和字典映射
- 默认情况下，字符 `0` 对应 `false`，字符 `1` 对应 `true`

### 现有方案对比

项目中已存在 `NopSysExtField` 扩展字段方案，采用 **KV 表模式**：

| 特性 | 现有 KV 表方案 | 紧凑扩展字段方案 |
|------|--------------|----------------|
| 存储方式 | 每行存储一个字段值 | 单行存储多个字段值 |
| 数据结构 | 稀疏表 | 紧凑字符串 |
| 查询效率 | 需多表 JOIN | 单表查询 |
| 扩展性 | 无限扩展 | 受限于字符串长度 |
| 字段类型 | 支持多种类型 | 仅支持枚举类型 |

## Analysis

### 方案一：位掩码模式（推荐）

#### 核心思路

使用 `varchar(N)` 字段，每个字符代表一个布尔/枚举字段：

```
字段值: "10A3"
        ↑↑↑↑
        │││└── 第4位：枚举值 "3"
        ││└── 第3位：枚举值 "A"
        │└── 第2位：布尔值 false (0)
        └── 第1位：布尔值 true (1)
```

#### 设计要素

**1. 配置表结构**

| 字段 | 类型 | 说明 |
|------|------|------|
| `table_name` | VARCHAR | 目标表名 |
| `field_name` | VARCHAR | 目标字段名（扩展字段名） |
| `position` | INT | 字符位置（从1开始） |
| `code_name` | VARCHAR | 编码名称（程序内部使用） |
| `display_name` | VARCHAR | 显示名称（UI展示） |
| `dict_name` | VARCHAR | 关联字典名（可选） |
| `default_value` | CHAR(1) | 默认值 |
| `is_required` | BOOLEAN | 是否必填 |

**2. 数据访问模式**

- **写入**：根据配置将多个枚举值拼接为字符串
- **读取**：根据配置解析字符串的每个字符位
- **查询**：使用 SQL 的 `SUBSTRING` 函数进行条件筛选

**3. 默认行为**

当 `dict_name` 为空时：
- `'0'` → `false`
- `'1'` → `true`

#### 优点

- **存储空间高效**：N 个字段仅需 N 个字符
- **查询性能好**：无需 JOIN，单表即可完成查询
- **Schema 稳定**：新增字段无需修改表结构
- **迁移成本低**：仅需修改配置，无需数据迁移

#### 缺点

- **类型受限**：仅支持枚举/布尔类型，不支持数值、日期等
- **长度限制**：受限于 `varchar` 最大长度（通常 65535）
- **查询复杂度**：复杂条件查询需要字符串操作函数
- **可读性差**：原始数据不易直接理解

#### 适用场景

- 需要频繁扩展布尔/枚举字段的表
- 查询以精确匹配为主，较少范围查询
- 需要保持 Schema 稳定的多租户场景

---

### 方案二：JSON 嵌入式模式

#### 核心思路

使用单个 JSON 字段存储扩展属性：

```json
{
  "flag1": true,
  "flag2": false,
  "status": "ACTIVE",
  "category": "TYPE_A"
}
```

#### 优点

- **灵活性高**：支持任意类型和嵌套结构
- **自描述性**：字段名即含义，无需额外配置
- **查询能力强**：现代数据库支持 JSON 路径查询

#### 缺点

- **存储空间大**：JSON 格式有额外开销
- **索引支持有限**：部分数据库对 JSON 字段索引支持不佳
- **类型松散**：缺乏强类型约束

---

### 方案三：传统 ALTER TABLE 模式

#### 核心思路

每次新增字段时执行 `ALTER TABLE ADD COLUMN`

#### 优点

- **类型完整**：支持所有数据库类型
- **查询高效**：原生字段索引支持
- **事务安全**：数据库原生支持

#### 缺点

- **Schema 变更代价高**：大表 ALTER 可能耗时很长
- **部署复杂**：需要 DBA 配合，难以自动化
- **多租户困难**：不同租户可能需要不同字段

---

## Comparison

| 维度 | 紧凑扩展字段 | JSON 模式 | 传统 ALTER | 现有 KV 表 |
|------|-------------|----------|-----------|-----------|
| **存储效率** | 高 | 中 | 中 | 低 |
| **查询性能** | 高 | 中 | 高 | 低 |
| **类型支持** | 枚举/布尔 | 任意 | 任意 | 任意 |
| **Schema 稳定性** | 高 | 高 | 低 | 高 |
| **扩展性** | 中（长度限制） | 高 | 低 | 高 |
| **可读性** | 低 | 高 | 高 | 中 |
| **索引支持** | 中 | 低 | 高 | 低 |

## Conclusion

### 推荐方案

**推荐采用「紧凑扩展字段方案」**，理由如下：

1. **契合业务场景**：用户明确需求是存储枚举类型字段，此方案针对性强
2. **性能优势明显**：相比 KV 表方案，避免了多表 JOIN 的开销
3. **实现复杂度适中**：配置表 + 字符串解析的实现模式清晰可控
4. **与现有系统兼容**：可作为 `NopSysExtField` 的补充方案，而非替代

### 否决方案理由

- **JSON 模式**：虽然灵活，但对于纯枚举场景过于笨重，且索引支持不足
- **传统 ALTER 模式**：Schema 变更代价过高，不适用于需要快速迭代的场景
