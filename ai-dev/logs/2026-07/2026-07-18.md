## Phase 1+2+3 完成（2026-07-18 下午）

### `traceCompleteFireErrorCode()` 确认异常类型

手动运行输出明确堆栈：异常为 `nop.err.orm.entity-is-readonly`（`OrmEntity.checkReadonly()`），路径为 `tryUpdateWithVersionCheck` 版本冲突 → `orm_readonly(true)` → retry loop setter → 抛出。**不是** `ERR_JOB_FIRE_STATUS_CONFLICT`（retry loop 从没执行到 5 次）。

### @SingleSession 限制定理

关键发现：在 `@SingleSession` 中，`requireEntityById` 通过 `_makeProxy` 查询 `cache.get(entityName, id)`，命中即返回同一 Java 对象。因此：
- `baseline = requireEntityById(schedule.id)` 返回与 `schedule` 参数同一对象
- retry loop 的 reload 永远得不到新数据
- **retry loop 在 `@SingleSession` 下注定失败**

**修复**：移除所有 retry loop（`completeFireAndUpdateSchedule` 和 `cancelFire`）。单次 `tryUpdateWithVersionCheck(schedule)` — 实体已被调用方修改（dirty），flush 包含所有 dirty 属性。不再需要 delta 计算或 baseline 重试。

### Bug C 修复

在 `JobTimeoutCheckerImpl.tryMarkDispatchTimeout` 中，`completeFireAndUpdateSchedule` 返回后检查 `fire.orm_readonly()`（`tryUpdateWithVersionCheck` 在版本冲突时设此标记）。为 true 时跳过 task 取消逻辑，避免 DB 中 RUNNING fire + CANCELED tasks 的不一致。

## 调查阶段发现（上午）

### Bug A 生产成立确认

通过读取 `orm-defaults.beans.xml` 确认 `SingleSessionMethodInterceptor` 和所有 `OrmEntityDao` 共享同一个 `nopOrmTemplate` 单例：
- `nopOrmTemplate` (id=`nopOrmTemplate`, singleton)
- `nopDaoProvider` → `constructor-arg ref="nopOrmTemplate"`
- `nopSingleSessionMethodInterceptor` → `constructor-arg ref="nopOrmTemplate"`
- 所有 `OrmEntityDao` 通过 `DaoProvider` 使用同一实例

因此 `@SingleSession` 下 `requireEntityById` 确实返回 Session 缓存的同一实体实例（Bug 成立）。

### Bug D 正名

**原假说**：`tryUpdateWithVersionCheck` 在版本冲突时抛出 `OrmException`。
**实际**：`checkUpdateResult` 正确检查 `orm_disableVersionCheckError()` 旗标，为 true 时只写 log + 标记 readonly，不抛异常。方法返回 false。
**证据**：`testTryUpdateWithVersionCheckOnDetachedReturnsFalse` 通过。

### Bug E 正名

**原猜测**：`orm_unload()` 将 state 设为 PROXY 后 setter 抛异常。
**实际**：`orm_unload()` 不清除 `fullyLoaded`，`forcePropLoaded` 短路返回，不调 `requireEnhancer()`，不抛异常。
