## nop-app-erp batch 迁移运行时验证发现并修复的两处平台 Bug（2026-07-18 下午）

执行 nop-app-erp `docs/plans/2026-07-18-1600-1-batch-migration-phase-1.md` Phase 5 时，写运行时验证测试 `TestErpAllJobYamlLoading`（断言 `/nop/job/conf/` 下 19 个 `.job.yaml` 全部能被 `LocalJobConfigLoader.scanJobConfigs` 加载并经 `JsonTool.loadDeltaBeanFromResource` 反序列化为 `LocalJobConfig`），暴露两处平台 Bug：

### Bug 1：`LocalJobConfigLoader.DEFAULT_JOB_DIR` 尾部斜杠导致 `.job.yaml` 扫描静默失败

- **现象**：`scanJobConfigs` 始终返回空列表，`init()` 不注册任何 `.job.yaml` 作业。
- **根因**：`DEFAULT_JOB_DIR = "/nop/job/conf/"`（尾部 `/`），而 `VirtualFileSystem.getAllResources(path, suffix)` 拒绝尾部 `/`（抛 `nop.err.core.resource.invalid-path: 不能以/结尾`）。`scanJobConfigs` 的 try/catch 吞掉该异常并 `LOG.info("job-dir-not-available")` 后返回空——**静默失败**，无任何作业被注册。
- **影响**：nop-job-local 的「每 job 一个 `.job.yaml`」新机制自交付以来从未真正运行过（所有 `.job.yaml` 都被静默跳过）。`scheduler.yaml` 内联 jobs 段仍工作（旧机制不经 `getAllResources`），掩盖了此 Bug。
- **修复**：`nop-job/nop-job-local/.../LocalJobConfigLoader.java` `DEFAULT_JOB_DIR` 与默认 `jobDir` 字段去掉尾部 `/` → `/nop/job/conf`。同步补注释说明 VFS 拒绝尾部 `/`。
- **验证**：`TestErpAllJobYamlLoading` VFS 直查返回 19 资源 + `scanJobConfigs` 经修复后返回 19 配置。

### Bug 2：`ConfigValueResolver.compile` 先截短 config 再取默认值，触发 `StringIndexOutOfBoundsException`

- **现象**：`JsonTool.loadDeltaBeanFromResource` 解析含 `@cfg:key|default` 表达式的 YAML 时抛 `StringIndexOutOfBoundsException: Range [37, 36) out of bounds for length 36`。
- **根因**：`nop-kernel/nop-core/.../bind/resolver/ConfigValueResolver.compile` 第 49-51 行顺序错误：
  ```java
  config = config.substring(0, pos);                                  // 先截短
  defaultValue = JsonTool.parseSimpleJsonValue(config.substring(pos + 1).trim()); // 再用原 pos 取默认值 → 越界
  ```
  截短后 `config.length() == pos`，`config.substring(pos + 1)` 起点越界。
- **影响**：所有经 `JsonTool.loadDeltaBean*` 加载的 delta bean（YAML/JSON）中含 `@cfg:...|default` 表达式的字段都会解析失败。`.job.yaml` 的 `enabled`/`cronExpr` 字段全部命中（19/19）。beans.xml 的 `@cfg:` 经 `nop-ioc` 的独立 `ConfigValueResolver` 解析（不同类），不受影响——故平台启动未暴露此 Bug。
- **修复**：调换顺序——先用原 `config` 取默认值，再截短：
  ```java
  defaultValue = JsonTool.parseSimpleJsonValue(config.substring(pos + 1).trim()); // 先取
  config = config.substring(0, pos);                                               // 再截短
  ```
- **验证**：修复后 19 个 `.job.yaml` 全部成功解析为 `LocalJobConfig`（含 `enabled: "@cfg:nop.job.<name>.enabled|false"` + `cronExpr: "@cfg:nop.job.<name>.cron-expr|<cron>"`）。

## Phase 1+2+3 完成（2026-07-18 下午）

### `traceCompleteFireErrorCode()` 确认异常类型

手动运行输出明确堆栈：异常为 `nop.err.orm.entity-is-readonly`（`OrmEntity.checkReadonly()`），路径为 `tryUpdateWithVersionCheck` 版本冲突 → `orm_readonly(true)` → retry loop setter → 抛出。**不是** `ERR_JOB_FIRE_STATUS_CONFLICT`（retry loop 从没执行到 5 次）。

### @SingleSession 限制定理

关键发现：在 `@SingleSession` 中，`requireEntityById` 通过 `_makeProxy` 查询 `cache.get(entityName, id)`，命中即返回同一 Java 对象。因此：
- `baseline = requireEntityById(schedule.id)` 返回与 `schedule` 参数同一对象
- retry loop 的 reload 永远得不到新数据
- **retry loop 在 `@SingleSession` 下注定失败**

**修复**：移除所有 retry loop（`completeFireAndUpdateSchedule` 和 `cancelFire`）。单次 `tryUpdateWithVersionCheck(schedule)` — 实体已被调用方修改（dirty），flush 包含所有 dirty 属性。不再需要 delta 计算或 baseline 重试。

### Bug C 修复

在 `JobTimeoutCheckerImpl.tryMarkDispatchTimeout` 中，`completeFireAndUpdateSchedule` 返回后检查 `fire.orm_readonly()`（`tryUpdateWithVersionCheck` 在版本冲突时设此标记）。为 true 时跳过 task 取消逻辑，避免 DB 中 RUNNING fire + CANCELED tasks 的不一致。

## 调查阶段发现（上午）

### Bug A 生产成立确认

通过读取 `orm-defaults.beans.xml` 确认 `SingleSessionMethodInterceptor` 和所有 `OrmEntityDao` 共享同一个 `nopOrmTemplate` 单例：
- `nopOrmTemplate` (id=`nopOrmTemplate`, singleton)
- `nopDaoProvider` → `constructor-arg ref="nopOrmTemplate"`
- `nopSingleSessionMethodInterceptor` → `constructor-arg ref="nopOrmTemplate"`
- 所有 `OrmEntityDao` 通过 `DaoProvider` 使用同一实例

因此 `@SingleSession` 下 `requireEntityById` 确实返回 Session 缓存的同一实体实例（Bug 成立）。

### Bug D 正名

**原假说**：`tryUpdateWithVersionCheck` 在版本冲突时抛出 `OrmException`。
**实际**：`checkUpdateResult` 正确检查 `orm_disableVersionCheckError()` 旗标，为 true 时只写 log + 标记 readonly，不抛异常。方法返回 false。
**证据**：`testTryUpdateWithVersionCheckOnDetachedReturnsFalse` 通过。

### Bug E 正名

**原猜测**：`orm_unload()` 将 state 设为 PROXY 后 setter 抛异常。
**实际**：`orm_unload()` 不清除 `fullyLoaded`，`forcePropLoaded` 短路返回，不调 `requireEnhancer()`，不抛异常。
