## Bugfix：AbstractParseTreeParser.buildError 解析报错期 ClassCastException

- **关联下游**：nop-app-erp mission verify（`mvn test` app-erp-all `ErpAllWebPagesTest`/`ErpAllWebPagesCollectTest` 全 210+ 页面级联失败）。
- **根因**：`nop-kernel/nop-antlr4/nop-antlr4-common/.../AbstractParseTreeParser.java` 的 `buildError(Throwable e)` 第 143 行无条件 `RecognitionException re = (RecognitionException) e;`。当 `twoPhaseParse` 第一阶段 SLL 失败（`BailErrorStrategy` 抛 `ParseCancellationException`）且该 logger 为 TRACE 时，第 93 行 `LOG.trace("...", buildError(ex))` 把 `ParseCancellationException`（非 `RecognitionException`，二者无继承关系）传入 buildError，强转即 `ClassCastException`，**掩盖真实解析错误**。
- **证据**：bytecode `javap` 证实崩溃点为 line 93（trace 路径）→ line 143（强转）；erp 侧确认同一失败在 F7 代码改动前后 commit 均复现（非 app 回归，是平台 latent bug）。
- **修复**：buildError 先判 `e instanceof RecognitionException`，否则尝试 `e.getCause() instanceof RecognitionException`（BailErrorStrategy 的 cause 才是真正的 RecognitionException），二者皆否则退回 `new NopException(ERR_ANTLR_PARSE_FAIL, e)` 不再强转。
- **验证**：`mvn test -pl nop-kernel/nop-antlr4/nop-antlr4-common,nop-kernel/nop-xlang -am` 全绿；`mvn install -pl nop-kernel/nop-antlr4/nop-antlr4-common -am -DskipTests` 装入本地仓库后 erp 侧 `mvn test` 全 154 模块 BUILD SUCCESS。
- **副作用**：修复后报错从含糊的 ClassCastException 变为精确的 `nop.err.antlr.common.parse-fail`（含 offendingToken/期望 token），**首次暴露** erp 侧 F7 提交在 view.xml `<c:script>` 内误用 XLang 不支持的 `function(...){}` 表达式（XLang 仅支持 `functionDeclaration` 语句与 `(p)=>{...}` 箭头表达式），下游据此完成 app 修复。
