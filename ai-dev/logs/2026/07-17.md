### 技术要点

- **接线点选择（BizModel 层而非 executor 层）**：executor 是无状态编排器，注入 scorer 会破坏其职责单一性 + 构造依赖膨胀。评分落盘逻辑（saveEntity 六行）全部在 `NopMetaQualityScoreBizModel`，在 BizModel 层调 `computeQualityScore` 即复用全部既有逻辑，零复制。
- **失败隔离（flush + clearSession）**：对齐既有 per-rule 隔离模式——每表评分成功后 `flushSession` 保留已落盘评分行，失败时 `clearSession` 清理未刷出脏实体但不回滚已 flush 的 checkpoint store（QualityResult）与已成功的评分行。
- **derived-column SKIP 触发自然评分失败**：测试 (c) 用 not_null 规则 `column=<expr_1>`（sql 视图合成列名）经 `requireColumn→isDerivedColumnName→skipDerivedColumn` 路径返回 SKIP（不伪造值、不静默跳过），使 FAIL 表只有 SKIP 结果 → scorer 抛 `ERR_SCORE_ALL_SKIP`，构成自然的 per-table 评分失败（无需 mock scorer）。

### 技术要点

- **GraphQL save 语义**：对无逻辑删除的实体（Module），GraphQL `save` action 为 insert 语义（非 upsert）——checkUniqueForSave 对已存在 PK 抛 entity-already-exists。故 Module 的 UPDATE 事件经 releaseModule mutation action 发布；save override 仍负责 CREATE 检测（before=null→CREATE）。save override 的 UPDATE 检测逻辑正确（按 PK 加载 before 区分），但对 Module 仅 CREATE 路径经 GraphQL save 触发。
- **快照序列化**：从实体模型列（IEntityModel.getColumns）构建 LinkedHashMap（Date→epoch millis）后 JsonTool.stringify，避免 ORM 内部状态混入快照。非 ORM/Map 对象回退 stringify→parse。
