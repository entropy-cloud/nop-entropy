`01-architecture-baseline.md` §4.4.1 新增 **D1.5 — 混合端点（entity ↔ external/sql）同库 JOIN 聚合机制**：

- **连接载体（候选 A 选定）**：复用 external/sql 端点 NopMetaDataSource `withConnection`，在该单一连接上同时访问 external 物理表 + entity 物理表（直查 `NopMetaEntity.tableName`，绕过 ORM session/EQL）。拒绝候选 B（平台 `IJdbcTransaction.getConnection()`）——external 物理表几乎不可能在平台库连接可见，反向可见性远差于 A。
- **同库判定（querySpace 字符串不可靠 + 连接可达性实测）**：显式承认 querySpace 字符串相等对混合端点语义不可靠（entity querySpace 是 ORM 注册体系，external querySpace 是 NopMetaDataSource 注册体系，两套独立注册表）。改用连接可达性实测：选定 external `withConnection` 作为基准连接，实测 entity 物理表是否在 `DatabaseMetaData.getTables(null, entitySchema, entityTableName, null)` 结果集中。可见 → 同库；不可见 → 显式失败 `ERR_AGGR_JOIN_MIXED_CROSS_DB_DEFERRED`（指向 1500-2，不静默降级 D5）。
- **schema 限定**：JOIN SQL 中两侧表名显式 `<schema>.<table>` 限定（entity 用 `NopMetaEntity.dbSchema`，external 用 `NopMetaTable.schema`，可空则不限定）。
- **measure/dimension 物理列解析（F1-2：仅 table 侧套用 side resolver）**：entity 侧经 `entityFieldId → metaEntityId → columnCode`（不套用 `JoinExternalSideResolver`）；external/sql 侧经 side（必填）+ `MetaTableFieldResolver`。新增 `JoinMixedSideResolver` 内部类按 entityFieldId 是否为 NopMetaEntityField PK 且归属 entity 端点判定侧别。

- **失败路径显式化（#24）**：不可同库 / 缺 datasource / right / self-join / side 缺失 / 列不存在 / SQL 执行失败 均抛 ErrorCode，不静默降级、不静默空 items、不伪造值。

## nop-gateway Dynamic Configuration & Hot-Reload Design Analysis

**Design doc**: `ai-dev/design/nop-gateway/00-dynamic-configuration-design.md`

Completed a comprehensive design analysis for nop-gateway's hot-reload/dynamic configuration system. Key decisions:

- **Config Model**: Hybrid (XDSL base + Admin API runtime overrides + in-memory merge). Rejected pure XDSL (no runtime) and pure API+DB (loses Delta/GitOps).
- **Distribution**: Layered — VFS watch (single-node) + nop-config polling (cluster fallback) + optional Redis Pub/Sub (cluster fast-path). Rejected etcd as too heavy for Nop's deployment range.
- **Atomic Updates**: Two-phase (validate → apply) with `AtomicReference<GatewayModel>` Copy-on-Write swap. All-or-nothing: if any route is invalid, the entire batch is rejected.
- **Filter Chain Hot Reload**: `AtomicReference<FilterChain>` per route. IoC bean resolution at chain-build time. In-flight requests hold old chain reference.
- **Credential Rotation**: Versioned credentials with `activeFrom`/`retireAt` lifecycle. Grace period support for in-flight requests.
- **Rollback**: DB-backed append-only versioned snapshots (`gateway_config_versions` table). Rollback creates a new version, never mutates history.
- **Integration**: nop-config's `ConfigChangeApplier` + `IConfigSource` for change detection. XDef schema for validation. IoC filter bean resolution.
- Added entry in `ai-dev/design/README.md`.

- **`normalizeSchemaForMatch` null 归一**：`null/空串/纯空白` → `null`，使 JDBC NULL 行（部分方言/旧数据）与新同步的 null schema 行正确匹配（Java `Objects.equals(null, null) == true`）。EQL `isNull` 路径不再需要（已绕过 EQL）。

### 技术要点

- **大小写不敏感命名空间规范化**：H2 ResultSetMetaData 返回大写列名（"K"），而 sql sourceSql/external buildSql 中可能为小写（"k"）。两者代表同一物理列，属合法规范化而非 GENUINE 错配。`crossDbMerge` 的 join key 提取 + Anti-Hollow 校验均大小写不敏感；但若字段真不存在（如 "__nope__"），仍显式失败（Anti-Hollow 保证成立）。
- **JOIN FROM 子句别名**：sql 子查询 `(<sourceSql>)` 必须带别名（H2 语法要求）。单表 SELECT 用 `_t`，JOIN 用 `t1`/`t2`——`tableFromForJoin(table, alias)` 生成带显式别名的 FROM 子句，避免 `(subquery) _t t1` 双别名语法错误。
- **同库 JOIN 仅投影 join key**：与 entity-entity 同库路径一致——不投影全部列（部分列名为 SQL/EQL 保留字会 parse-fail），仅投影 join key 列（右表加 alias 前缀防冲突）。跨库拼接路径返回各侧全部列。
- **LIMIT 参数绑定**：`fetchTableRows` 加 MAX_CROSS_DB_ROWS+1 上限防 OOM，`executeJdbcQuery` 按序绑定 filter 参数 + limit/offset。
