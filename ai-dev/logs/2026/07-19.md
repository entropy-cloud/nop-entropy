### Phase 3（OFFSET/LIMIT + Lineage DoS + 配套工程缺陷）

- **AR-04**：抽出公共 helper `SqlPagination.appendLimitOffset(sb, limit, offset, dialect)`（已有），把剩余 3 处 inline LIMIT/OFFSET 拼接（`MetaJoinExecutor` 同库 entity-entity + 同库 table-table + 单侧取数 buildTableSelectSql）全部迁移；MySQL offset-only 自动补 `LIMIT 18446744073709551615`。10 处拼接点统一调用 helper。
- **AR-09**：`NopMetaLineageEdgeBizModel.buildLineageGraph/buildTableNameIndex` 从 `dao().findAll()` 改为 `findAllByQuery(QueryBean with limit=max+1)`，超限抛 `ERR_LINEAGE_GRAPH_TOO_LARGE` / `ERR_LINEAGE_TABLE_INDEX_TOO_LARGE`（默认 100_000，可经 `@cfg:nop.metadata.lineage.max-edges` / `@cfg:nop.metadata.lineage.max-tables` 配置）。
- **维度09-08**：`MetaCatalogCollector.countRows` 与 `MetaTableProfiler.queryLong` 的 `throw new SQLException(...)` 改为 `throw new NopException(ERR_*_AGGREGATE_NO_ROW).param("sql", sql)`（不混用 SQLException 控制流）。
- **AR-12/13**：`MetaQualityRuleExecutor.queryLong/queryTimestamp/querySingleValue` 3 个 helper 全部从 `Statement.executeQuery(sql)` 改为 `PreparedStatement`；catch 块补 `.param("error", messageOf(e))`（原 ErrorCode 描述含 `{error}` 占位符但 throw 时只设 `sql`）。
- **维度13-03 custom_sql 沙箱化（核心）**：`judgeCustomSql` 入口新增 `validateCustomSqlSandbox`——拒绝含分号/UNION/INTO OUTFILE/LOAD DATA/CALL/EXEC/SHUTDOWN/DROP/TRUNCATE/ALTER/CREATE/GRANT/REVOKE/INFORMATION_SCHEMA/MYSQL.USER 等危险关键字（trim+大写 contains 匹配）；details 写入 `sqlHash`（SHA-256 短摘要 16 字符，用于审计追溯）；querySingleValue 改 PreparedStatement（辅助，不解决注入，因 SQL 文本是用户配置）。新增 ErrorCode `ERR_QUALITY_CUSTOM_SQL_BLOCKED`。
- **AR-14 / 维度09-11**：`NopMetaDataSourceBizModel.collectCatalogForTable` 找不到 metaTableId 时新增 `ERR_TABLE_NOT_FOUND`（原误用 `ERR_DATASOURCE_NOT_FOUND` 且把 metaTableId 写入 dataSourceId 参数）。

### Phase 4（data-auth 行级权限 + webhook SSRF 防护）

- **维度13-02 data-auth**：3 个目标实体（NopMetaDataSource / NopMetaQualityCheckpoint / NopMetaModelChangedEvent）均在 nop-metadata.orm.xml 含 createdBy 列（NopMetaModelChangedEvent 用 changedBy）；data-auth.xml 写入 admin 角色（无过滤全量）+ default 角色（user，按 `createdBy == $user.userId` 过滤；NopMetaModelChangedEvent 用 changedBy）双层规则。
- **维度13-04 webhook SSRF（fail-closed）**：`CheckpointActionDispatcher.dispatchWebhook` 入口新增：
  - URL 协议白名单（http/https），拒绝 file:/ftp:/data:/gopher: 等
  - 主机 allowlist：默认禁内网（RFC1918 + 169.254 + loopback + IPv6 ::1）；显式配 `nop.metadata.checkpoint.webhook-allowed-hosts` 后允许
  - method 白名单（POST/PUT）
  - 显式 timeout（默认 30s，可经 `nop.metadata.checkpoint.webhook-timeout-seconds` 配置）
  - 新增 ErrorCode `ERR_CHECKPOINT_WEBHOOK_URL_BLOCKED` / `ERR_CHECKPOINT_WEBHOOK_METHOD_BLOCKED`
  - BizModel 经 `@InjectValue` 加载 config 后通过 `configureWebhookSsrf(csv, timeout)` 注入到 dispatcher

## nop-metadata expression 型 Measure 列级血缘（Plan 2026-07-18-1800-1，Opt-followup-impl）

**Plan**: `ai-dev/plans/2026-07-18-1800-1-nop-metadata-expression-measure-column-level-lineage-impl.md`

把 roadmap 唯一 `planned` 项 `Opt-followup-impl` 推进到 `done`：依 design-first plan `2026-07-18-1500-1` 的 D1-D5 裁定 + 本 plan 新增的 D6 replace 语义，在 `NopMetaLineageEdgeBizModel` 实现 `extractMeasureLineage(metaTableId)` action——加载表的所有 `NopMetaTableMeasure` → 对每个 `expression != null` 的 measure 经 D5 契约提取列引用 → flat-collect 产自环边（D1+D3+D4）→ D5 失败处理分层（表级前置失败直接中断 / per-measure validator 失败隔离）→ D6 replace 语义重抽 → 返回 `{extractedEdgeCount, unresolved, errors}`。新增 dict `measure_parse` 值，新增 8 条端到端测试。

### 落地内容

- **Phase 0（design doc 裁定同步，design-only 无代码变更）**：
  - `01-architecture-baseline.md` §2.6.1 D5 段落补充「失败处理分层」精细化：(a) 表级前置失败（resolver per-table 调用，无法标 measureName，直接中断 action）+ (b) per-measure 隔离（仅覆盖 validator.validateStatic 失败）。
  - §2.6.1 新增 **D6 段落（replace 语义）**：action 开始按 `(sourceTableId=T AND targetTableId=T AND lineageSource=measure_parse)` 删旧边再全量重插；理由（expression 可变 + accumulate-only 残留过时边致召回错误）；删除条件同时指定 source+target+lineageSource 防未来 D1 扩展歧义。
  - §2.6.1 D4 段落 en label 措辞修订：移除「`Measure Expression Parse`（en，i18n-en 行）」，改为「en 选项 label 跟随既有模式」（lineage-source dict 选项 label 无 en 翻译，既有 manual/sql_parse/open_lineage/hook 均无 en 选项翻译）。
  - successor scope 段落更新引用 D1-D6。
  - 行号偏移声明：Phase 0 写入后 §2.6.1 行号偏移；按 Rule #20 不回写历史计划行号引用。

- **Phase 1（dict measure_parse 值 + 常量重新生成）**：
  - `lineage-source.dict.yaml` 追加选项 `value: measure_parse` / `label: 指标表达式解析`（紧跟既有 4 值之后）。
  - 经 `mvn install -pl nop-metadata -am -DskipTests` 触发 `nop-metadata-codegen` 重新生成 `_NopMetadataCoreConstants.java`，新增常量 `LINEAGE_SOURCE_MEASURE_PARSE = "measure_parse"`（非手编）。

- **Phase 2（extractMeasureLineage action 实现 + 端到端测试）**：
  - `NopMetaLineageEdgeBizModel` 新增 `@BizMutation extractMeasureLineage(metaTableId, context)` action，行为依 §Scope 行为契约：
    - 加载表（不存在抛 `ERR_LINEAGE_TABLE_NOT_FOUND`）
    - resolver per-table 解析字段集合（表级前置失败直接抛 ErrorCode，**不进 per-measure 隔离**）
    - replace 删除旧 measure_parse 边（条件 `sourceTableId=T AND targetTableId=T AND lineageSource=measure_parse`）
    - 加载所有 NopMetaTableMeasure，对每个 `expression != null` 的 measure per-measure try/catch：
      - 调 `ExpressionMeasureValidator.validateStatic(saveTimeLoose())` 取 identifiers
      - identifier 与字段集合 case-insensitive 比对（对齐 validator.containsIgnoreCase + sql_parse toLowerCase 先例）
      - 属于字段集合的裸名 → 产 flat-collect 自环边（D1+D3），`transformType = aggFunc != null ? aggregated : derived`（D4 边界）
      - `l.`/`r.` 限定名 → 进 unresolved（标 `join-context-deferred`）
      - 裸名不在字段集合 → 进 unresolved
      - validator 抛异常 → 进 errors（标 measureName + 错误信息），该 measure 不产边，不中断整批
    - 返回 `{extractedEdgeCount, unresolved, errors}`（对齐 `extractColumnLineageFromSql`）
  - 新增 `MetaTableFieldResolver` 字段（无状态，`new MetaTableFieldResolver()`，同既有 extractor 实例化模式）。
  - 新增私有 helper：`upsertMeasureParseEdge`（五元组去重）+ `deleteMeasureParseEdges`（D6 replace 删除）。
  - `TestNopMetaLineageEdgeBizModel.java` 新增 8 条 e2e 测试：
    - 测试 1 成功路径（extractedEdgeCount == 4 严格 + 4 边可查 + targetColumn == measureName + transformType == aggregated）
    - 测试 2 召回路径（D2 直接边查询命中）
    - 测试 3 per-measure 失败隔离（measure2 含黑名单关键字失败，measure1 边正常落盘）
    - 测试 4 BFS 非污染（getDownstream/getImpactAnalysis 不返回 T 自身）
    - 测试 5 dict 值生效（lineageSource == measure_parse 经 `_NopMetadataCoreConstants.LINEAGE_SOURCE_MEASURE_PARSE` 比对）
    - 测试 6 重抽 replace 幂等（expression 变更后旧列边删除、无残留、无重复）
    - 测试 7 边界（aggFunc null → derived）
    - 测试 8 resolver 表级前置失败（baseEntityId null → 抛 `ERR_FIELD_RESOLVE_BASE_ENTITY_NULL`）
