### Follow-up Findings (NOT YET FIXED)

After Oracle verification, user asked two follow-up questions that revealed additional issues:

**1. Memory Control for Large Projects — INEFFECTIVE**

Explore agent (bg_8478601d) analyzed memory consumption:
- `analysisResultsMap` holds FULL `ProjectAnalysisResult` per index — estimated ~560MB for nop-entropy-scale
- `persistAnalysisResult()` writes to DB but NEVER evicts from memory — data in BOTH DB AND memory
- `findSymbolsPage()` loads ALL symbols via `table.getAll().stream().filter()` then paginates in-memory
- `getFiles()` returns ALL files at once — no pagination
- `getSymbolUsages()` loads ALL files then filters — O(n) per call
- `CallGraph` stored twice (lazy + in analysisResult)
- `SymbolTable` uses dual indexing (byQualifiedName + byId) — 2x memory for symbols
- `CodeFileAnalysisResult.sourceCode` stores full source text per file — never cleared

**2. GraphQL API Design — CONVENTION VIOLATIONS**

Explore agent (bg_4ac78837) reviewed all BizQuery/BizMutation methods:
- `findSymbols` returns `PageBean<SymbolDTO>` but method name lacks `findPage_` prefix — Nop frontend auto-detects `findPage_` for pagination support
- `getById` name conflicts with CrudBizModel built-in `get` method parameter signature
- `findFiles` returns `List<CodeFileAnalysisResult>` — not paginated, not a standard name
- `getByPath`, `fileTree` — non-standard method names
- view.xml files use API URLs that may not match actual method signatures
