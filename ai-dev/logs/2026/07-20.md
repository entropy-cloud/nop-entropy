### Phase 1：关系与索引补齐

- **维度04-02**：NopMetaLineageEdge 新增 3 个 to-one（sourceTable/targetTable/pipeline）+ NopMetaTable 新增 lineageAsSource/lineageAsTarget 反向 to-many + NopMetaPipeline 新增 lineageEdges 反向 to-many。
- **维度04-01**：33 处既有 to-one 全部在父方补对应 to-many 反向，并在被引用 to-one 上加 `refPropName`。最终 ORM 文件 `<to-one>` 数 == `<to-many>` 数 == 36（grep 数量匹配校验通过）。

### Phase 2：UK / precision / dict i18n / version 列名 / DictItem isDelta / delVersionProp

- **维度04-06**：为 22 个缺自然键的实体补 `<unique-keys>`（NopMetaEntity / NopMetaDictItem / NopMetaOrmModel / NopMetaDomain / NopMetaTable / NopMetaDict / NopMetaPipeline / NopMetaManifest / NopMetaTableDimension / NopMetaTableMeasure / NopMetaTableFilter / NopMetaTableJoin / NopMetaEntityRelation / NopMetaEntityUniqueKey / NopMetaEntityIndex / NopMetaQualityRule / NopMetaQualityCheckpoint / NopMetaProfilingRule / NopMetaDataContract / NopMetaReconciliationConfig / NopMetaReconciliationEntity 等）。
- **维度04-04**：`mediumtext` domain precision 从 16777216 改为 16777215（真正映射 MEDIUMTEXT 而非 LONGTEXT），17 个列实例同步修正。
- **维度04-05**：`IX_NOP_META_TABLE_DEDUP`（unique="false" 但名字暗示唯一）重命名为 `IX_NOP_META_TABLE_LOOKUP`（语义对齐：非唯一保留非唯一）。
- **维度04-07**：23 个 dict 共 79 个 option 全部补 `i18n-en:label`；codegen 重跑后 `_vfs/i18n/en/_nop-metadata.i18n.yaml` 含全部英文 label。
- **维度04-08（Decision b）**：`delFlag` domain 从 domains 移除（裁定理据：全模块无任何 column 引用；所有删除都是物理删除）。在源文件保留裁定注释 + 重新启用判定准则。
- **维度04-09**：所有 32 个实体的 audit version 列 `code="DEL_VERSION"` 改为 `code="VERSION"`（与 nop-auth / nop-job 等平台模块一致）；同时修复 NopMetaModule 内 `moduleVersion` prop 的 `code` 从误用的 `VERSION` 改为正确的 `MODULE_VERSION`（此前与 audit version 列冲突，被 codegen 拒绝）。

- **维度04-11**：`deleteVersionProp` 重命名为 `delVersionProp`（与 `delFlagProp` 前缀一致）；同步更新 design baseline。

### Phase 3：表名 recon → reconciliation 重命名

- **维度19-01**：3 个 Reconciliation 实体表名重命名：`nop_meta_recon_config/result/entity` → `nop_meta_reconciliation_config/result/entity`；3 个对应索引 `IX_NOP_META_RECON_*` → `IX_NOP_META_RECONCILIATION_*`。code propagated to `_app.orm.xml` / deploy/sql / design docs。

### Phase 4：跨库 in-memory 合并的 SQL 语义与溢出修正

- **AR-05（NULL 语义）**：`MetaJoinExecutor.crossDbMerge` 建索引分支与查找分支均增加 `if (key == null) continue;`（与 SQL `NULL != NULL` 语义一致）；旧实现用 `String.valueOf(null) = "null"` 建索引导致两侧 null 行被错配。
- **AR-05（类型一致性，Decision a 严格类型匹配）**：新增 `verifyCrossDbKeyTypeConsistency`——从两侧抽样一行非 null key，比较 Java class；不一致显式抛 `ERR_JOIN_CROSS_DB_KEY_TYPE_MISMATCH`（旧实现 `String.valueOf(1L) = "1"` 与 `String.valueOf("1") = "1"` 会静默匹配）。
- **AR-08**：`MetaJoinExecutor.truncate` 与 `MetaAggregationExecutor.truncateCrossDb` 的 `Math.toIntExact(Long)` 替换为显式 `Integer.MAX_VALUE` 范围检查，超限抛 `ERR_PAGINATION_OFFSET_TOO_LARGE` / `ERR_PAGINATION_LIMIT_TOO_LARGE` 并附 `offset`/`limit` 参数。
- **加注释**：在 `crossDbMerge` JavaDoc 明确"NULL=NULL 不匹配"语义与 Decision 理据。
