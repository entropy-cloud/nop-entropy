# 2026-06-23 开发日志

### docs-for-ai 修正 nop-task/nop-wf 关系事实，补全 nop-task 使用文档

- 类型：ad-hoc 文档修正 + 缺口补全（无 plan）
- 触发：使用方（nop-app-erp）在评估"复杂业务流程如何用 task flow 编排"时，发现 `docs-for-ai` 里 nop-task 仅有 50 行模块概览，且存在两处事实错误：(1) 称 nop-wf 基于 nop-task 构建；(2) 未说明 task flow 的事务模型与 VFS 动态更新能力，导致使用方误判 task flow 不适合核心流程
- 验证的事实（源码核实）：
  - nop-wf 各模块 pom **不依赖** nop-task，源码零 `import io.nop.task` → 两引擎独立，旧文档的"基于构建"关系不成立（`nop-wf/{pom.xml, nop-wf-*/pom.xml}`、`grep import io.nop.task nop-wf/` = 0）
  - 事务模型：`nop-task-core/service` 无 `@Transactional`；事务由 `TransactionTaskStepDecorator`（`nop-task-ext/dao/`）按 step 可选挂载，默认复用外部 action 事务（`xbiz.xdef:31` 的 `task:name`/`task:version` + `<txn>`）
  - VFS 动态更新：`TaskFlowManagerImpl.getTaskFlowModel` → `ResourceComponentManager.loadComponentModel` → `IResourceLoadingCache`；失效经 `removeCachedModel(path)` / `clearCache(modelType)`
  - xbiz action 原生绑定 task flow：`BizActionModel.task:name/task:version`（`xbiz.xdef:30-33`，`xmlns:task="task"`）
  - `mutation` 上的 `bo:triggerStateChange` 关联 nop-fsm 状态机（`xbiz.xdef:72`）
- 做了什么：
  - `03-modules/nop-task.md`：从 50 行概览扩写为"模块概览 + 使用指南"，新增"模型与加载""事务模型""两种用法（xbiz 绑定 / Java 调用）""步骤怎么写""与其他机制的关系（nop-wf / BizModel / 状态机 / nop-rule）"节，修正 nop-wf 关系表述
  - `03-modules/nop-wf.md`：功能概览与依赖节去掉"基于 nop-task 构建"，改为"独立引擎，不依赖 nop-task"
  - `03-modules/reusable-modules-overview.md`：nop-wf 行的"依赖模块"列 `nop-task` → `nop-service-framework`；场景表去掉"基于 nop-task"
