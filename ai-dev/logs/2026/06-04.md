# AI 开发日志 — 2026-06-04

### 2026-06-04（design 文档重写）

- 按 `ai-dev/design/00-design-writing-guide.md` 重写 `ai-dev/design/crud/crud-api-codegen-design.md`：移除类签名、方法清单、代码片段、实施步骤和开放问题，改为只保留当前设计结论、分层边界、标签职责和拒绝方案。
- 设计裁定保持不变：CRUD typed API 是客户端契约层；服务端真实发布仍由 BizModel 决定；`no-api` 仅控制 CRUD typed API 生成，不影响 BizModel 注册。
- 补充裁定：客户端契约层遵循同构表达，`batchModify` 的批量输入与 `common` 公共 patch 也应使用同一个强类型输入模型；服务端 `CrudBizModel` 继续使用 `Map` 属于内部实现细节。
- 实施收口：`ICrudApi.batchModify` 已改为 `List<I>` + `I common`；`docs-for-ai/02-core-guides/api-and-graphql.md` 额外注明前端 `operationRegistry` 的 `[Map]` 仅表示 BizModel 运行时 adapter 载荷，不代表 `*-api` typed client 契约退回 `Map`。
- API 辅助迭代器瘦身：`CrudApiPageIterator` 与 `CrudApiItemIterator` 去除无行为意义的输入泛型 `I`，改为只保留输出泛型 `O`，内部持有 `ICrudApi<?, O>`；查询迭代逻辑只依赖 `findPage` 输出分页结果，不需要透传实体输入类型。
- CRUD typed API 单层契约开始实施：`crud-api` 模板已改为直接在 `beans/` 与 `crud/` 生成公共 `*Bean` / `*Api`，并统一让关系字段与接口泛型引用非下划线公共类型；旧 `_gen/_*Bean` 与 `_{Entity}Api` 模板改为空输出，准备通过再生成清理旧双层产物。
- 迁移策略确认：`XCodeGenerator` 只负责覆盖同名目标文件，不会自动删除“模板已不再产出”的旧保留层文件；因此在验证模块 `nop-auth-api` 中额外清理了一次旧公共 wrapper 文件，再通过 `nop-auth-meta/postcompile` 重新生成单层契约。
- 验证结果：`nop-auth-api` 已不再存在 `_gen` Bean 或 `_{Entity}Api`；`NopAuthUserInputBean.relatedGroupList` 等关系字段已改为 `List<NopAuthGroupInputBean>`，`NopAuthGroupApi` 也直接 `extends ICrudApi<NopAuthGroupInputBean, NopAuthGroupOutputBean>`。
- Doc-sync: `ai-dev/design/crud/crud-api-codegen-design.md` 已按 design guide 收敛。

### 2026-06-04（no-api 规则）

- CRUD API 发布控制收敛：新增实体级标签 `no-api`，用于跳过 CRUD API 的 Bean / 接口生成；明确 `not-pub` / `published=false` 表示字段在 API / GraphQL 视角下不存在，`no-web` 仅控制页面生成，不再复用它们表达 API 发布语义。
- 实施：更新 `ai-dev/design/crud/crud-api-codegen-design.md` 与 `nop-kernel/nop-codegen/.../crud-api/@init.xrun`，让 `crud-api` 生成链只处理具备 `entityName` 且未带 `no-api` 标签的实体 xmeta。
- 边界裁定：`no-api` 只影响 typed client codegen，不影响 BizModel 注册，也不改变 GraphQL / RPC 的服务端发布；若后续需要控制服务端是否发布，单独引入服务端发布标签。
- Doc-sync: `docs-for-ai/02-core-guides/api-model-and-codegen.md` 已同步 `no-api` 规则。
