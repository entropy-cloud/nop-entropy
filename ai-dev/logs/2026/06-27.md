### ApiResponse / ApiRequest 重写 toString() 暴露核心字段

- 类型：nop-api-core bug 修复（调试可见性），无 Plan（ad-hoc）
- 触发：nop-app-mall 的 mission-driver CHECK 步骤卡死 23 分钟。根因之一是 GraphQL 把异常包装成失败的 `ApiResponse`，而 `ApiMessage` 继承链未重写 `toString()`，测试断言 `"..." + response` 只打印 `ApiResponse@hash`，真正的 `msg/code/errors` 对调试者完全不可见，导致 build agent 在「提取错误」上反复编译跑测试。
- 改动（3 文件，`nop-kernel/nop-api-core/src/main/java/io/nop/api/core/beans/`）：
  - `ApiMessage`：新增 `protected appendHeaders(StringBuilder)`（仅在 `hasHeaders()` 时追加 `,headers={...}`）和 `protected static truncate(Object, int)`（超长字段截断到 200 字符并标 `...(原长度)`），供两个子类复用，避免 header 拼接逻辑散落。
  - `ApiResponse.toString()`：输出 `ApiResponse[status=,httpStatus=?,code=,msg=,errors=,data=(截断),headers=]`，空字段跳过。`data` 截断防日志爆炸，注释提示「需完整内容用 `JsonTool.stringify`」。
  - `ApiRequest.toString()`：输出 `ApiRequest[data=(截断),selection=,properties=,headers=]`。
  - 格式遵循现有 `ErrorBean.toString()` 的 `StringBuilder` + `BeanName[field=value]` 约定。
- 验证：
  - `mvn test -pl nop-kernel/nop-api-core` → `Tests run: 38, Failures: 0`，BUILD SUCCESS
  - 已 `mvn install` 到 `~/.m2`（jar 时间戳 17:11）
  - nop-app-mall 全量 `mvn test` → `Tests run: 123, Failures: 0, Errors: 0`，BUILD SUCCESS（无回归）
  - 冒烟验证输出格式：失败响应现在打印 `ApiResponse[status=-1,code=nop.err.orm.mandatory-prop-is-null,msg=实体对象...picUrl...为null]`；大 data 截断为 `data=...(500)`；headers 正确包含 `authorization`/`X-Trace-Id`
- 风险：改变全局所有调用点（日志/调试/序列化兜底）的 toString 行为。已确认 nop-api-core 与 nop-app-mall 均无测试断言依赖旧 `ApiResponse@hash`/`ApiRequest@hash` 输出。
- 影响：今后 `ApiResponse`/`ApiRequest` 在断言失败、日志、调试器中可直接看到核心字段，避免同类「异常被吞进 hashCode」事故。
- Doc-sync: No owner-doc update required（toString 实现细节，非对外契约）

### nop-wf 审批流设计文档

- 类型：ai-dev/design 文档（无代码变更）
- 触发：用户要求分析 nop-wf 设计，判断复杂审批流程如何实现，补充设计文档
- 调研结果：
  - nop-wf 引擎核心在 `WorkflowEngineImpl`，支持 step/join/flow 三种步骤类型
  - 执行分组（execGroupType）提供 or-group/and-group/seq-group/vote-group 四种多人审批模式
  - wf-gen:CosignSupport 通过 `x:post-extends` 代码生成方式实现会签（co-sign）
  - 条件路由通过 `splitType="or|and"` + `<when>` 条件实现
  - 驳回（forReject/doReject）支持驳回到上一步或指定祖先步骤
- **发现 4 个已知问题：**
  - P0: `ExecGroupSupport` 中 `shouldExecGroupComplete/shouldExecGroupReject/skipExecGroupMembers` 均未被引擎调用 → 会签/并签/串签的组完成判定未生效
  - P1: `WorkflowStepImpl.getStepsInSameExecGroup` 过滤逻辑反向（移除同组而非保留同组）
  - P2: `isVoteGroupComplete/isVoteGroupReject` 百分比比较使用了 `passWeight` 而非 `passPercent`
  - P3: `oa.xlib.ExitCosignStep` 调用不存在的 `getStepsWithSameStepGroup` 方法
  - P4: `TestCountersign` 被 `@Disabled`
- 产物：`ai-dev/design/nop-wf/approval-flow-design.md`
- Doc-sync: `docs-for-ai/03-modules/nop-wf.md` 和 `docs-for-ai/INDEX.md` 已更新，添加对新设计文档的引用
