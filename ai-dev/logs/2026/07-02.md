## Entry 9 — Lattice vs AGE 对比分析

**分析报告**: `ai-dev/analysis/2026-07-02-lattice-vs-age-comparison.md`

完成 Lattice（`~/ai/lattice`，产品化 AI Coding Control Plane）与 AGE Template（`~/app/attractor-guided-engineering-template`，方法论骨架）的全维度对比。

核心发现：
- Lattice 是**可安装的产品**（install.sh），聚焦单次 AI Coding 任务的规约-验证-留痕；AGE 是**方法论骨架**（复制即用），聚焦项目长期收敛方向
- Lattice 用 pipeline + gates + eval JSON 做可复现验证；AGE 用独立 subagent 回到活仓库做 closure audit
- 两者在知识提取上有个共同点（都强调从实践中反向提取），但机制不同（Lattice 电子流 draft→review→promote；AGE 晋升阶梯 bug→lesson→prompt→script→lint）
- nop-entropy 继承的是 AGE 方法论，不是 Lattice——沿袭了文档分层、独立 closure audit、晋升阶梯等实践

## Entry 8 — ORM 列域设计修正

修正 `docs-for-ai/02-core-guides/orm-model-design.md`「列域设计」章节的内部矛盾与概念混淆。起因：在 nop-app-erp 执行最佳实践合规审计时，发现该章节对"人"相关列的指导自相矛盾，导致应用层 18 域出现三种不一致建模（裸 VARCHAR / stdDomain=userId / 员工 to-one）。根因是文档未按**语义角色**区分列类型，把"业务操作人"误并入"被动审计"或"员工引用"。

### Changes

#### `docs-for-ai/02-core-guides/orm-model-design.md` — 列域设计章节重构

重构 `## 列域设计：stdDomain 与 domain` 下的子节。

修正的三处缺陷：

1. **内部矛盾**：旧版 `operatorId` 同时出现在两处——「用户引用列」作 `stdDomain="userId"`（VARCHAR(36)），「员工引用列」作 BIGINT + to-one → ErpMdEmployee。同一字段名给出相反建议。现已统一为 userId。

2. **概念混淆**：旧版把 `postedBy`/`approvedBy`/`closedBy`（业务动作责任字段）与 `createdBy`/`updatedBy`（被动审计）一并归入 `domain="createdBy"` VARCHAR(50) 登录名。现按语义角色拆分：业务动作责任 → `stdDomain="userId"`；仅框架自动填充的 createdBy/updatedBy → `domain="createdBy"`。

3. **操作人误建员工引用**：旧版推荐 `pickerId`/`handlerId`/`operatorId`（业务操作人）建员工 to-one。现明确：凡"谁执行/负责/拥有此操作"一律 `stdDomain="userId"`；员工 to-one 仅留给"员工记录本身是业务数据"的 HR 主体实体。
