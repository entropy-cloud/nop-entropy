核心发现：

## Entry 1 — ORM 列域设计修正

修正 `docs-for-ai/02-core-guides/orm-model-design.md`「列域设计」章节的内部矛盾与概念混淆。起因：在 nop-app-erp 执行最佳实践合规审计时，发现该章节对"人"相关列的指导自相矛盾，导致应用层 18 域出现三种不一致建模（裸 VARCHAR / stdDomain=userId / 员工 to-one）。根因是文档未按**语义角色**区分列类型，把"业务操作人"误并入"被动审计"或"员工引用"。

## Changes

### `docs-for-ai/02-core-guides/orm-model-design.md` — 列域设计章节重构

重构 `## 列域设计：stdDomain 与 domain` 下的子节（原「用户引用列 / 审计字段 / 员工引用列 / 总结对照」→ 新「判别原则 / 操作人责任人归属列 / 被动审计列 / 员工引用列仅 HR 主体 / 总结对照」）。

修正的三处缺陷：

1. **内部矛盾**：旧版 `operatorId` 同时出现在两处——「用户引用列」作 `stdDomain="userId"`（VARCHAR(36)），「员工引用列」作 BIGINT + to-one → ErpMdEmployee。同一字段名给出相反建议。现已统一为 userId。

2. **概念混淆**：旧版把 `postedBy`/`approvedBy`/`closedBy`（业务动作责任字段）与 `createdBy`/`updatedBy`（被动审计）一并归入 `domain="createdBy"` VARCHAR(50) 登录名。但前者需按审核人过滤/统计/显示，`stdDomain="userId"` 才正确。总结对照表还出现 `approverId`→userId 而 `approvedBy`→createdBy 的同概念双建模。现按语义角色拆分：业务动作责任 → `stdDomain="userId"`；仅框架自动填充的 createdBy/updatedBy → `domain="createdBy"`。

3. **操作人误建员工引用**：旧版推荐 `pickerId`/`handlerId`/`operatorId`（业务操作人）建员工 to-one。这与本节自身「用户引用列」已把 operatorId/assignedToId/teamLeaderId 列为 userId 的选择冲突。现明确：凡"谁执行/负责/拥有此操作"一律 `stdDomain="userId"`；员工 to-one 仅留给"员工记录本身是业务数据"的 HR 主体实体（合同/薪资/考勤/绩效/部门归属）。

新增「判别原则」表（操作人·责任人·归属 / 被动审计 / HR 主体实体 三类）与判别问题。员工引用列补"前置分派边界场景"说明：若确需分派给未开账号的 HR 人员，该分派字段可建员工引用，但 ErpMdEmployee 必须携带 userId 链接，且"谁实际执行"仍为 userId。tip 补"最易踩的坑"提示。

## Why

判别原则：**"该列记录的是认证主体的动作/归属，还是引用一个独立于登录的 HR 资源？"** 前者一律 `stdDomain="userId"`（写入捕获 `IUserContext.getUserId()`，平台自动解析展示 + 可过滤）；后者才用员工 to-one。操作人建员工引用的三大问题：(1) 需写入时 user→employee 解析，对系统账号/外部审计员/无档案管理员失败；(2) 耦合员工实体生命周期，污染历史事实；(3) `stdDomain="userId"` 已免费提供展示解析。

`nop_auth_user.userId` 是 String UUID，引用用户的 FK 列必须 VARCHAR(36)；员工（ErpMdEmployee）PK 是 BIGINT——两者类型不同，混淆会导致生成代码类型不匹配编译失败。

## Impact

本文件被 `docs-for-ai/INDEX.md` 列为「模型设计」核心规范，影响所有 Nop 应用项目的"人"相关列建模。修正后与该章节自身「用户引用列」的选择自洽，消除 AI 生成时的歧义分支。
