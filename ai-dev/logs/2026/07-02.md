### 分析报告：FlowLong / Warm-Flow vs nop-wf 工作流引擎深度对比

- **新增文件**：`ai-dev/analysis/2026-07-02-flowlong-warmflow-vs-nop-wf-comparison.md`
- **动机**：核查 nop-wf 是否能作为 FlowLong（com.aizuda）和 Warm-Flow（org.dromara）的上位替代
- **方法**：并行启动 3 个 explore agent 深度分析三个项目源码（FlowLong `c:/can/sources/flowlong`、Warm-Flow `c:/can/sources/warm-flow`、nop-wf），所有结论带 `file:line` 证据
- **覆盖度结论**：nop-wf 在引擎核心能力（节点/路由/会签/驳回/转办/子流程/审批人解析/监听器）上 **基本覆盖** 两个开源项目，且在 XDSL 可逆计算、DAG 校验、trampoline 执行队列、信号机制、业务实体绑定、错误重试、全局事件订阅、动态汇聚分组等方面 **明显领先**
- **识别 5 大缺口**（按优先级）：
  - **P0**：AI 审批集成（FlowLong 独有，含 AiConfig/AiResponse/置信度阈值/降级策略/异步回调/AI 路由）
  - **P0**：内置调度器（FlowLong 独有，cron 扫描 + 分布式锁 + 超时自动通过/拒绝 + 提醒次数限制）
  - **P1**：离职转办（批量改办理人）、票签表达式策略可插拔（Warm-Flow 的 default@@/spel@@/passCount=N 前缀路由）、节点模型动态加减签
  - **P2**：穿越时空（补审）、流程图状态快照、内置表单
  - **P3**：审批人=提交人 4 种内置策略、作废/暂存显式状态枚举
- **否决方案**：多 ORM 解耦（破坏 Nop 集成度）、AI 绑定具体大模型（应保留 SPI）
- **状态**：analysis 为 open，结论待评审；后续若确认补齐 P0/P1，指向 `ai-dev/design/wf-ai-integration/design.md` 和调度器实施计划
- **验证**：`node ai-dev/tools/check-doc-links.mjs --strict` 无新增错误（26 个错误均为仓库预存的 plans 引用问题）

### nop-wf 扩展设计审计修正（3 轮独立子 agent 审计）

- **审计方式**：启动 3 个独立子 agent（全新上下文，互不继承），分别核查：① 设计文档的事实声明（扩展点是否存在/可用）② 待实施功能的可行性 ③ 被拒方案合理性 + 文档自洽性
- **核心发现（3 个 agent 一致确认）**：设计初稿**基于对源码的错误理解**——把两个已声明但未接线的"死槽位"当成可用扩展点：
