**Plan**: `ai-dev/plans/2026-07-18-1100-1-nop-metadata-expression-measure-design.md`

### 落地内容

- **D12 写入 §4.4.2**（`01-architecture-baseline.md`，D11 之后）：
    - **D12.1 表达式语言选型**：选定 **方言原生 SQL 片段**（dialect-native SQL fragment，同一份 expression 文本在三条路径上语义一致）。拒绝 EQL（函数白名单 `FORMATDATETIME`/`STDDEV_SAMP` 等被拒，§4.4.2 D7 + §4.4.3 D1 已实测）/ 平台表达式引擎（纯内存、违反 D6 pushdown + D10 规模守卫、与三路径模型冲突）/ 混合方案（per-path 语义分叉、contract drift），各附理由。
    - **D12.2 三条路径执行契约**：
        - **entity 路径**：bypass EQL，经平台物理 JDBC Connection 直查（既有先例 §4.4.3 D1 `TableReferenceExecutor.java:73-83`：`orm.getSessionFactory().txn()` + `runInTransaction(SUPPORTS)` + `IJdbcTransaction.getConnection()`）。是否保留 `orm().executeQuery` EQL 快路径（仅 EQL 兼容子集）作为 successor 技术调研项。
        - **external-sql 路径**：复用 `withConnection` 原生 SQL + 标识符白名单 §2.7.1 D3 + 值参数绑定（FilterToSqlTranslator 模式）。
        - **跨库内存路径**：首版显式失败（`metadata.aggr-expression-memory-not-computable`），对齐 D10 既有「不在上列 aggFunc（含 expression）→ 显式失败」铁律；expression 内存可算子集白名单为 successor 评估项。
    - **D12.3 安全模型**：注入面 = 用户 `expression` 文本；三道防御闸门——parse 阶段关键字/函数黑名单（拒绝 INSERT/UPDATE/DELETE/DROP/CREATE/ALTER/TRUNCATE/CALL/EXEC/GRANT/REVOKE/MERGE 等 DDL/DML + SLEEP/BENCHMARK/LOAD_FILE/INTO OUTFILE/PG_SLEEP/COPY 等副作用函数）+ 标识符白名单（复用 §2.7.1 D3 `^[A-Za-z_][A-Za-z0-9_]*$`，列引用取自 `MetaTableFieldResolver` 解析集合）+ 值参数绑定（对齐 FilterToSqlTranslator，禁止裸字符串拼接）。失败一律显式（不静默 fallback/截断/sanitize）。
    - **D12.4 失败路径 ErrorCode 体系**（4 类 + 容量超限，successor 起点）：`metadata.aggr-expression-unparseable`（不可解析）/ `-unsafe`（不安全）/ `-dialect-unsupported`（方言不支持）/ `-memory-not-computable`（跨库内存不可算，对齐 D10）/ `-too-long`（超 VARCHAR(1000) 容量上限，对齐 §2.5.2 D1 json-4000 铁律：不截断、不静默存入截断后的脏数据）。5 处 `ERR_AGGR_EXPRESSION_MEASURE` 抛点（`MetaAggregationExecutor.java:1129` / `:1756` / `:1817` / `:2355` / `:2371`）successor 落地后按 D12.2 替换为真实执行逻辑；落地前维持不变。

## nop-metadata queryAggregation having/orderBy 增强（Plan 2026-07-18-0900-2）

**Plan**: `ai-dev/plans/2026-07-18-0900-2-nop-metadata-aggregation-having-order-by.md`

把 `queryAggregation` 从仅支持 `filter/limit/offset` 扩展到支持 `having`（聚合后过滤，如 `SUM(amount) > 1000`）+ `orderBy`（按 measure/dimension 排序），三条执行路径（entity 原生 SQL / external-sql 原生 SQL / 跨库内存 GROUP BY）一致支持。

### 落地内容

- **签名扩展**：`NopMetaTableBizModel.queryAggregation` 从 8-arg 扩展到 10-arg——新增 `@Optional @Name("having") TreeBean having` + `@Optional @Name("orderBy") List<OrderFieldBean> orderBy`，固定在 `offset` 之后（向后兼容：缺席时维持既有行为零变化）。
- **FilterToSqlTranslator 扩展（R1 B2 修复）**：新增 `translate(TreeBean filter, Function<String,String> fieldResolver)` 重载。`fieldResolver` 非空时叶子条件 name 经回调解析为 SQL 表达式（如 `SUM(AMOUNT)`），**跳过 `validateIdentifier`**（因 aggSql 含括号会触发白名单失败）；该表达式中的列名已在 measure/dimension 加载时经白名单校验。`fieldResolver` 为空时维持既有 `requireField + validateIdentifier` 行为。既有 `translate(TreeBean)` 行为不变（委托 `translate(filter, null)`）。新增 `ERR_FILTER_FIELD_RESOLVER_MISS` 防御性兜底 ErrorCode（实际未选定 name 由更具体的 `ERR_AGGR_HAVING_UNKNOWN_NAME` 在反查表阶段失败）。
- **name→SQL 表达式反查表（R1 M3 修复）**：新增 `buildNameToExprTable`（单表）+ `buildJoinNameToExprTable`（JOIN）+ `buildCrossDbNameToAliasTable`（跨库内存）3 个 helper——从 `loadMeasures`/`loadDimensions` 返回的原始实体列表（仍持 measureName/dimensionName，**非**从已丢 name 的 MeasureSpec）构建 `Map<String, String>`。未知 name → `ERR_AGGR_HAVING_UNKNOWN_NAME` / `ERR_AGGR_ORDER_BY_UNKNOWN_NAME` 显式失败（不静默跳过）。
- **三条路径一致支持**：
  - **entity 单表原生 SQL**（`executeEntityAggregation`）：GROUP BY 后追加 HAVING + ORDER BY 子句。参数绑定顺序 filter→having→limit/offset。
  - **external/sql 单表原生 SQL**（`buildExternalAggregationSql`）：同上追加。
  - **JOIN 同库 3 条**（`executeEntityEntityJoinAggregation` 内联 + `buildExternalExternalJoinSql` + `buildMixedSameDbJoinSql`）：HAVING/ORDER BY 子句追加，name → `JoinMeasureSpec.qualifiedAggCol`（含 l./r. 前缀）/ `JoinDimensionSpec.qualifiedCol`。
  - **跨库内存 GROUP BY**（`executeCrossDbJoinAggregation`）：内存 GROUP BY 产出 group → `MemoryFilterEvaluator.filter` 过滤 → `MemoryOrderByComparator.sort` 排序 → `truncateCrossDb` 截断（D3 顺序：orderBy → limit/offset）。`MetaJoinExecutor.executeJoin` 签名不变（R1 m4：executeJoin 不感知 having/orderBy）。
- **新增组件**：
  - `MemoryFilterEvaluator.java`：内存 TreeBean 求值器，递归 and/or/not，op 集合 eq/ne/gt/ge/lt/le/like/in/between/is-null/not-null，Number→BigDecimal 统一类型强转，name case-insensitive 匹配大写化 alias。
  - `MemoryOrderByComparator.java`：内存多键比较器，按 `List<OrderFieldBean>` 逐字段排序（desc/nullsFirst 生效），类型强转同 MemoryFilterEvaluator。
- **新增 ErrorCodes**：`ERR_AGGR_HAVING_UNKNOWN_NAME` / `ERR_AGGR_ORDER_BY_UNKNOWN_NAME` / `ERR_AGGR_HAVING_UNSUPPORTED_OP` / `ERR_FILTER_FIELD_RESOLVER_MISS`（防御性兜底）。

### 测试覆盖（Anti-Hollow）

- **新增 17 用例独立单元测试** `TestMemoryFilterAndOrderBy.java`：覆盖 op 集合（eq/ne/gt/ge/lt/le/like/in/between/is-null/not-null + and/or/not 嵌套）+ 类型强转（Long vs Integer vs BigDecimal vs Double）+ case-insensitive alias 匹配 + null 策略 + 多键排序 + desc/nullsFirst 生效 + 未知 name 显式失败。

## nop-metadata sql 视图字段类型推断方案 B 落地（Plan 2026-07-18-0900-1）

**Plan**: `ai-dev/plans/2026-07-18-0900-1-nop-metadata-sql-view-field-type-inference.md`

把 `tableType=sql` 逻辑表的 sourceSql 字段类型从方案 A（恒 null）推进到方案 B（`LIMIT 0` + `ResultSetMetaData.getColumnTypeName` 推断真实类型），当 `querySpace` 可达数据源时。

### 落地内容

- **新增** `nop-metadata-service/.../sqlview/SqlViewFieldTypeInferrer.java`：独立类型推断组件。经 `MetaDataSourceResolver.resolveActiveOrThrow` + `IMetaDataSourceConnectionService.withConnection` 跑 `SELECT * FROM (<sourceSql>) _t LIMIT 0` + `ResultSetMetaData.getColumnTypeName` 取列类型，按 D3 列序对齐（index↔1-based）补全 `SqlViewField.type`。方言守卫 H2/MySQL/PostgreSQL，其他显式失败。失败路径全部显式抛 ErrorCode（不静默 fallback type=null、不吞异常，R1 B5 修复）。
- **集成** `NopMetaTableBizModel`：
  - 新增字段 `sqlFieldTypeInferrer`（延迟初始化——`connectionService` 经 `@Inject` 在构造时不可用，仿 `tableRefExecutor` 用 `ensureSqlFieldTypeInferrer()` 模式）。
  - `createSqlTable`：`extract()` 后、`toFieldMaps()` 前，若 querySpace 参数非空 → 调 inferrer 补 type（R2 N2 修复，无"或"歧义）。
  - `resolveTableFields`：`fieldResolver.resolve()` 后、若 `tableType=sql && table.querySpace` 非空 → 调 inferrer 补 type 并重构造 `ResolvedTableField`（R2 N1 修复，**`MetaTableFieldResolver` 不改**，10+ 仅消费 name 的调用方零影响）。
- **测试** 扩展 `TestNopMetaTableQueryBizModel`（+7 用例，共 19）：createSqlTable 推断真实类型（INT→INTEGER/BIGINT→BIGINT/VARCHAR→CHARACTER VARYING）、无 querySpace→type=null（向后兼容）、resolveTableFields 端到端推断、列对齐（别名 + `<expr_1>`）、3 条失败路径（querySpace 无匹配 / DISABLED / sourceSql 语法错误）。方言守卫的 H2 已支持路径已端到端验证；unsupported 方言需 mock DatabaseMetaData，超出端到端测试范围。
- **既有用例调整**：`testQuerySqlTableNoMatchingDataSource` 由 createSqlTable 设置改为 saveManualSqlTable（plan 0900-1 后 createSqlTable 在 querySpace 提供时本身会失败——这是新的正确行为，本测试专测 queryTableData 失败路径，故绕过 createSqlTable 设置以保留原覆盖意图）。新增 helper `saveManualSqlTable`。

### 偏差

- 实测 H2 经 `getColumnTypeName` 返回 **SQL 标准名**（`CHARACTER VARYING` / `DOUBLE PRECISION`）而非简写（`VARCHAR` / `DOUBLE`）。这与 D2 裁定一致："driver 原生返回名，不归一化"——测试断言按 driver 实际返回值写。
