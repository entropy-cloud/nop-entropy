## nop-metadata entity 路径时间维度 granularity 分桶补齐（Plan 2026-07-18-1100-2，Phase 1）

**Plan**: `ai-dev/plans/2026-07-18-1100-2-nop-metadata-entity-path-temporal-granularity.md`

把 entity 路径时间维度 granularity 分桶从「首版按物理列直查、granularity 暂不下沉到 SQL」（§4.4.2 D7 follow-up）推进到「entity 路径与 external/sql 路径一致支持约定 granularity 分桶」。Phase 1 = 机制裁定（design only）。

### Phase 1 落地内容（机制裁定）

- **live 核实 EQL 函数清单**：审计 `default.dialect.xml` + `h2.dialect.xml` + `postgresql.dialect.xml` + `mysql.dialect.xml` 的 EQL 函数注册表——EQL 已知日期函数仅 `year(col)` 返回 INTEGER、`date(col)` 截断到日返回 DATE、`extract(<field> FROM <col>)` 返回 INTEGER、`current_date/current_timestamp/now`；**缺失** `DATE_TRUNC`、`FORMATDATETIME`、`DATE_FORMAT`、`quarter`、`month`、`week`、`hour`。证实候选 b（EQL 已知函数）覆盖度不足且 year 桶类型与 external 路径（返回 DATE）不一致。
- **裁定**：选定候选 a — entity 路径 granularity 下沉**bypass EQL**，经平台物理 JDBC Connection 直查原生 SQL，与 external/sql 路径产出**同一份分桶 SQL 文本**（复用 `GranularityBucketing.translate`，dialect 分发 H2/MySQL/PostgreSQL）。
- **接线裁定**：复用 `TableReferenceExecutor.execute(TableReference, ConnectionAction)`（§4.4.3 D1，`executeOnPlatformConnection` 既有 entity 分派路径，`orm.getSessionFactory().txn()` + `runInTransaction(SUPPORTS)` + `IJdbcTransaction.getConnection()`）；`MetaQueryContext` 暴露 `tableRefExecutor()`，由 `NopMetaTableBizModel.ensureTableRefExecutor()` 装配；entity 聚合路径检测到任一 temporal dimension 有非空 granularity 时进入 bypass 分支。
- **拒绝方案及理由**：候选 b（EQL 已知函数，覆盖度不足 + 类型分叉）；候选 c（先取数再内存分桶，违反 D6 pushdown 铁律 + 规模风险）；跨库内存路径 dimension 内存分桶（已记入 Deferred，跨库 JOIN 走 D10 内存 GROUP BY，dimension 无 SQL 表达式层，时间桶内存截断需另行设计）。
- **与 §4.4.1 D1.5 拒绝候选 B 的区别澄清**：D1.5 拒绝 `IJdbcTransaction.getConnection` 仅适用于**混合端点**（external 物理表几乎不可能在平台库可见）；本节为**单表 entity 聚合**，entity 物理表就在平台库，候选 a 可达性天然成立，D1.5 拒绝理由不适用。本机制与 §4.4.3 D1（`TableReferenceExecutor.executeOnPlatformConnection`）+ §4.4.2 D12.2（expression 型 Measure entity bypass EQL）共享同一 Connection 入口与同一先例。
- **D6 ORM 隐式过滤旁路语义不变**：bypass 物理 Connection 仍为物理表直查、绕过 ORM 隐式过滤，D6 既有「首版不限制 + 显式文档提示」语义保留。
- **失败路径沿用既有 ErrorCode**：`metadata.aggr-granularity-not-supported`（GranularityBucketing 既有抛点）/ `metadata.aggr-unsupported-dialect`（既有 SUPPORTED_DIALECTS 检查在 bypass 路径同等生效）/ `metadata.tableref-entity-query-space-not-jdbc`（TableReferenceExecutor 既有抛点）。无静默 fallback 裸列。

## nop-metadata queryAggregation having/orderBy 增强（Plan 2026-07-18-0900-2）

**Plan**: `ai-dev/plans/2026-07-18-0900-2-nop-metadata-aggregation-having-order-by.md`

把 `queryAggregation` 从仅支持 `filter/limit/offset` 扩展到支持 `having`（聚合后过滤，如 `SUM(amount) > 1000`）+ `orderBy`（按 measure/dimension 排序），三条执行路径（entity 原生 SQL / external-sql 原生 SQL / 跨库内存 GROUP BY）一致支持。

### 落地内容

- **签名扩展**：`NopMetaTableBizModel.queryAggregation` 从 8-arg 扩展到 10-arg——新增 `@Optional @Name("having") TreeBean having` + `@Optional @Name("orderBy") List<OrderFieldBean> orderBy`，固定在 `offset` 之后（向后兼容：缺席时维持既有行为零变化）。
- **FilterToSqlTranslator 扩展（R1 B2 修复）**：新增 `translate(TreeBean filter, Function<String,String> fieldResolver)` 重载。`fieldResolver` 非空时叶子条件 name 经回调解析为 SQL 表达式（如 `SUM(AMOUNT)`），**跳过 `validateIdentifier`**（因 aggSql 含括号会触发白名单失败）；该表达式中的列名已在 measure/dimension 加载时经白名单校验。`fieldResolver` 为空时维持既有 `requireField + validateIdentifier` 行为。既有 `translate(TreeBean)` 行为不变（委托 `translate(filter, null)`）。新增 `ERR_FILTER_FIELD_RESOLVER_MISS` 防御性兜底 ErrorCode（实际未选定 name 由更具体的 `ERR_AGGR_HAVING_UNKNOWN_NAME` 在反查表阶段失败）。

  - **entity 单表原生 SQL**（`executeEntityAggregation`）：GROUP BY 后追加 HAVING + ORDER BY 子句。参数绑定顺序 filter→having→limit/offset。

  - `MemoryFilterEvaluator.java`：内存 TreeBean 求值器，递归 and/or/not，op 集合 eq/ne/gt/ge/lt/le/like/in/between/is-null/not-null，Number→BigDecimal 统一类型强转，name case-insensitive 匹配大写化 alias。
