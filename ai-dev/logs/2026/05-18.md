## nop-job Plan 19: Test Coverage & Fault Tolerance Hardening

### What was done

Created `ai-dev/plans/19-nop-job-test-coverage-and-fault-tolerance-hardening.md` — a comprehensive test plan to verify nop-job's fault tolerance capabilities added in Plan 18.

### Plan structure

7 phases covering:
1. Dynamic partition unit tests (JobPartitionResolver)
2. Retry bridge & alarm handler call path tests
3. Timeout detection & progress reporting tests
4. Block strategy tests (DISCARD/OVERLAY/RECOVERY/SKIP)
5. E2E lifecycle tests (happy path + failure + timeout + recovery)
6. Fault tolerance integration tests (crash recovery + rebalance + worker failure + eventual consistency + idempotency + stabilization window)
7. Bug fixes from testing & regression verification

### Review process

Used team mode with 3 independent review agents:
- **arch-reviewer**: Verified gap coverage alignment with analysis docs, confirmed phase dependencies
- **test-reviewer**: Evaluated test infrastructure compatibility, identified edge cases
- **fault-tolerance-reviewer**: Compared against PowerJob/Snail-Job fault tolerance patterns, identified 3 critical missing scenarios

### Key review findings (Round 1)

All 3 agents confirmed the plan structure is sound. Fault-tolerance-reviewer identified 3 critical gaps:
1. **Missing eventual consistency test**: After coordinator crash, verify ALL schedules eventually processed
2. **Missing idempotency test**: Retry bridge not called twice for same fire; task not executed twice
3. **Missing stabilization window jitter test**: Instance list rapidly changing during rebalance

These were incorporated as [Round 2] items into the plan.

### G1-G4 implementation audit (post-review code check)

After the review, conducted a code audit of G1-G4 actual implementation status:

| Gap | Status | Finding |
|-----|--------|---------|
| G1 Worker heartbeat | **NOT implemented** | Only `TASK_STATUS_SUSPICIOUS` constant exists. Worker has no INamingService dependency, Coordinator has no Worker liveness check |
| G2 Retry integration | **Partial** | `IJobRetryBridge` + `NoOpJobRetryBridge` exist, CompletionProcessor calls them. But `nop-job-retry-adapter` module does NOT exist, no real `IRetryEngine` integration |
| G3 Dynamic partition | **DONE** | `JobPartitionResolver` fully implemented, all 4 Scanners use it |
| G4 Dispatch timeout | **NOT implemented** | TimeoutChecker only checks RUNNING tasks, no DISPATCHING timeout check |

Plan 19 was updated to include Phase 1A/1B/1C for the missing implementations, before the test phases.

## Juggle vs Nop-Task + Nop-Job 对比分析

### What was done

Created `ai-dev/analysis/2026-05-18-juggle-vs-nop-task-comparison.md` — a deep comparative analysis of Juggle (v1.6.0) API orchestration engine against Nop's nop-task (task orchestration) and nop-job (job scheduling) engines.

### Key findings

1. Juggle excels at **API orchestration** — visual designer, multi-protocol (HTTP/Dubbo/WS), suite marketplace, low learning curve
2. Nop-Task excels at **general task orchestration** — continuation model, 20+ step types, decorator system, interrupt/resume
3. Nop-Job provides **distributed scheduling** — partition, heartbeat, timeout, block strategies
4. Juggle's core engine is **closed-source** (juggle-core JAR) — significant technical risk
5. Nop-Task + Nop-Job has deeper architecture but steeper learning curve
6. nop-job is mid-rewrite — old code still present, new coordinator/worker modules functional

### 修正的事实错误

| 错误 | 旧值 | 新值 |
|------|------|------|
| ElementTypeEnum 类型数 | 9（含 PARALLEL, EVENT） | 7（无 PARALLEL, 无 EVENT） |

| 表达式引擎 | "基于 Aviator" | 自定义递归下降解析器 + Aviator 作为补充 |
| WorkRunnerImpl | "直接用 Thread 实现" | ThreadPoolExecutor + LinkedBlockingQueue + dispatch thread |

### 关键新发现

1. **`FlowExecutor` finally 无条件 return** — finally 块的 return 覆盖 try/catch 返回值，状态永远为 FINISH
2. **`printStackTrace()` 出现 3 次** — FlowExecutor + AbstractDispatcher（JuggleCoreException + JsonProcessingException）
3. **硬编码中文日志** — EndNodeExecutor/ConditionNodeExecutor/CodeNodeExecutor 均含中文错误消息
4. **`EventNode` 模型存在但空且无 executor** — 未完成的功能占位
5. **`WorkRunnerImpl` 架构更复杂** — extends Thread 是调度线程，内部有 ThreadPoolExecutor
6. **Validator 层完整** — 7 个类，与 executor 平行的模板方法设计
7. **代码混淆极激进** — 所有类名/方法名/字段名均为单字母，运行时堆栈完全不可读
8. **Java 8 编译** — 依赖 Nashorn（JDK 15 废弃，JDK 21 移除）
9. **条件表达式引擎是自定义 DSL** — DNF 模型，10 个解析器类
10. **MemoryVariableManager 使用 HashMap** — 非线程安全，多线程竞态

## IMessageService 生态补充

用户指出 Nop 平台已有 `IMessageService` 可用于 MQ 发送和监听。

### 分析发现

`IMessageService` 位于 `nop-kernel/nop-api-core`，聚合 `IMessageSender` 和 `IMessageSubscriber` 两个子接口：

| 接口 | 核心方法 | 功能 |
|------|---------|------|
| IMessageSender | `sendAsync(topic, msg, opts)` → `CompletionStage<Void>` | 异步消息发送 |
| IMessageSubscriber | `subscribe(topic, listener, opts)` → `IMessageSubscription` | Topic 订阅 |
| IMessageConsumer | `onMessage(topic, msg, ctx)` → `Object` | 消费回调 (null=ack, CS=async, ConsumeLater=delay) |
| IMessageSubscription | cancel/suspend/resume | 订阅生命周期管理 |

**6 种实现**：Local (内存线程安全)、Pulsar、SysDao (DB)、MQTT、Lettuce (Redis)、Debezium (CDC)
