### 修复 TestJobConcurrency flaky test（全量并行 -T 1C 偶发失败）

- 现象：全量 `./mvnw test -T 1C` 时 `nop-job-coordinator` 的 `TestJobConcurrency.testFullPipelineTwoCoordinators` 偶发失败（`expected:<3> but was:<4>`），单独运行稳定通过 → flaky。该模块本次无产品代码改动（预先存在的 flaky）。
- 根因（trigger 时序边界，非乐观锁问题）：`TestJobConcurrency.newSchedule` 未设 `lastFireTime` → `TriggerSpecHelper.toEvalContext` 的 `getLastScheduledTime()` 返回 0 → `PeriodicTrigger.nextScheduleTime`（fixed-rate 分支，`PeriodicTrigger.java:66`）以 epoch(0) 为起点、按 `period=1000ms` 对齐到整秒点，返回 `>= now` 的下一个整秒。plannerA 顺序处理 3 个 schedule 调用 `calculateNextFireTime`（afterTime=`scheduleStore.getCurrentTime()` 实时钟），若处理跨越整秒边界，不同 schedule 推进后的 nextFireTime 不同（S vs S+1000）；plannerB `fetchDueSchedules` 时 nextFireTime=S 的 schedule 仍 `<= now_B` 被重复 fire → 4 fire（期望 3）。高负载并行下处理耗时才使 plannerB 跨越整秒边界，故单独运行稳定、并行偶发。
- 修复（测试层，让测试专注验证乐观锁竞争、脱离 trigger 时序边界）：`newSchedule` 增大 `repeatIntervalMs` `1000→60000` 并设 `lastFireTime=now-61000`，使 plannerA 推进后 nextFireTime=`now+59000`（远在未来），plannerB 永不重复 fire。所有用 `newSchedule` 的测试均不依赖 interval 具体值或周期触发次数，增大 interval 安全。

### nop-stream-core: 修复 TestProcessingGuaranteeBehavior 无限 hang 导致整个 build 卡死（ad-hoc bug fix）

- 现象：`./mvnw test -T 1C` 卡住 14 分钟无进展，mvn 主进程与 surefire 子进程 CPU 均 0%（hang 死非慢）。jstack 定位到 `nop-stream-core` 的 `TestProcessingGuaranteeBehavior.testAlignedVsNonAlignedBarrierBehavior` 在 `ResultPartition.read:111`（`LinkedBlockingQueue.poll`）上无限阻塞；`nop-stream-core/target/surefire-reports/` 历史堆积 203 个 dump 文件，证明该测试长期反复 hang
- 根因（测试逻辑 bug）：aligned gate（`barrierAlignment=true`）要求所有 channel 收齐 barrier 才能完成对齐；但原测试只给 `p0_aligned` 写了 barrier，`p1_aligned` 既无 barrier 也未 `close()` → `barriersRemaining` 永远 =1，channel 0 永远被 skip，`isAllFinished()` 永远 false → `InputGate.readMultiChannel` 的 `parkNanos` 循环永久自旋。且 surefire 未配置 `forkedProcessTimeoutInSeconds`（pom.xml:201 被注释），测试方法无 `@Timeout`，整个 build 无限挂起
- 深层矛盾：原断言2 `assertFalse(alignedRecords.contains("after-barrier-aligned"))` 在逻辑上不可能在不 hang 的情况下成立 —— aligned（exactly-once）正确语义是对齐完成后释放 barrier 后缓冲数据；要阻塞 after-barrier 就必须让对齐永不完成，而那必然 hang。故断言2 本身错误
- 修复（`nop-stream/nop-stream-core/src/test/java/io/nop/stream/core/execution/TestProcessingGuaranteeBehavior.java`）：
  - 给 `p1_aligned` 补 `CheckpointBarrier`（aligned/non-aligned 两 gate 的 p1 均补齐 barrier，使对齐能完成）
  - 所有 4 个 `ResultPartition` 在 read 前 `close()`（发 EOS，使 `isAllFinished()` 为 true，read 循环自然终止）
  - 修正断言2：`assertFalse` → `assertTrue`（对齐完成后释放 barrier 后数据，符合 exactly-once 语义）
  - 新增时序断言：aligned 下 `firstBarrierIndex < firstRecordIndexWithValue("after-barrier-aligned")`，保留"对齐期间阻塞 barrier 后数据"的验证价值（原断言2 的本意），用正确的、可终止的方式表达
  - 抽取 `collectRecords`/`firstBarrierIndex`/`firstRecordIndexWithValue` 三个 helper
- 按用户要求未启用 surefire 全局超时（理由：不同机器速度不同），改为从测试逻辑层面保证可终止
- 验证：
  - `./mvnw test -pl nop-stream/nop-stream-core -am -Dtest=TestProcessingGuaranteeBehavior -Dsurefire.failIfNoSpecifiedTests=false` → 2 tests, 0 failures, BUILD SUCCESS（无 hang）
  - `./mvnw test -pl nop-stream/nop-stream-core` → 982 tests, 0 failures, 0 errors, 1 skipped, BUILD SUCCESS（无回归、无 hang）
- 清理：删除 `target/surefire-reports/` 下历史 hang 残留的 206 个 `.dump`/`.dumpstream` 垃圾文件

  3. **max 5 rounds 太宽松**，AI 倾向用满追求"consensus"。
  4. Revise 子 agent 改 Minor 时引入新 Major（P4 教训）。

- 接口兼容：PLAN_AUDIT step 的 `resultTag`（默认 AI_STEP_RESULT）、transitions（approved→EXECUTE_PLAN / issues→retry PLAN_DRAFT）、`<ISSUES>` 输出格式均不变；prompt 同时被 `deep-audit-loop.json:46` 复用，改动对两者一致适用。
