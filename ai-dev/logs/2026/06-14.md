### goal-driver: 优化 PLAN_AUDIT prompt（审查耗时与行为）

- 背景：分析通宵运行 `_tmp/2026-06-13-233645-goal-driver`（9 个完整计划周期），发现 PLAN_AUDIT 平均 **19m38s/次**，是拟制 PLAN_DRAFT（7m48s）的 2.5 倍，是除 EXECUTE 外最大耗时。逐个读取 3 份 audit 日志（P2 plan161 29m、P3 plan162 7m30s、P4 plan163 31m54s）定位根因。
- 根因（4 点）：
  1. **AI 在已 PASS 时主动加轮**：prompt 判据是"0 Blocker + 0 Major 即通过"，但 AI 把 Minor 也当返工触发器。P4 的 R1 本已 0B/0M（仅 11 Minor），AI 主动返工 → 改坏 → R2 查出 2 Major → 再修再审，从 7m 膨胀到 32m。
  2. **主 agent 做了大量与子 agent 重复的 context gathering**：P2 主 agent 在派 review 子 agent 前，自己 Read 15+ 源文件、5 design doc、grep 统计（约 13 分钟），但子 agent 是独立会话看不到，会自己重探索——纯浪费。P3 不做探索、直接派子 agent，反而最快。
  3. **max 5 rounds 太宽松**，AI 倾向用满追求"consensus"。
  4. Revise 子 agent 改 Minor 时引入新 Major（P4 教训）。
- 改动：`ai-dev/tools/opencode-goal-driver/prompts/plan-audit.md` 完整重写，落实 4 条硬规则：
  - **coordinator 约束**：主 agent 只读 `{{PLAN_FILE}}` + plan guide，禁止 Read/Grep/Glob 源码与 design doc；所有 repo 验证交给 review 子 agent。
  - **通过即停**：R1 若 0B/0M 立即 approved，禁止为 Minor 或"consensus"返工。
  - **返工范围限定**：仅 Blocker/Major 触发 revise；revise 子 agent 禁止动 Minor 文本。
  - **轮数收紧**：max 5 → **max 2 review rounds**，禁止 Round 3+ 及 polish/confirm 轮。
- 接口兼容：PLAN_AUDIT step 的 `resultTag`（默认 AI_STEP_RESULT）、transitions（approved→EXECUTE_PLAN / issues→retry PLAN_DRAFT）、`<ISSUES>` 输出格式均不变；prompt 同时被 `deep-audit-loop.json:46` 复用，改动对两者一致适用。
- 验证：`node --test`（core/intg/group-subflow/transitions）→ 63 pass / 0 fail。预期效果：审查行为统一向 P3 看齐（7m30s），平均从 19m38s 降到 ~10m。
- Doc-sync: No owner-doc update required（goal-driver 内部 prompt 调优）。

- 新增 `completion/LlmCompletionJudge.java` implementing `ICompletionJudge`，行为语义：
  - **Null/blank guard**：assistantMessage 为 null 或内容空白 → 直接 Complete，不调 LLM（call count=0）
  - **Cancel guard**：`ctx.isCancelRequested()` → fallbackDecision，不调 LLM（call count=0）
  - **Prompt 构造**：system prompt（Judge 指令）+ user message（task goal from `plan.getGoal()` → `agentModel.getDescription()` → "N/A" + 最近 N 条非 system 消息作为 context + 被评判的 assistant 响应）；temperature/maxTokens/model override 设到 ChatOptions
  - **LLM call**：`chatService.call(request, null)` 同步调用，try-catch 包裹，任何异常 → fallback，不向 ReAct 循环传播
  - **Error/null response guard**：response 为 null / `isSuccess()==false` / message 为 null / verdict 内容空白 → fallback
  - **Verdict 解析**：取第一个非空行，case-insensitive 匹配 COMPLETE/CONTINUE/ESCALATE；CONTINUE 用剩余文本作 continuation message（空则用默认），ESCALATE 用剩余文本作 reason；无法匹配 → fallback
