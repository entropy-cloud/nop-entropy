### 设计决策：Plan 归属模型

- **Plan 属于创建它的 Agent（主 Agent），有唯一 owner**
- 子 Agent 通过 `call-agent` 执行时可只读访问 Plan，运行时强制写保护
- 子 Agent 试图修改 Plan → 运行时报错 + 提示"只能由主 Agent 修改"
- 主 Agent 调用子 Agent 时附带交互规则（DelegationSpec 语义，不独立为概念）
- 不需要独立的 ExecutionPlanState——Plan 本身跟踪执行进度
- 否决了 analysis 审查中提出的"三层 Plan 概念拆分"（ProjectPlanDoc / ExecutionPlanState / DelegationSpec），改为两层：Plan（主 Agent 拥有，含进度跟踪）+ DelegationSpec（call-agent 的参数，不独立命名）

### 多 Agent + Todo 隔离实证

- 分析 opencode SQLite 中 4915 个子 agent 会话与 todo 的关系
- **结论：每个 Agent 独立 todo 列表，无需跨 Agent 聚合**
  - 605 个子 agent 有自己的 todo（12.3%），与父 agent todo 完全隔离
  - 父 agent todo = 调度层（"派了哪个子 agent / 哪个 wave 完成"）
  - 子 agent todo = 执行层（"具体步骤 1→2→3"）
  - 父子 todo 无联动：子 agent 通过 task 返回值汇报结果，父 agent 更新自己的 todo
- 深入分析 opencode `task.ts` 源码（338 行）：foreground（默认阻塞等待）+ background（实验性，立即返回+异步注入结果）
- **并行调度三模式**：串行逐个 task、同轮 parallel tool calls、background 异步
- **Todo item 与子 agent 没有 1:1 映射**：1 条 todo 可对应 0~N 个子 agent，粒度由 LLM 决定
- **Replace-All 天然支持并行**：并行子 agent 全部返回后一次性更新整个列表
- **Nop call-agent 需要 foreground + background 双模式**
