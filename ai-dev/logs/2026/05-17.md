### 2026-05-17（下午）— 吸引子层治理

对比 chaos-flux PPT（26 页方法论）和实际 `docs/` 结构后，识别出核心差距：`ai-dev/design/` 缺少 attractor index（层级 + precedence model），不像 chaos-flux 的 `docs/architecture/README.md` 那样定义了 Governing Principles → Normative Architecture → Platform-Extension → Focused Subsystem 的层级。

**根本原因分析**：两项目做法不同是因为文档受众不同。Chaos-flux 的 `docs/` 是单一统一树（AI-only 项目）；Nop-entropy 有 `docs/`（人工历史）+ `docs-for-ai/`（平台使用知识）+ `ai-dev/`（AI 开发过程），三棵树服务不同读者。这个分裂是正确的。

**从 chaos-flux 借鉴的核心是方法论**（PPT 第 25 页）：五个结构性问题必须有答案（谁定义方向？冲突时信谁？怎么发现漂移？怎么保留记忆？怎么判定完成？），而不是表面目录结构。

- 新增 `ai-dev/design/README.md`：定义层级（Writing Guide → Cross-Cutting → Subsystem）、precedence model（platform principles → subsystem local → conflict resolution）、与 `docs-for-ai/` 和 `docs/` 的关系
- 评估 governing-principles 文档：**不需要**。Nop 的 governing principles（可逆计算、model-first）已在 `docs/theory/` 和 `docs-for-ai/00-start-here/ai-defaults.md` 中覆盖，不像 chaos-flux 需要在项目内重新定义 Flux 七原语
- 更新 `ai-dev/design/00-design-writing-guide.md`：组织方式引用 README 层级
- 更新 `AGENTS.md`：design 行标注 attractor layer，路由增加先看 README
