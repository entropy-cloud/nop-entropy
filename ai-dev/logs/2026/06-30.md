### 修改计划 + 设计文档先行

创建 `ai-dev/plans/283-nop-stream-code-quality-fixes.md`（经两轮对抗性审查）。

窗口工厂接线问题（审计 defect #2）在设计文档中补充了架构决策（先于代码）：

- `core-design.md` §1.2.1：`StreamComponents` 由 `StreamExecutionEnvironment` 持有为字段，构造时 SPI 加载，随 API 调用链向下传递
- `core-design.md` §2.2.1：窗口算子工厂注入设计——工厂获取路径、core-only 快速失败、拒绝了什么
- `core-design.md` §2.5：`StreamExecutionEnvironment` 核心字段表增加 `components` 字段
- `window-design.md` §10.1：工厂注入约束——core-only 不走 WindowedStream API，用 OperatorTestHarness

设计决策要点：
1. `StreamComponents` 是 environment 的字段（非局部变量），解决时间点不匹配问题
2. `IWindowOperatorFactory` 通过 SPI 自动发现（与现有 IDeploymentPlanProvider 模式一致）
3. 工厂缺失时快速失败（不允许静默回退到不支持 checkpoint 的路径）
4. core-only 不支持 WindowedStream API 窗口操作（core 是接口层，不是执行层）

### 代码深度审计

在 `ai-dev/analysis/2026-06-30-nop-stream-code-audit.md` 中记录了完整的深度审计报告，包含：

**包结构**：4/9 模块为空壳（api/checkpoint/flink/flow），`execution` 包膨胀（26 文件混入 5 种职责），`TimerService` 在两个包中有名称冲突

**实现完整性**：6 处 `UnsupportedOperationException` 桩方法，InputGate 中有 6 处从 `Optional<StreamElement>` 返回 raw null（**高严重度**），78 个文件使用通配符导入，WindowedStreamImpl 使用已废弃 API 作为默认回退路径

**测试覆盖**：~260 个测试类 / ~1,776 个 @Test / 0.73 测试源码比。覆盖最强的：CEP、Checkpoint 协调器、状态后端、窗口操作、执行引擎。覆盖最弱的：`common.functions.co`（零测试）、Operator State（未实现零测试）

**E2E 测试**：9 个 core 集成测试 + 多个 runtime E2E 测试，但无真实跨 JVM 分布式 E2E，无 Operator State E2E，无分支/合并多链场景 E2E

**对比 Flink/SeaTunnel**：nop-stream 测试密度更高（0.73 vs ~0.5），但模块成熟度低（空壳多，代码规范差）
