## newEntity() 原则落地：nop-job 全模块 `new NopXxx` → `dao.newEntity()`

- **背景**：发现 nop-job 大量 `new NopJobTask()/NopJobFire()` 直接构造实体（生产 builders + planner + store + bizmodel + ~16 个测试），违反「实体应经 `newEntity()` 创建」原则。文档只讲了 Delta 派生类理由，缺 ORM enhancer/运行时能力理由。
- **改动**：
  - **生产代码 8 处** → `newEntity()`：`DefaultJobTaskBuilder`/`PartitionTaskBuilder`/`RpcBroadcastTaskBuilder`/`AdaptiveJobTaskBuilder`（注入 `IDaoProvider`，composite builder 在 `setDaoProvider` 中向 fallback 传播）、`JobPlannerScannerImpl`（注入 `IDaoProvider`）、`JobScheduleStoreImpl`（用既有 `fireDao()`）、`NopJobFireBizModel`（`newEntity()`）、`NopJobScheduleBizModel`（`daoProvider().daoFor(NopJobFire.class)`）。
  - **测试**：builder 单测升级为 `JunitBaseTestCase` + `@NopTestConfig(localDb=true, initDatabaseSchema=TRUE)` + `@Inject IDaoProvider`；集成测试（`TestJobCoordinatorScanner`/`TestJobConcurrency` 等）在构造 builder/planner 后 `setDaoProvider(daoProvider)`。
  - **AdaptiveJobTaskBuilder 多赋值循环**：保留 working-tree 中的多赋值 loop 版本（用户 WIP，一个 fire → 多 worker 各一个 task），把过时的 `testMultipleAssignmentsFailFast` 改为 `testMultipleAssignmentsCreateMultipleTasks`（断言 2 赋值产 2 task、taskNo 1/2）。
- **踩坑（关键）**：builder 单测只 `extends JunitBaseTestCase` 漏加 `@NopTestConfig(localDb=true)` → 容器初始化抛 `配置[nop.datasource.driver-class-name]的值为空`。根因：`@Inject IDaoProvider` 拉起 `nopDataSource` bean，需 `localDb=true` 提供 H2 配置。
- **数据丢失与恢复**：会话中途一次 `git pull` + `git reset`（reflog: `reset: moving to HEAD`）丢弃了全部未提交改动。所幸 `stash@{0}`（WIP on merge commit）保存了它们，`git stash pop` 恢复（附带恢复了 nop-sys 等他人 WIP，未触碰）。**教训：频繁提交。**
- **验证**：`./mvnw test -pl nop-job/nop-job-dao,nop-job/nop-job-coordinator,nop-job/nop-job-worker,nop-job/nop-job-service` 全绿（dao 36 / coordinator 159 / worker 36 / service 41 = 272，0 失败）。
- **Doc-sync**：
  - `docs-for-ai/04-reference/safe-api-reference.md`、`03-runbooks/write-bizmodel-method.md`、`02-core-guides/orm-model-design.md`：补全 `newEntity()` 双理由（Delta 派生类 + ORM enhancer 运行时能力）。
  - `docs-for-ai/02-core-guides/testing.md` §`@NopTestConfig` 关键点：新增「触发规则」——`JunitBaseTestCase` 一旦 `@Inject` DB 相关 bean（`IDaoProvider`/dao/`newEntity()`），必须 `@NopTestConfig(localDb=true)`。

## nop-sys 广播事件 findNext 时间窗口重构

- **背景**：广播事件消费原先围绕 `NopSysBroadcastCursor`（subscriber cursor + lease）建模。
  由于 `AppConfig.hostId()` 在未配置时为随机 UUID，subscriberId 无法稳定绑定到每台服务器，
  持久化 cursor 对缓存失效等临时性广播事件是过度设计。
- **变更**：
  - 新建 `BroadcastEventProcessor` 类，通过 `IEntityDao.findNext(lastEntity, filter, null, limit)`
    keyset pagination + 内存 `lastEvent[topic]` 游标 + 时间窗口过滤消费广播事件。
  - 消费失败不阻塞后续事件（catch + log + continue）；`ConsumeLater` 返回值记 warn 后跳过。
  - 每条事件 fan-out 给该 topic 的全部订阅者。
  - `SysDaoMessageService` 退化为 `IMessageService` 门面，广播消费委托给 processor。
  - 移除 `NopSysBroadcastCursor` 实体及跨 5 个子模块的全部手写引用（api/dao/service/meta/web）。
- **测试**：15 个测试全部通过（6 个新广播测试 + 9 个非广播/发送路由测试）。
- **关联**：`ai-dev/plans/289-nop-sys-broadcast-findnext-refactor.md`、`ai-dev/design/nop-sys/sys-event-architecture.md`

## NopJobTask 参数解析重构：taskPayload 语义化 + getEffectiveParams

- **背景**：`NopJobTask.taskPayload` 原由 `DefaultJobTaskBuilder` 写入冗余信封
  `{jobFireId, jobParamsSnapshot}`，且全仓库无生产代码读取它（worker 执行路径
  `DefaultJobExecutionContextBuilder.resolveJobParams` 直接读 fire 的 snapshot，
  绕过 task）。字段语义不清、与 `PartitionTaskBuilder`/`RpcBroadcastTaskBuilder`
  （不设 taskPayload）不一致。

- **改动**：
  - `DefaultJobTaskBuilder`：`task.setTaskPayload(null)`——taskPayload 改为
    **task 级专用参数槽位**（task 级覆盖 fire 共享参数），default single 任务无 task 级参数。
  - `NopJobTask.getEffectiveParams(NopJobFire fire)`：新增只读方法，以 fire 的
    `jobParamsSnapshot` 为基准、用 task 的 `taskPayload` 覆盖合并（task 级优先），
    返回 mutable `HashMap`（支持后续 headers 注入）。
  - `DefaultJobExecutionContextBuilder.resolveJobParams`（worker 路径）与
    `DefaultJobCancelHandler.resolveJobParams`（cancel 路径）：统一改调
    `task.getEffectiveParams(fire)`，消除两处重复的 fire-snapshot 读取 + schedule 兜底逻辑。
  - `NopJobScheduleBizModel.resolveJobParams` **不改**：它是 fire 快照的**生产方**
    （schedule + 手动 override → `fire.jobParamsSnapshot`），无 task 上下文，属不同关注点。

- **偏差**：最初拟用无参 `getEffectiveParams()` 走 ORM `getJobFire()` 关联懒加载。
  实测 `TestDefaultJobCancelHandler` 4 例失败——`DynamicOrmEntity.internalGetRefEntity`
  在关联未链接时调 `requireEnhancer()`，对 `new NopJobTask()` 这类 **未 orm_attach**
  的 detached 实体抛异常，被 `cancelRunningTask` 的 try/catch 吞掉导致 `cancelAsync`
  未调用。改为**显式传 fire 参数**：调用方（worker/cancel）均已持有 fire，传参零成本、
  无 enhancer/session 依赖、detached 单测亦可运行。关联 getter 懒加载仍可用于
  "只有 task 一端引用"的场景，但调用方已持有双方时传参更稳更省。

- **验证**：`./mvnw test -pl nop-job/nop-job-dao,nop-job/nop-job-coordinator,nop-job/nop-job-worker,nop-job/nop-job-service -am`
  全绿（dao 36 / coordinator 159 / worker 36 / service 41，含 `TestDefaultJobCancelHandler` 12、
  `TestDefaultJobTaskBuilder` 3、`TestJobWorkerScanner` 15、`TestNopJobScheduleBizModel` 15）。
- **Doc-sync**：
  - `docs-for-ai/02-core-guides/orm-model-design.md` §关系声明 新增「运行时语义：关联的透明懒加载」
    小节——补全 ORM `<to-one>`/`<to-many>` 的懒加载运行时行为（与 JPA 一致、需 orm-attached、
    session 缓存复用、detached 实体抛 requireEnhancer、传参 vs 懒加载判别要点）。
    此前该语义仅 `domain-logic-and-ddd.md:46` 一行带过，ORM 专题文档未提。
