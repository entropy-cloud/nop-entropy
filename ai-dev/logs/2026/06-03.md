## Plan 08 Closure — nop-code Frontend Plan Completed

Plan 08 (nop-code-frontend-plan) completed. All 8 phases implemented, build passes.

### Changes
- **NopCodeIndex.view.xml**: Added symbolCounts distribution table to stats dialog (C3/C22)
- **NopCodeSymbol.xmeta**: Removed redundant `kinds` prop, added `ui:filterOp="in"` on `kind` prop for standard Nop filter mechanism (C14)
- **NopCodeSymbol.view.xml**: Query form uses `kind` instead of `kinds`
- **code-browser.view.xml**: Added sourceCode display panel (C9)
- **type-hierarchy.view.xml**: Direction field uses dict dropdown, results rendered as `input-tree` instead of raw JSON (C15/C16)
- **call-hierarchy.view.xml**: Same improvements as type-hierarchy (C18/C19)
- **dashboard.view.xml**: Added symbolCounts distribution table to stats dialog (C22)
- **docs-for-ai/03-runbooks/custom-query-with-querybean.md**: Added multi-select IN query best practice (filter_{field}__in pattern, ui:filterOp)

### Key Lesson
AI created a redundant `kinds` (List<String>) prop because it didn't know Nop's standard `filter_` prefix mechanism for IN queries. The correct pattern is `ui:filterOp="in"` on the existing `kind` prop — the frontend automatically generates `filter_kind__in` field name and the backend receives a standard `FilterBeans.in("kind", values)`.
