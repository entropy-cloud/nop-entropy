## 维度 03：API 表面积与契约一致性

### [维度03-01] copyMap 方法名暗示深拷贝但实际未拷贝

- **文件**: `nop-job/nop-job-coordinator/src/main/java/io/nop/job/coordinator/engine/JobPlannerScannerImpl.java:228-230`
- **证据片段**:
  ```java
  private Map<String, Object> copyMap(Map<String, Object> map) {
      return map == null ? Collections.emptyMap() : map;
  }
  ```
- **严重程度**: P2
- **现状**: 方法名叫 copyMap 但只返回原始引用。buildFire 中用于将 schedule 的 jobParams 传递给 fire 的 jobParamsSnapshotComponent。JsonOrmComponent 内部会 JSON 序列化，实际安全，但方法名误导。
- **风险**: 未来维护者可能误认为这是安全的深拷贝，在其他上下文中错误使用。
- **建议**: 改名为 emptyIfNull 或实际执行 new HashMap<>(map)。
- **信心水平**: 70%
- **误报排除**: JsonOrmComponent.set_jsonValue 内部做 JSON 序列化/反序列化保证数据隔离，实际不是数据问题。
- **复核状态**: 未复核

## 维度 12：GraphQL 与 API 层

### [维度12-01] scheduleStatus insertable 未限制，允许通过 CRUD 创建时设置任意状态

- **文件**: `nop-job/nop-job-meta/src/main/resources/_vfs/nop/job/model/NopJobSchedule/NopJobSchedule.xmeta:5`
- **证据片段**:
  ```xml
  <prop name="scheduleStatus" defaultValue="0" updatable="false"/>
  ```
  对比生成的 _NopJobSchedule.xmeta: `insertable="true" updatable="true"`
- **严重程度**: P2
- **现状**: Delta xmeta 将 scheduleStatus 设为 updatable=false 但未覆盖 insertable（继承 true）。可通过 GraphQL CRUD 创建时传入任意 scheduleStatus 值，绕过状态机约束。
- **风险**: 可直接创建 scheduleStatus=1（ENABLED）的 schedule，跳过 enableSchedule mutation。但 nextFireTime 为 null 的 ENABLED schedule 不会被 planner 扫描。
- **建议**: 设为 insertable="false" updatable="false"，让创建时自动使用 defaultValue。
- **信心水平**: 65%
- **误报排除**: 创建为 ENABLED 但无 nextFireTime 的 schedule 不会被引擎扫描，实际危害有限。
- **复核状态**: 未复核

### 维度 12 其他检查结论

- 所有 @BizMutation 方法正确映射到 GraphQL mutation
- 分页查询使用标准 QueryBean + doFindPage/doFindList
- 整个模块无硬编码 SQL，SQL 注入风险为零

## 维度 13：安全与权限模型

### [维度13-01] 所有 mutation 权限硬编码为 roles="admin"

- **文件**: `nop-job/nop-job-service/src/main/resources/_vfs/nop/job/model/NopJobSchedule/NopJobSchedule.xbiz:3-22`
- **证据片段**:
  ```xml
  <mutation name="enableSchedule">
      <auth roles="admin" permissions="NopJobSchedule:enableSchedule"/>
  </mutation>
  ```
- **严重程度**: P3
- **现状**: 所有 8 个 mutation 的 roles 都硬编码为 admin，加上 permissions 和 action-auth.xml 形成三层控制。admin-only 可能有意的安全基线。
- **风险**: 如果业务需要非 admin 角色触发 schedule，需修改 xbiz。
- **建议**: 如果确认 admin-only 正确则无问题。如需灵活控制可去掉 roles="admin" 只保留 permissions。
- **信心水平**: 60%
- **误报排除**: Nop 平台项目中 roles="admin" 配合 permission 是标准双重保险模式。
- **复核状态**: 未复核

## 维度 14：异步与事务模式

### [维度14-01] @SingleSession 扫描方法中批量处理 fire 记录

- **文件**: `nop-job/nop-job-coordinator/src/main/java/io/nop/job/coordinator/engine/JobCompletionProcessorImpl.java:123-148`
- **严重程度**: P2（但实际风险低）
- **现状**: doScan() 使用 @SingleSession，循环中 completeFireAndUpdateSchedule 使用 REQUIRES_NEW 隔离了写操作。Session 期间持有连接并累积实体对象。
- **风险**: REQUIRES_NEW 已隔离关键写操作，不会形成数据库长事务。主要风险是内存累积。
- **建议**: 可在每条 fire 处理完后清理 Session 缓存。性能优化而非正确性问题。
- **信心水平**: 80%
- **误报排除**: REQUIRES_NEW 已经将写操作隔离到独立事务。
- **复核状态**: 未复核

### [维度14-02] persistSchedule 乐观锁重试可能丢失并发引擎字段更新

- **文件**: `nop-job/nop-job-service/src/main/java/io/nop/job/service/entity/NopJobScheduleBizModel.java:140-162`
- **严重程度**: P2（但实际风险低）
- **现状**: 重试时只恢复了引擎字段（restoreEngineFields），业务字段保留在内存对象上。对 job 调度场景安全。
- **建议**: 当前实现安全。如需更严格可在重试时检查 scheduleStatus 预期值。
- **信心水平**: 75%
- **误报排除**: restoreEngineFields 已覆盖所有引擎管理字段。job 调度场景下并发状态变更极少。
- **复核状态**: 未复核

### [维度14-03] handleExecutionResult 更新 task 失败后未重试

- **文件**: `nop-job/nop-job-worker/src/main/java/io/nop/job/worker/engine/JobWorkerScannerImpl.java:259-271`
- **严重程度**: P2
- **现状**: taskStore.updateTask 因乐观锁冲突失败时，若 fresh task 仍在 RUNNING 状态，仅记录 warn 日志不做重试。执行结果可能丢失。
- **风险**: RUNNING 状态下的版本冲突会导致执行结果丢失，task 停留在 RUNNING 直到被 TimeoutChecker 超时处理。
- **建议**: 在版本冲突且 fresh task 非终态时，用 fresh 版本号重新设置结果字段并重试更新。
- **信心水平**: 75%
- **误报排除**: 大多数版本冲突来自 TimeoutChecker 同时修改。RUNNING 状态下的冲突可能是 tryMarkSuspiciousIfWorkerGone 修改了 updatedBy 等字段。
- **复核状态**: 未复核

### 维度 14 其他检查结论

- 无 txn() 调用，事务管理通过 @Transactional 和 @SingleSession
- 无 txn().afterCommit() 调用
- 所有 Scanner 使用 scheduleWithFixedDelay，stopScanning 正确取消 Future
- 所有 Store 使用乐观锁 + 重试机制处理并发
- 异步错误处理完整，所有 scanner 的 scanOnce 有顶层 try-catch
