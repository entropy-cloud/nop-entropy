## 维度 13：安全与权限模型

### [维度13-01] filterByFilePattern glob→regex 转换存在 ReDoS 风险

- **文件**: `nop-code/nop-code-service/src/main/java/io/nop/code/service/impl/CodeSearchService.java:243-249`
- **证据片段**:
  ```java
  private List<CodeSearchResultDTO> filterByFilePattern(List<CodeSearchResultDTO> results, String filePattern) {
      if (filePattern == null || filePattern.isEmpty()) return results;
      String pattern = filePattern.replace(".", "\\.").replace("*", ".*").replace("?", ".");
      return results.stream()
              .filter(r -> r.getFilePath() != null && r.getFilePath().matches(pattern))
              .collect(Collectors.toList());
  }
  ```
- **严重程度**: P2
- **现状**: 用户输入的 filePattern 经过简单 glob→regex 转换后直接用于 `String.matches()`。恶意构造可能导致正则引擎 CPU 占用过高。
- **风险**: DoS 攻击面。影响范围有限（在已过滤的结果集上操作）。
- **建议**: 对连续通配符进行归一化：`pattern = pattern.replaceAll("\\.\\*+", ".*");`
- **信心水平**: 90%
- **误报排除**: API 暴露的端点不应假设调用者行为。
- **复核状态**: 未复核

### [维度14-02] CodeCacheManager 的 LinkedHashMap 非线程安全操作

- **文件**: `nop-code/nop-code-service/src/main/java/io/nop/code/service/impl/CodeCacheManager.java:57-95`
- **证据片段**:
  ```java
  private final Map<String, CacheEntry> analysisCacheMap = new LinkedHashMap<>(16, 0.75f, true);

  CacheEntry getValidEntry(String indexId) {           // 无 synchronized
      CacheEntry entry = analysisCacheMap.get(indexId); // LinkedHashMap.get 触发 LRU 重排
      if (entry == null) return null;
      if (entry.isExpired()) {
          analysisCacheMap.remove(indexId);             // LinkedHashMap.remove
          return null;
      }
      return entry;
  }
  ```
- **严重程度**: P2
- **现状**: access-order `LinkedHashMap` 的 `get` 操作会修改内部链表结构（LRU 重排）。`getValidEntry()` 和 `getOrCreateEntry()` 没有 synchronized，但 `getOrRebuildSymbolTable()` 等方法有 synchronized。
- **风险**: 并发请求下可能导致 `ConcurrentModificationException` 或内部链表损坏。
- **建议**: 将 `getValidEntry()` 和 `getOrCreateEntry()` 也标记为 `synchronized`。
- **信心水平**: 88%
- **误报排除**: JDK 文档明确指出非同步的 LinkedHashMap 在并发修改时行为未定义。
- **复核状态**: 未复核
